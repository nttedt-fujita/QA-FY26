**目的**: 記録ツール設計時に AI 連携を見据えた設計判断ができるようにする

## 2. 連携設計で考慮すべき要素

### 2.1 データ設計（最も重要）

記録ツールを設計する段階で、将来の AI 連携を見据えたデータモデルにしておく必要がある。

| 考慮事項 | 詳細 | なぜ重要か |
|---|---|---|
| **画像の保存設計** | 検査写真をどこに、どの形式で保存するか | AI 学習には元画像へのアクセスが必要。リサイズ済みのみだと精度に影響 |
| **画像とメタデータの紐付け** | 画像 1 枚ごとに「部品ID・ロットID・サプライヤー・検査日・判定結果・不良モード」を紐付ける | AI の教師データとして使うには、画像単体ではなくラベル情報がセットで必要 |
| **不良モードの分類体系** | 不良の種類（寸法不良・外観不良・機能不良・材料不良 等）を統一コードで管理 | AI のクラス分類ラベルに直結する。後から分類体系を変えると学習データの作り直しが必要 |
| **画像フォーマット** | JPEG or PNG、解像度の統一 | AWS SageMaker の場合 PNG/JPEG、最小 64×64〜最大 4096×4096 ピクセル |
| **撮像条件の記録** | カメラ種類・照明条件・撮影角度・距離 | AI モデルの精度は撮像条件に大きく依存。条件がバラバラだとモデルの汎化性能が下がる |

#### 画像保存のアーキテクチャ選択肢

| 方式 | 概要 | メリット | デメリット |
|---|---|---|---|
| **S3（AWS）直接保存** | 検査ツールから直接 S3 にアップロード | SageMaker との連携が最もシンプル。スケーラブル | AWS 依存。ネットワーク帯域が必要 |
| **ローカル NAS + 定期同期** | 現場 NAS に保存し、バッチで S3 に同期 | オフライン時も撮影可。現場の通信環境に依存しない | 同期の仕組みが必要。リアルタイム性は低い |
| **DB に直接格納（BLOB）** | PostgreSQL 等に画像バイナリを格納 | シンプルな構成 | 画像量が増えると DB 肥大化。AI 連携時に取り出しが面倒 |

**推奨**: テキストデータは DB（RDS 等）、画像は S3、両者をキー（検査ID）で紐付ける構成。
これが AWS の AI サービスとの連携で最もスムーズ。

---

### 2.2 API / インターフェース設計

記録ツールと AI サービスを繋ぐ際のインターフェース。

| パターン | 概要 | ユースケース |
|---|---|---|
| **バッチ処理** | 記録ツールに蓄積された画像を定期的に AI に送信し、結果を DB に書き戻す | 受け入れ検査のように「即時判定」が不要な場合に適する。コスト最適化しやすい |
| **リアルタイム API** | 検査員が写真を撮った瞬間に AI に送信し、判定結果を即座に返す | ライン検査のようにスピードが求められる場合。常時エンドポイント稼働 → コスト高 |
| **半自動（ヒューマンインザループ）** | AI が判定 → 低確信度の場合は検査員が最終判断 → その結果が学習データにフィードバック | 導入初期に最も現実的。AI の精度向上と検査員の判断支援を両立 |

**受け入れ検査の場合**: バッチ処理 or 半自動が現実的。
ライン検査ではないので、リアルタイム性は求められない。

---

### 2.3 AI モデルのライフサイクル管理

| フェーズ | やること | 記録ツール側の要件 |
|---|---|---|
| **データ収集** | 日常の検査で画像 + ラベルを蓄積 | 画像アップロード機能、不良モード選択 UI |
| **学習データ準備** | 蓄積データから学習セットを抽出 | ラベル付き画像のエクスポート機能（S3 manifest 形式等） |
| **モデル学習** | SageMaker 等でモデルをトレーニング | 記録ツール外の作業（SageMaker Console / Notebook） |
| **推論（検査支援）** | 新しい検査画像を AI に渡して判定 | API 呼び出し機能、判定結果の表示・記録 |
| **フィードバック** | AI 判定の正誤を検査員が確認 | 「AI 判定合ってた / 間違ってた」のフィードバック UI |
| **再学習** | フィードバックデータでモデル更新 | フィードバックデータのエクスポート |

---

### 2.4 セキュリティ・ガバナンス

| 項目 | 考慮事項 |
|---|---|
| **画像データの機密性** | 部品の画像は製品情報に該当する可能性。クラウドに上げてよいか社内ポリシー確認が必要 |
| **アクセス制御** | 検査画像・AI 判定結果へのアクセス権限設計 |
| **データ保持期間** | 検査記録・画像の保存期間（法的要件・品質規格要件） |
| **AI 判定の責任** | AI の判定結果を最終判断とするか、あくまで補助とするか。品質責任の所在 |
| **監査証跡** | AI が判定した記録と、人間が最終承認した記録を分離して保持 |

---

### 2.5 現場導入の実務面

| 項目 | 考慮事項 |
|---|---|
| **撮像デバイス** | 専用カメラ vs スマートフォン vs タブレット。現場の作業性と画質のバランス |
| **ネットワーク環境** | 検査現場の通信環境（Wi-Fi / 有線 / オフライン）。画像アップロードの帯域 |
| **検査員の操作負荷** | 写真撮影→アップロード→結果確認のフローが今の作業にどれだけ負荷を加えるか |
| **教育・トレーニング** | 検査員への新ツール・AI 操作の教育コスト |

## 4. AI 連携を見据えた記録ツールの設計原則

Phase 2 で記録ツールを設計する際、以下を最低限組み込んでおくことで、将来の AI 連携がスムーズになる：

### 4.1 Must（必須）

- 検査画像の保存機能（オブジェクトストレージ連携）
- 画像と検査メタデータの紐付け（検査ID をキーに）
- 不良モードの統一分類コード
- 良品 / 不良品のラベル付け
- データエクスポート機能（CSV / JSON）

### 4.2 Should（強く推奨）

- 撮像条件の標準化ガイドライン策定
- S3 互換ストレージの採用（将来の SageMaker 連携を考慮）
- AI 判定結果の記録フィールド（将来用に空でも用意）
- フィードバック UI の設計（AI 判定の正誤を記録）

### 4.3 Could（あれば望ましい）

- AI 推論 API の呼び出し機能
- 検査結果ダッシュボード
- サプライヤースコアカードの自動生成

## 5. 推奨アプローチ

```
Phase 2（今）
  記録ツールを設計する際に、上記 Must / Should を組み込む
  画像保存は S3 互換で設計
  AI のことは意識するが、AI 機能自体はまだ実装しない
      │
Phase 3（データ蓄積後）
  蓄積データでパレート分析
  最も工数がかかっている部品×不良モードを特定
  その特定領域に対して AI の PoC を実施
      │
      ├─ まず: マルチモーダル LLM で試す（最も低コスト・高速）
      │       → 精度・コスト・実用性を評価
      │
      └─ 次に: SageMaker Canvas or 専用モデルで精度向上
              → PoC の結果次第で本格導入判断
      │
Phase 4（本格導入）
  半自動ワークフロー（AI 判定 + 人間最終承認）で運用開始
  フィードバックループでモデル継続改善
```
