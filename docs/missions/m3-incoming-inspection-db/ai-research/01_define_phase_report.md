# 受け入れ検査プロセス改善 — Define フェーズ 暫定レポート

**作成日**: 2026-03-07
**フェーズ**: DMAIC - Define（暫定）
**対象**: SCM 部品受け入れ検査プロセス
**ステータス**: 一次調査前・仮説段階

---

## 1. Problem Statement（問題定義）

> SCM におけるドローン部品の受け入れ検査に月 300h 以上の工数がかかっており、
> かつ改善に必要な検査記録が適切に蓄積されていない。
> 検査の有効性（不良流出をどれだけ防いでいるか）が定量的に評価できない状態にある。

### 問題の本質

- 300h が「多すぎるのか適切なのか」を判断する根拠がない
- 検査が後工程への不良流出をどの程度防いでいるか不明
- 記録がないため、改善サイクル（PDCA）が回せない
- 「AI を使えばいいのでは」「DB 化しよう」「ツールを作ろう」という手段先行の議論が発生しているが、解くべき問題が定義されていない

---

## 2. 暫定的な課題構造

### 2.1 見えている事実

| 項目 | 現状 |
|------|------|
| 検査方式 | 抜取検査 |
| 月間工数 | 約 300h |
| 記録の状態 | 改善のための記録が適切に取れていない |
| 検査有効性の把握 | 未定量化 |
| サプライヤー信頼度 | 基本的に信用できない前提だが、定量的根拠なし |

### 2.2 見えていないこと（一次調査で明らかにすべき）

- 300h の内訳（どの作業に何時間かかっているか）
- 検査対象の部品種類数・サプライヤー数・月間ロット数
- 抜取検査計画の根拠（AQL、サンプルサイズの設定理由）
- 検査項目数と判定基準の明確さ
- 不合格時のフロー（返品？再検査？手戻り工数は？）
- 後工程での品質クレーム発生状況（外部失敗コスト）
- 検査員の人数・スキルレベル・判定のばらつき

## 3. ステークホルダーの声（Voice of Customer）

| ステークホルダー | 関心事 | 本質的ニーズ |
|---|---|---|
| 現場（SCM 検査担当） | 工数がかかりすぎている | 業務負荷の適正化 |
| マネジメント | 記録がない、改善が回らない | 可視化・データに基づく意思決定 |
| 技術部門 | 「AI で解決できるのでは」 | 既存技術資産の活用 |
| 製造ライン（後工程）| 不良部品が来ないこと | **検査の本質的な目的** |
| QA 組織（新設） | 最初の改善テーマとして成果を出す | 組織としての存在意義の確立 |

### 真の顧客

受け入れ検査の本質的な目的は「不良部品を後工程に流さないこと」。
工数削減だけを目的にすると検査の質が落ちるリスクがある。
**有効性の担保と効率化は両立させる必要がある。**

## 5. 暫定結論

### 5.1 「ツール制作」の位置づけ

ツール（DB 化・AI 活用）は手段であり、目的ではない。
ただし、**DMAIC の Measure フェーズに進むためにはデータ収集基盤が必要**であり、
記録の仕組みを作ること自体が Define → Measure への橋渡しになる。

つまり：

- ✗ 「ツールを作れば問題が解決する」
- ○ 「問題を解決するプロセスの中で、記録基盤としてのツールが必要になる」

### 5.2 「AI 活用」の位置づけ

社内の橋梁点検 AI を受け入れ検査に転用する案が出ているが、
**何を検査するか・何を基準に判定するかが明確でない段階で AI を導入しても効果は測れない。**
検査基準の明確化と記録の蓄積が先行すべき。

### 5.3 サプライヤーに対するスタンス

「基本的に信用できない」という前提は妥当だが、
それを定量的に裏付けるデータがない。
サプライヤーごとの不良率・傾向を記録し、
将来的にはデータに基づいてサプライヤーを層別し、
検査の重点配分を最適化する方向を目指す。
