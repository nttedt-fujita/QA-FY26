既存の Excel データが「不完全でも」使える可能性がある。完璧なデータを待つよりも、今ある Excel データから「どの部品に工数が集中しているか」の大まかな傾向だけでも抽出できれば、提案 A（抜取検査計画見直し）の根拠として十分役立つ。一次調査では「今ある Excel から何が読み取れるか」を最初に確認すべき。

## 07. AI 連携設計とコスト分析への補足

### ベンチャー制約を踏まえた現実的な構成の再評価

07 では構成 A〜D の 4 パターンを提示したが、ベンチャーの制約を踏まえると:

- **構成 D（専業ベンダー）**: 初期 ¥120〜1,100 万は非現実的。除外
- **構成 B（SageMaker + 記録ツール内製）**: 初期 ¥50〜200 万は Phase 4 以降の話。今は不要
- **構成 A（SageMaker Canvas）**: PoC としてはあり得るが、まだ早い
- **構成 C（マルチモーダル LLM）**: 最も軽量。データが貯まった段階で最初に試すべき

結論として、07 の内容は「将来の参考資料」として保持するが、直近のアクションとしては記録基盤の構築が優先。AI 連携の詳細設計は Phase 3 のデータ分析結果が出てから着手する。

## 08. アプローチ案への補足

### 提案 B の修正：Google フォーム vs プロトタイプツール

プロトタイプに既に着手していたという情報を踏まえると、選択肢は 2 つ:

**選択肢 1: プロトタイプを活かす**

プロトタイプの設計を、このセッションの議論を踏まえて再定義した上で続行する。
表記揺れの精査結果をそのままマスタデータとして活用できるメリットがある。

ただし条件として:
- 07 の Must/Should を満たすデータ設計に修正する（特に画像保存と不良モード分類）
- 過剰に作り込まず、まず「記録が取れる」最小限で動かす
- 1〜2 週間以内にリリースできる規模に抑える

**選択肢 2: Google フォーム + スプレッドシートで即日開始**

プロトタイプを一旦止めて、最速で記録運用を開始する。
ツールの完成度よりも「データの蓄積を 1 日でも早く始める」ことを優先する判断。

プロトタイプの設計知見は、将来の本格ツール開発時に活きる。

**判断基準**:
- プロトタイプが 1〜2 週間以内に現場投入できる状態なら → 選択肢 1
- それ以上かかりそうなら → 選択肢 2 で即日開始し、並行でプロトタイプを改修

### 提案 D のロードマップ修正

```
Week 1:
  ├─ 既存 Excel データの棚卸し（何が使えるか確認）
  ├─ 現場ヒアリング（工数内訳の把握）

  └─ 記録ツールのデータ設計確定（07 の Must/Should 反映）
         │
Week 3-4:
  ├─ 記録ツール現場展開・運用開始
  └─ 抜取検査計画の変更提案
         │

```
