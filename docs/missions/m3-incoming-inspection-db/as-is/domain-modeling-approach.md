## 1. 現状認識

### Excelデータから判明した事実

Session 19-23でExcel（受入検査作業集計）を分析した結果:

| 項目 | 判明した事実 |
|------|-------------|
| **記録範囲** | 杉山さんのみExcel記録、他は紙記録 |
| **データ量** | 574件（2024-09〜2026-03、約18ヶ月） |
| **月間工数** | 最大67時間/月（概算300時間とは乖離） |
| **品名** | 約220種、表記揺れあり（名寄せで統合可能） |
| **検査内容** | 外観/外観+寸法/外観+通電など約40種 |
| **作業者** | 西村・杉山が主、複数人表記あり |

### README.mdの仮説とのギャップ

| README.mdの想定 | 実際のExcel | ギャップ |
|----------------|-------------|----------|
| lot_id（ロットID） | **存在しない** | 1行=1検査記録、ロット概念なし |
| supplier_id（サプライヤ） | **存在しない** | サプライヤ情報なし |
| po_number（発注番号） | **存在しない** | 発注との紐付けなし |
| fw_version / hw_version | **存在しない** | サイレントチェンジ監視不可 |
| sample_qty（サンプル数） | **存在しない** | 抜取検査の記録なし |
| inspection_items（検査項目マスタ） | 検査内容列に自由記述 | マスタ化されていない |

**結論**: README.mdの設計は「あるべき姿」であり、現行Excelとは大きく異なる。

## 2. ドメインモデリングの方針

### 2つのモデルを分けて考える

| モデル | 目的 | 優先度 |
|--------|------|--------|
| **As-Isモデル** | 現行Excel運用の再現 | 高（まずここから） |
| **To-Beモデル** | 理想的な品質管理DB | 中（段階的に移行） |

### Step 1: As-Isモデルの作成（今回〜次回）

現行Excelをそのままモデル化する。

```
[検査記録] ← メインエンティティ
    ├── 品名（表記揺れあり）
    ├── 作業者（複数人の場合あり）
    ├── 検査内容（自由記述）
    ├── 入荷日（欠損あり）
    ├── 検査完了日
    ├── 工数（分）
    └── カテゴリ（メカ/エレキ/Api）
```

**ユビキタス言語候補**（Excelから抽出）:

| 用語 | Excelでの表現 | 定義（仮） | 確認状態 |
|------|--------------|-----------|---------|
| 検査記録 | 1行 | 1回の検査作業の記録 | 確定 |
| 品名 | 品名列 | 検査対象の部品名 | 確定 |
| 作業者 | 担当者列 | 検査を実施した人 | 要確認（複数人の扱い） |
| 検査内容 | 検査内容列 | 実施した検査の種類 | 要確認（マスタ化の是非） |
| 入荷日 | 入荷日列 | 部品が入荷した日 | 要確認（空欄の理由） |
| 工数 | 工数(分)列 | 検査にかかった時間 | 確定 |
| カテゴリ | メカ/エレキ/Api | 部品の分類 | 要確認（誰が分類？） |

### Step 2: ヒアリングで確認すること

As-Isモデルを固めるために確認が必要な項目:

| # | 確認事項 | 対象者 |
|---|---------|--------|
| 1 | 複数人で検査する場合のルール（工数はどう記録？） | 西村/杉山 |
| 2 | カテゴリ（メカ/エレキ/Api）は誰がどう決める？ | 石川 |
| 3 | 入荷日が空欄のケース（紙記録分？作業依頼分？） | 西村/杉山 |
| 4 | 紙記録の内容はExcelと同じ項目？ | 西村 |
| 5 | 「不合格」の場合の記録方法は？ | 石川 |

### Step 3: To-Beモデルの検討（ヒアリング後）

As-Isモデルが固まったら、To-Be（理想形）との差分を検討:

| 追加すべき概念 | 目的 | 優先度 |
|---------------|------|--------|
| ロット | 同一発注・入荷単位での管理 | 高 |
| サプライヤ | サプライヤ別品質分析 | 高 |
| 発注番号 | トレーサビリティ | 中 |
| FW/HWバージョン | サイレントチェンジ検出 | 中 |
| 検査項目マスタ | 検査内容の標準化 | 中 |

## 3. プロトタイプ作成の判断基準

### プロトタイプを作るタイミング

| 条件 | 状態 | 作れる？ |
|------|------|---------|
| As-Isモデルが固まっている | △（ヒアリング未実施） | 一部可 |
| ユビキタス言語が合意されている | △（確認待ち） | 一部可 |
| 画面・機能の優先度が決まっている | × | 不可 |

### 今作れるプロトタイプの範囲

| 機能 | 作れる根拠 | 作れない理由 |
|------|-----------|-------------|
| **分析ダッシュボード** | CSVデータ構造が確定、分析ロジック完成 | - |
| **CSVインポート** | 変換・名寄せ・クレンジング済み | - |
| データ入力フォーム | - | As-Isモデル未確定、紙→デジタル移行の要件不明 |
| マスタ管理 | - | To-Beモデルの検討が必要 |

**結論**: 分析ダッシュボードはドメインモデル確定前でも作成可能だが、入力系は待つべき。
