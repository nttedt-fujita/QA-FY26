# EARS形式 要求仮説

**作成日**: 2026-03-04 (Session 7)
**更新日**: 2026-03-06 (Session 32) — Excel分析結果（Session 19-30）を反映

---

## 上位の要求（Need）

> 「自社製品の品質を定量的に保証できる体制を作る」（Session 4で合意）

この Need を実現するために、M3（受入検査DB）・M4（工程不良DB）が必要。

---

## ステータスの定義

| ステータス | 意味 |
|-----------|------|
| **確認済み** | 石川さん・チームとの会話で合意が取れている |
| **事実ベース** | Excelの客観的事実から直接導出（反論の余地が小さい） |
| **数値確認済み** | Session 19-30のデータ分析で数値的に確認 |
| **仮説** | 事実から推測した要求。ヒアリングで検証が必要 |

## 1. データ記録の構造化（Ubiquitous）

### R-01: 統一された列構成でデータを記録する

> **システムは**、検査対象の種別（メカ/エレキ/Api）に関わらず、**統一された項目で受入検査結果を記録する**

- **ステータス**: 数値確認済み
- **根拠**:
  - Excel課題A — メカ/エレキ/Apiで列構成が異なる
  - Session 19: CSV変換時に6シートのマッピングが必要だった

### R-02: 検査数量・検査方法・作業者を分離して記録する

> **システムは**、検査数量（数値）、検査方法（全数/抜取/外観等）、作業者（誰が）を**それぞれ独立した項目として記録する**

- **ステータス**: 数値確認済み
- **根拠**:
  - Excel課題B — 検査数量列に87パターンの異なる表記が混在
  - Session 21: 名寄せルール39件を作成したが、完全な正規化は不可能

### R-03: 検査結果と備考情報を分離して記録する

> **システムは**、合否判定、不良の詳細、対処内容、測定結果、申し送り事項を**それぞれ独立した項目として記録する**

- **ステータス**: 数値確認済み
- **根拠**:
  - Excel課題F — 「検査結果・備考」1列に6種類の情報が混在
  - Session 30: 「良　ピン跡に黒ずみ（不問）」など、良判定に懸念事項が埋もれている

### R-04: 作業者・サプライヤを一意に識別できる

> **システムは**、作業者およびサプライヤを**表記揺れなく一意に識別できる**形で記録する

- **ステータス**: 数値確認済み
- **根拠**:
  - Session 21: 品名の名寄せルール18件を作成
  - 「金森産業（株）」「金森産業株式会社」など複数表記が存在

### R-05: 不良数量を正確な数値で記録する

> **システムは**、不良数量を**数値として記録する**（「保留」「1個不足」「12個くらい」等のテキストではなく）

- **ステータス**: 数値確認済み
- **根拠**:
  - Session 30: 不良数量の記録が不完全（空欄でも備考に不良記述あり、「保留」など数値以外の値）
  - 不良品数を正確にカウントできない状態

## 2. データの紐付け・追跡（Event-driven）

### R-06: 受入検査で不良が発生したとき、その後の対応を追跡できる

> **When** 受入検査で不良が検出されたとき、**システムは**対応ステータス（要協議/対処中/完了）と対処内容を**記録・追跡する**

- **ステータス**: 事実ベース
- **根拠**:
  - Excel課題F/H — 「要協議」「再発防止依頼要」等のアクション項目が記録されるが、追跡されていない
  - 不具合発生シートには日付・数量・対策ステータスがない

### R-07: 受入検査データと工程不良データを紐付けできる

> **When** 工程で不良が発生したとき、**システムは**その部品の受入検査記録を**特定・参照できる**

- **ステータス**: 確認済み
- **根拠**: 石川さん「紐づいているのが理想」（ヒアリングメモ#01）
- **現状の課題**: 現行Excelにはロット概念がない（Session 25で確認）

### R-08: 不具合発生の日付・数量・対策を記録する

> **When** 不具合が発生したとき、**システムは**発生日、数量、原因分類、対策内容、対策完了日を**記録する**

- **ステータス**: 事実ベース
- **根拠**:
  - Excel課題H — 不具合発生シートに日付・数量・対策ステータスがない
  - 「いつ」「何件」「どう対処されたか」を追跡できない

## 3. 分析・可視化（State-driven / Event-driven）

### R-09: サプライヤ別・品目別の不良傾向を分析できる

> **When** 品質担当者が分析を行うとき、**システムは**サプライヤ別・品目別の不良件数・不良率の推移を**表示できる**

- **ステータス**: 数値確認済み（Phase 1で実現可能）
- **根拠**:
  - Session 19-23: 月別・品名別の集計ツールを実装済み
  - 名寄せ処理により集計精度が向上
- **Phase 1で実現**: ○

### R-10: 外注業者の変更前後で品質傾向を比較できる

> **When** 組み立て外注業者が変更されたとき、**システムは**変更前後の品質データを**比較表示できる**

- **ステータス**: 確認済み
- **根拠**: 石川さん「外注業者が変わっても、傾向を分析で判断できるようにしたい」
- **現状の課題**: 現行Excelには外注業者の情報がない（M4の範囲）

### R-11: 作業者別の検査傾向を分析できる

> **When** 品質担当者が分析を行うとき、**システムは**作業者別の検査件数・工数を**表示できる**

- **ステータス**: 数値確認済み（Phase 1で実現可能）
- **根拠**:
  - Session 19: 作業者別傾向を分析済み（西村→杉山の引き継ぎを確認）
  - 作業者によって平均工数が異なる（西村30分/件、杉山80-130分/件）
- **Phase 1で実現**: ○

### R-12: 月別の検査工数・件数を把握できる

> **When** 品質担当者が分析を行うとき、**システムは**月別の検査件数・工数の推移を**表示できる**

- **ステータス**: 数値確認済み（Phase 1で実現可能）
- **根拠**:
  - Session 20: 月別サマリーを出力済み（2024-09〜2026-03）
  - 2025-12〜2026-02に急増（56〜67時間/月）を確認
- **Phase 1で実現**: ○

### R-13: 検査内容別の工数を把握できる

> **When** 品質担当者が分析を行うとき、**システムは**検査内容別の平均工数を**表示できる**

- **ステータス**: 数値確認済み（Phase 1で実現可能）
- **根拠**:
  - Session 19: 検査内容別の工数を集計済み
  - 外観のみ34分、外観+通電750分、動作確認1000分
- **Phase 1で実現**: ○

## 4. 運用制約（Unwanted / Optional）

### R-14: 複数の製品種別に対応する

> **システムは**、ドローン本体およびプロポ（コントローラー）を含む**複数の製品種別の検査データを管理する**

- **ステータス**: 確認済み
- **根拠**: 石川さん「ドローン本体だけでなく、プロポの組み立てもある」

### R-15: 検査基準が未整備でも検査記録を残せる

> **If** 検査基準書が未整備の部品であっても、**システムは**検査記録と「基準書未整備」というステータスを**記録できる**

- **ステータス**: 事実ベース
- **根拠**:
  - Excel課題G — エレキシートは「検査基準書は無い、現在は図面が頼り」と明記
  - 基準書がないまま検査している現実がある

### R-16: 「不問」判定の根拠を記録できる

> **If** 不良を検出したが「不問」と判定する場合、**システムは**判定者と判定根拠を**記録する**

- **ステータス**: 事実ベース
- **根拠**:
  - Excelメカシート — 「ピン跡に黒ずみ（不問）」「機能部品なので見た目は不問にしたい」等
  - 不問判定が散見されるが根拠が記録されていない

### R-17: タブレットから簡単に入力できる

> **システムは**、検査現場で**タブレットから最小限の操作で検査結果を入力できる**

- **ステータス**: 確認済み（方向性）
- **根拠**: 藤田さん「大量に検査するなら簡単さ・シンプルさがかなり大事」
- **Phase**: Phase 2（ヒアリング後）

### R-18: クラウド上でデータを管理する

> **システムは**、検査データを**クラウド上に保存し、ネットワーク経由でアクセスできる**

- **ステータス**: 確認済み
- **根拠**: 石川さん「クラウドはあった方が良い」
