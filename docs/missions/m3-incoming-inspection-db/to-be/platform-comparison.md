## 判断の論点

### 1. 「品質Gメンバーが保守できるか」

| 選択肢 | 藤田が不在でも動くか | 変更・拡張できるか |
|--------|-------------------|------------------|
| A | ◎ 動く | ◎ ノーコードで可能 |
| B | ○ kintone部分は動く | △ Python部分は藤田のみ |
| C | △ サーバが動いていれば | × 開発スキル必要 |

### 2. 段階的アプローチの可能性

```
Phase 1: kintone単体で記録・閲覧を開始（数日で稼働）
    ↓
Phase 2: Python分析を追加（API連携で高度な分析）
    ↓
Phase 3: 必要に応じて自前開発に移行（or kintone+分析で十分ならそのまま）
```

**この段階的アプローチなら、リスクを最小化しながら進められる。**

## 推奨案

**選択肢B「kintone + 外部分析」の段階的アプローチを推奨**

理由:
1. **既存資産の活用**: kintoneが社内導入済み → 初期コスト最小
2. **早期に成果**: Phase 1（kintone単体）は数日で稼働 → 早く「動くもの」を見せられる
3. **品質管理に必要な分析も対応**: SPC・パレートはPython連携で実現
4. **保守リスクの分散**: kintone部分は誰でも保守可能
5. **判断の先送り回避**: 自前開発の判断はPhase 2の結果を見てから決められる

**石川さんへの説明時のポイント**:
- 「まずkintoneで小さく始めて、必要に応じて分析機能を追加する」
- 「全部自前開発するリスク（工数・保守）を避けつつ、品質管理に必要な機能は確保する」
