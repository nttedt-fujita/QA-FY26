**ステータス**: ドラフト

## 背景

### 現状の課題

| 課題 | 詳細 |
|------|------|
| 手作業による転記ミス | u-center画面を見てExcelに手入力 |
| 標準化されていない | RTK FIX時間測定など、定まった手法がない |
| 比較が困難 | 複数装置の結果を並べて見れない |

### 小板橋さんからのコメント（Session 16）

> 「自動化できるものは自動化したい」
> 「ばらつきなく評価できるようにしたい。確立できれば誰でもできるように」

## 要求一覧（EARS記法）

### N1: 自動検査の開始

**Event-driven**:
> ユーザーが「検査開始」ボタンを押したとき、システムは接続されたGNSS装置に対して検査を自動的に開始する。

### N2: 接続確認

**Event-driven**:
> GNSS装置がUSB経由で接続されたとき、システムは装置を自動検出し、シリアル番号を表示する。

### N3: 屋内検査（受入検査）

**Ubiquitous**:
> システムは、屋内で確認可能な項目（通信疎通、FWバージョン、設定パラメータ）を検査する。

### N4: 検査結果の保存

**Event-driven**:
> 検査が完了したとき、システムは結果をSQLiteに保存する。

### N5: レポート出力

**Event-driven**:
> ユーザーがレポート出力を要求したとき、システムはPDFおよびCSV形式でレポートを生成する。

### N6: ファイル名にシリアル番号を含める

**Ubiquitous**:
> システムは、出力ファイル名にGNSS装置のシリアル番号を含める。

### N7: 過去データの再取得

**Event-driven**:
> ユーザーが過去の検査結果を検索したとき、システムはSQLiteから該当データを取得して表示する。

### N8: 複数装置の比較

**Event-driven**:
> ユーザーが複数の検査結果を選択したとき、システムは結果を並べて比較表示する。

### N9: 複数台同時接続

**Ubiquitous**:
> システムは、複数のGNSS装置（2〜5台程度）を同時に接続し、検査を並行実行する。

### N10: 認知負荷の軽減

**Ubiquitous**:
> システムは、各装置の状態を一覧で表示し、オペレーターの認知負荷を軽減する。

## スコープ（対象範囲）

### 含まれるもの

| 項目 | 備考 |
|------|------|
| 受入検査（屋内） | 通信疎通、FW確認、設定確認 |
| 複数台同時接続 | 2〜5台程度 |
| データ保存 | SQLite |
| レポート出力 | PDF + CSV |
| 過去データ検索・比較 | SQLiteから再取得 |

### 含まれないもの（一旦保留）

| 項目 | 理由 |
|------|------|
| RTK FIX時間測定 | 受入検査ではやらない |
| 屋外での受信感度測定 | 別フェーズで検討 |
| スペアナ波形取得 | 別フェーズで検討 |

---

## 屋内で確認可能な項目

| 確認項目 | UBXメッセージ | 備考 |
|----------|-------------|------|
| 通信疎通 | 応答確認 | 繋がるか |
| FWバージョン | UBX-MON-VER | 期待値と一致するか |
| 設定パラメータ | UBX-CFG-* | 期待値と一致するか |
| チップシリアル | UBX-SEC-UNIQID | 装置識別 |

---

## 運用想定

| 項目 | 想定 |
|------|------|
| 使用者 | 品質G（小板橋さん他）|
| 使用場所 | 屋内（評価室）|
| 接続方法 | USB経由 |
| 同時接続台数 | 2〜5台 |
| 1日の検査台数 | 数台〜10台程度（想定）|
