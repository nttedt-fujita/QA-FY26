## 1. 背景

### 1-1. 現状の課題

小板橋さんヒアリング（Session 16）より：

| 作業 | 現状 | 課題 |
|------|------|------|
| 受信感度（C/N0）記録 | u-center画面を見てExcelに手入力 | **手間、転記ミスリスク** |
| 衛星位置記録 | GNSSView画像をキャプチャ | 仰角・方角の数値化が困難 |
| RTK FIX時間測定 | ストップウォッチ計測 | **定まった手法がない** |
| スペアナ波形確認 | u-center画面キャプチャ | 比較が困難 |
| ジャミング判定 | 目視確認 | 定量的でない |

→ Phase 1（検証項目の妥当性検証）が先。ツール設計はPhase 2。

## 2. 自動化可能な項目

Session 17でUBXプロトコル仕様を確認した結果、以下の項目が自動化可能：

### 2-1. 自動化マッピング

| 現状（手作業） | 自動化可能なUBXメッセージ | 取得データ |
|-------------|------------------------|-----------|
| 受信感度を画面見て手入力 | **UBX-NAV-SIG** | L1/L2別、全衛星のC/N0（dBHz） |
| RTK FIX時間をストップウォッチ計測 | **UBX-NAV-STATUS** | ttff（初期FIX時間、ミリ秒） |
| | **UBX-NAV-PVT** | carrSoln監視でRTK状態遷移検出 |
| スペアナを画像キャプチャ | **UBX-MON-SPAN** | 256点スペクトラム数値データ |
| ジャミング判定を目視 | **UBX-MON-RF** | jammingState（0〜3の数値） |
| 水平・高さ精度を画面読み | **UBX-NAV-PVT** | hAcc / vAcc |
| 衛星数を画面読み | **UBX-NAV-PVT** | numSV |

各メッセージの詳細仕様は [08-ubx-protocol-index.md](08-ubx-protocol-index.md) を参照。

## 3. 通信経路の選択肢

### 3-1. 経路1: F9Pと直接通信（u-center代替）

```
[評価ツール] --UART--> [F9Pチップ]
```

**メリット**:
- 最も直接的なデータ取得
- UBXプロトコルをフル活用可能
- u-centerと同等の情報を取得

**デメリット**:
- 機体からF9Pを外す必要がある場合あり
- PX4との同時接続の考慮が必要

### 3-2. 経路2: PX4フライトログ経由

```
[F9P] --UBX--> [PX4] --uORB--> [SD/MAVLink] --> [ログ解析ツール]
```

**メリット**:
- 飛行中のデータも取得可能
- 既存のログ解析基盤を活用

**デメリット**:
- PX4が全UBXメッセージをuORBに変換しているわけではない
- L1/L2別のC/N0など、一部情報が欠落する可能性

### 3-3. 経路3: MAVLink経由（GCS連携）

```
[F9P] --UBX--> [PX4] --MAVLink--> [GCS/評価ツール]
```

**メリット**:
- リアルタイム監視が可能
- Mission Planner等との連携

**デメリット**:
- MAVLinkで伝送される情報に制限あり

## 4. ツール化の候補構成

### 4-1. 静的確認ツール（u-center代替）

**目的**: 受信感度、スペアナ、設定確認の自動化

**構成案**:
```
Python + pyserial
  |
  +-- UBXプロトコル解析ライブラリ
  |     - UBX-NAV-SIG（受信感度）
  |     - UBX-MON-SPAN（スペアナ）
  |     - UBX-CFG-*（設定確認）
  |
  +-- CSV/JSON出力
  |
  +-- 可視化（matplotlib or plotly）
```

### 4-2. 動的確認ツール（フライトログ解析）

**目的**: RTK状態遷移、飛行中精度の自動解析

**構成案**:
```
PX4 ULogファイル
  |
  +-- pyulog（既存ライブラリ）
  |
  +-- sensor_gps, vehicle_gps_position 抽出
  |
  +-- RTK状態遷移グラフ
  +-- 精度時系列グラフ
```
