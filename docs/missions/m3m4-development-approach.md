## 前提

- **作業者**: ふじた 一人

- **利用者**: 品質G全員（入力・閲覧）

## 1. 開発サイクル

各フェーズを順に進め、**フェーズごとに石川さんのレビュー・承認**を得る。

```
要求確認 → 要件定義 → 基本設計 → 詳細設計 → 実装
   ↑                                           │
   └───────── フィードバック ←─────────────────┘
```

| フェーズ | 目的 | 成果物 | レビュー対象者 |
|---------|------|--------|--------------|
| **要求確認** | Needと業務の理解 | ヒアリング結果、ドメインモデル、ユビキタス言語集 | 石川さん、小板橋さん |
| **要件定義** | 何を作るか決める | 要件定義書（機能/非機能）、画面イメージ | 石川さん |
| **基本設計** | どう作るか決める | アーキテクチャ構成図、技術選定結果、コスト試算 | 石川さん、小笠原課長 |
| **詳細設計** | 作れるレベルに落とす | ER図、API設計、画面設計 | 石川さん |
| **実装** | 動くものを作る | プロトタイプ → MVP | 品質G全員（試用） |

### フェーズ移行の条件

- **前フェーズの成果物がレビュー済みであること**
- 大きな方針変更があれば前フェーズに戻る（手戻りは早いほど安い）

## 2. フィードバックサイクル（アジャイル的運用）

一人作業だが、**細かく打ち返してもらう**ことで方向のズレを早期に検出する。

### 打ち合わせ周期の提案

| 種類 | 周期 | 時間 | 内容 | 参加者 |
|------|------|------|------|--------|
| **定例報告** | 週1回 | 15分 | 進捗報告、次週の予定、困りごと | 石川さん |
| **レビュー会** | フェーズ完了時 | 30-60分 | 成果物のレビュー、方針判断 | 石川さん（+必要に応じて課長） |
| **ユーザーテスト** | プロトタイプ完成時 | 30分 | 実際に触ってもらう、フィードバック収集 | 品質G全員 |

### レビュー資料のフォーマット

各レビューで提出する資料の統一フォーマット:

```
# [フェーズ名] レビュー資料

## 前回からの変更点
- 変更1
- 変更2

## 今回の成果物
（成果物の本体 or リンク）

## 確認したいこと
- [ ] 質問1
- [ ] 質問2

## 次のフェーズでやること
- アクション1
- アクション2
```

**ポイント**: 「確認したいこと」を明示する → 石川さんが何に答えればよいか分かる

## 3. ドメインモデリング（要求確認フェーズの中心作業）

DB設計の前に、**業務の概念と言葉を整理する**。

### なぜドメインモデリングから始めるか

- ソフトじゃない人と**同じ言葉**で話せるようになる
- テーブル設計は「概念が整理された後」にやる方が手戻りが少ない
- 現場の業務フローを理解せずにDB設計すると、使いにくいシステムになる

### 進め方

```
Step 1: ユビキタス言語の収集
  └── 現場で使われている言葉を集める（ヒアリング）
      例: 「ロット」「検収」「NG品」「再検査」...

Step 2: ドメインモデルの構築
  └── 概念とその関係を図にする（クラス図的なもの）
      例: ロット --含む--> 部品、ロット --検査される--> 検査結果

Step 3: 業務フローの整理
  └── 現在の業務の流れを可視化する
      例: 入荷 → 数量確認 → 抜取 → 検査 → 合否判定 → 記録

Step 4: レビュー
  └── 石川さん・小板橋さんに「この理解で合ってますか？」
```

### ユビキタス言語集のフォーマット

| 用語 | 定義 | 備考 |
|------|------|------|
| ロット | 同一発注・同一入荷の部品の単位 | lot_idで一意に識別 |
| 検収 | ？ | ヒアリングで確認 |
| ... | ... | ... |

**重要**: この言語集はDB設計・画面設計・マニュアルすべてで使う共通語彙になる

## 4. 全体スケジュール（1Q詳細）

```
Week 1-2: 要求確認（ヒアリング・ドメインモデリング）
  ├── 現行Excel運用のヒアリング（受入検査・工程不良）
  ├── ユビキタス言語の収集
  ├── 業務フローの可視化
  └── 【成果物】ユビキタス言語集、ドメインモデル図、業務フロー図

Week 3-4: 要件定義
  ├── 機能要件の整理（MECEで分解）
  ├── 非機能要件の整理（AWS前提の場合の制約）
  ├── kintone vs 自前開発の比較検討
  └── 【成果物】要件定義書

Week 5-6: 基本設計
  ├── アーキテクチャ設計（AWS構成案）
  ├── コスト試算
  ├── 保守範囲の定義（自分でやる / 外注する）
  └── 【成果物】アーキテクチャ構成図、コスト試算書

Week 7-8: 詳細設計
  ├── ER図（ドメインモデル → 論理モデル → 物理モデル）
  ├── API設計
  ├── 画面設計（ワイヤーフレーム）
  └── 【成果物】ER図、API仕様、画面設計書
```

## 5. 体制と保守の考慮（概要）

### 一人でやる範囲 vs 外注する範囲

| フェーズ | 一人でやる | 外注候補 |
|---------|-----------|---------|
| 要求確認〜要件定義 | ◎ 自分がやるべき | - |
| 基本設計 | ◎ 自分がやるべき | - |
| 詳細設計 | ○ プロトタイプレベル | 本番設計はレビュー依頼 |
| 実装（プロトタイプ） | ◎ 自分で作る | - |
| 実装（本番） | △ スコープ次第 | 本番化は外注が健全 |
| 保守・運用 | △ 簡易なもののみ | インフラ運用は外注推奨 |
