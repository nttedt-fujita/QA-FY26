# kintone vs AWS自前開発 比較レポート

**作成日**: 2026-03-05
**セッション**: Session 9
**目的**: M3/M4のプラットフォーム選定における判断材料

---

## エグゼクティブサマリー

| 観点 | 結論 |
|------|------|
| 記録・閲覧だけなら | kintoneで十分 |

## 2. kintoneの制限値

| 項目 | 制限値 |
|------|--------|
| 登録レコード数 | 無制限（100万件で動作確認済み） |
| アプリ内フィールド数 | 500個 |
| テーブル内行数 | 5,000行 |
| 画像ファイル | 10MB以下（超えるとサムネイル非表示） |
| 一度に取得できるレコード | 500件 |
| 一度に登録・更新・削除 | 100件 |
| 1日のAPIリクエスト数 | 10,000件 |
| 1アプリのプラグイン数 | 20個 |

**Source**: [kintone 制限値いろいろ - Qiita](https://qiita.com/rex0220/items/9e248db42621f3d9bf14)

## 3. kintoneカスタマイズの限界

### 3.1 カスタマイズ方法

| 方法 | 内容 | リスク |
|------|------|--------|
| プラグイン | 機能追加（最大20個/アプリ） | 有料のものが多い |
| JavaScript | 画面・動作カスタマイズ | **バージョンアップで動かなくなるリスク** |
| API連携 | 外部システム連携 | 1日10,000リクエスト制限 |

### 3.2 JSカスタマイズのリスク（原文抜粋）

> JavaScriptカスタマイズのデメリットとしては、「ハードルの高さ」や「開発期間」の問題が挙げられ、また「**kintoneのバージョンアップでうまく動作しなくなる**」などのリスクも抱えています。

**Source**: [kintone(キントーン)のカスタマイズ方法を考えよう](https://www.rshd.co.jp/news/kintone-1.html)

## 5. kintone + Python案の構成

```
┌─────────────┐      API      ┌─────────────┐
│   kintone   │ ───────────→  │   Python    │
│  （入力）    │  データ取得   │  （分析）    │
└─────────────┘              └─────────────┘
       ↑                           ↓
    ユーザー                   グラフ・レポート
    （入力）                   （別画面で見る）
```

**問題点**:
- 入力画面と分析画面が**分離**
- kintoneのAPI制限（1日10,000リクエスト、1回500件）
- 分析結果をkintoneに戻すのが面倒
- **結局Python部分は自分で保守** → 「保守がサイボウズ任せ」の利点が半減

---

## 6. AWS自前開発案の構成

```
┌────────────────────────────────────────┐
│            自作アプリ                   │
│  ┌──────────┐    ┌──────────────────┐ │
│  │  入力     │    │  分析ダッシュボード  │ │
│  │  フォーム  │    │  ・管理図          │ │
│  │          │    │  ・パレート図       │ │
│  │          │    │  ・SPC             │ │
│  └──────────┘    └──────────────────┘ │
│           ↓              ↑             │
│         ┌─────────────────┐           │
│         │      DB         │           │
│         └─────────────────┘           │
└────────────────────────────────────────┘
```

**メリット**:
- 入力と分析が**同じアプリ内**
- 分析ロジックを**自由に組み込める**
- データの流れが**シンプル**
- 将来の拡張も自由

|------|---------|------------------|-------------|

## 8. 結論

### 「分析が肝」なら自前開発

- kintoneは「記録・閲覧」は得意だが「分析」は弱い
- kintone + Pythonでも分析は可能だが、**入力と分析が分離**する
- **分析をプロダクトの中心に据えるなら、最初から分析前提で設計できる自前開発が適切**

### ただし最終判断は保留

- **プロトタイプを作ってヒアリングしながら決める**のが正しいアプローチ
- 「SPC・管理図は不要」と判明したらkintone単体で済む可能性もある
- 今の段階で決め打ちしない
