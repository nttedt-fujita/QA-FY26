## プロジェクト分類

**業務システム開発（社内DXプロジェクト）**

品質保証業務のデジタル化を目的とした、要件定義・設計・実装・運用設計を含むフルスタック開発案件。外注先の現場作業者と社内品証チームの双方が利用するクラウドベースの品質管理システム。

## 技術選定の基本方針

### AIがコードを書く時代の技術選定

特定言語の実装経験よりも「なぜその技術を選んだか」の判断力が重要。本プロジェクトの技術選定は、制約条件（一人開発、他ミッション並行、AI支援活用、保守引き継ぎ前提）から合理的に導出する。選定根拠を説明できること自体がポートフォリオとしての価値。

### 構成方針

- 制約条件から最も合理的な構成を選ぶ（学習目的で技術を選ばない）
- サーバーレスAPI + マネージドDB で保守負荷を最小化する
- AI（Cursor / Claude）で開発速度を補う前提で、一人でも現実的な構成にする
- 退職後に外部エンジニアが引き継げるよう、標準的な技術を選定する
- データ膨張予測を踏まえ、現在の規模に適切なサイズ感で構築する
- 将来拡張はレイヤーを分離して対応する（現在のシステムを肥大化させない）

## データ膨張予測

※ 詳細なシナリオ別試算は `qa_db_data_growth_analysis_20260305.md` を参照

### サマリー

3シナリオ（保守的/成長/急成長）で試算した結果、全シナリオで RDS PostgreSQL で対応可能。

| シナリオ | 10年累計レコード数 | DBサイズ | 推奨インスタンス | 月額 |
|---|---|---|---|---|
| 保守的（現状維持） | 約30万件 | 約150MB | db.t4g.micro | $12〜18 |
| 成長（年50%成長+出荷検査） | 約300万件 | 約1.5GB | micro → small | $12〜25 |
| 急成長（年100%成長+全拡張） | 約3,500万件 | 約17.5GB | small → medium | $25〜50 |

- Aurora Serverless v2 が必要になるシナリオは存在しない
- DynamoDBの採用理由もない（コスト差が小さく、リレーション設計の複雑化が上回る）
- RDSインスタンスのスケールアップは数分で完了するため、小さく始めて段階対応が最も合理的
- フライトログ・センサーデータは年間数億〜数十億ポイントになりうるため、品質DBとは完全にレイヤー分離（S3+Athena）

### 試算の前提値はヒアリングで要検証

現在の仮定（月間入荷50回、組立100台、検査10項目等）は全て未検証。特に以下がデータ量を大きく左右する。

- 事業成長率（3年・5年の生産計画）
- データ保持期間（PL法考慮で10年以上の可能性）
- 画像撮影方針（不良のみ vs 全件で桁が変わる）
- 検査項目数の増加予定（規格対応等）

## 推奨技術スタック（一次選定）

### フロントエンド

| 項目 | 選定 | 選定理由 |
|---|---|---|
| フレームワーク | Next.js（App Router） | React系で情報量最多、AI支援との相性良好、SSR/SSG柔軟 |
| UIライブラリ | shadcn/ui + Tailwind CSS | コンポーネント単位導入、カスタマイズ性高 |
| タブレット対応 | PWA（Progressive Web App） | 下記参照 |
| ホスティング | AWS Amplify Hosting | Next.js統合容易、CI/CD自動化 |

**PWAを選ぶ理由:** 一人開発でネイティブアプリ（iOS/Android）の2コードベース維持は非現実的。PWAならブラウザベースでホーム画面追加・オフラインキャッシュが可能。ストア審査不要で外注先にも即配布可能。端末OS（iOS/Android）を選ばない。オフライン要件が重い場合のみネイティブを再検討。

### バックエンド

| 項目 | 選定 | 選定理由 |
|---|---|---|
| API | API Gateway + Lambda | サーバー管理ゼロ、従量課金で低コスト |
| ランタイム | Node.js（TypeScript） | 下記参照 |
| 認証 | Amazon Cognito | ユーザー管理・認証のフルマネージド、外注先グループ管理対応 |

**TypeScript統一の理由:** 「Goの経験を積むため」ではなく、制約条件からの合理的帰結。一人開発・他ミッション並行の状況でフロント・バックエンド・IaCの言語を統一し認知負荷を最小化することが最優先。型定義のフロント・バックエンド共有も可能。パフォーマンス差は日数百リクエスト規模では無意味。

### データベース

| 項目 | 選定 | 選定理由 |
|---|---|---|
| メインDB | **RDS PostgreSQL（db.t4g.micro）** | 下記参照 |
| 画像・大量データ | S3 | 不良品写真、将来のセンサーデータ |
| 将来の大量データ分析 | S3 + Athena（将来追加） | 現時点では未実装 |

**RDS PostgreSQL（db.t4g.micro）を選ぶ理由:**

- 受入検査⇔ロット⇔工程不具合の紐付けはRDBの得意領域
- SQLで集計・分析クエリが書きやすく、QuickSightとも直結
- 5年運用で3万件程度 → 最小インスタンスで十分な性能
- **Aurora Serverless v2を採用しない理由:** 最小ACU 0.5でアイドル時でも月$40〜50。自動スケーリングのメリットが活きない規模では単にコスト過剰。RDS db.t4g.microなら月$15程度
- DynamoDBを採用しない理由: リレーション（ロット⇔検査⇔不具合）が多く、DynamoDBのアクセスパターン設計が複雑になる

### 分析・可視化

| 項目 | 選定 | 選定理由 |
|---|---|---|
| ダッシュボード | Amazon QuickSight or アプリ内（Recharts等） | ヒアリング後に判断 |

QuickSight（Author $24/月）の要否は分析要件のヒアリング後に判断。簡易的な分析ならアプリ内ダッシュボードで十分な可能性あり。

### インフラ管理

| 項目 | 選定 | 選定理由 |
|---|---|---|
| IaC | AWS CDK（TypeScript） | TypeScript統一、CloudFormationより記述量少 |
| CI/CD | GitHub Actions + Amplify | push自動デプロイ、手動作業排除 |

### 開発ツール

| 項目 | 選定 |
|---|---|
| IDE | Cursor（AI支援付き） |
| AI支援 | Claude（設計壁打ち・コード生成・レビュー） |
| バージョン管理 | GitHub（privateリポジトリ） |
| タスク管理 | GitHub Issues or Notion |

## 開発計画（マイルストーン）

### 前提

- 開発は他ミッションと並行（週の50〜60%程度を想定）
- AI支援で開発速度を補うが、ヒアリング・現場検証は自分でやる
- **プロトタイプファースト** — さっさとヒアリングしながらプロトタイプを作り、現場付き添いでデータ取り・確認作業のサイクルを回す

### Phase 0: 準備・ヒアリング（〜2週間）

**目標:** ドメインモデルの確定、業務フローの把握

- [ ] 関係者ヒアリング（情報整理シートの未確認事項を埋める）
- [ ] 現場の作業フロー観察（受入検査・組立工程）
- [ ] 既存の記録方法・帳票の収集
- [ ] ロット定義の確定
- [ ] 将来拡張の構想確認（出荷検査、フライトログ等）
- [ ] 運用コスト予算の確認（将来拡張コスト含む）
- [ ] ER図ドラフト → ヒアリング結果で確定

**成果物:** ER図、業務フロー図、ロット定義文書、将来拡張方針

### Phase 1: プロトタイプ開発（2〜3週間）

**目標:** 動くものを作って現場で見せられる状態にする

- [ ] AWS環境セットアップ（CDKで構築）
- [ ] DBスキーマ作成（まずは受入検査テーブル）
- [ ] タブレット入力画面のプロトタイプ（受入検査）
- [ ] 基本的なCRUD API
- [ ] 認証の基本実装（Cognito）

**成果物:** 受入検査の入力・閲覧ができるプロトタイプ

### Phase 2: 現場検証 第1回（1〜2週間）

**目標:** 実際の作業者に使ってもらい、フィードバックを得る

- [ ] 外注先で作業者に付き添いながらプロトタイプ使用
- [ ] 入力のしやすさ、項目の過不足、動線の問題を確認
- [ ] 実データでDBの構造が破綻しないか検証
- [ ] フィードバックを整理・優先順位付け

**成果物:** フィードバック一覧、改善計画

### Phase 3: 改善 + 工程不具合機能追加（2〜3週間）

**目標:** フィードバック反映 + 工程不具合管理の実装

- [ ] Phase 2のフィードバック対応
- [ ] 工程不具合の入力・閲覧機能
- [ ] ロットを介した受入検査⇔工程不具合の紐付け表示
- [ ] 不良品写真のアップロード機能（S3連携）

**成果物:** 受入検査 + 工程不具合の両機能が動くシステム

### Phase 4: 現場検証 第2回 + 分析機能（2〜3週間）

**目標:** 本格運用前の最終検証 + 分析・レポート機能

- [ ] 両機能を通しで現場検証
- [ ] ダッシュボード構築（QuickSight or アプリ内）
- [ ] 基本分析ビュー（ロット別不良率、サプライヤー別傾向など）
- [ ] データエクスポート機能

**成果物:** 分析機能付きの運用可能なシステム

### Phase 5: ドキュメント整備・引き継ぎ準備（1〜2週間）

**目標:** 自分がいなくても運用・改修できる状態にする

- [ ] 運用マニュアル（管理者向け・作業者向け）
- [ ] システム構成図・アーキテクチャドキュメント
- [ ] IaCのREADME整備
- [ ] トラブルシューティングガイド
- [ ] 引き継ぎ先候補との共有

**成果物:** 運用ドキュメント一式

## リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 他ミッションで時間が取れない | スケジュール遅延 | Phase単位で動く成果物を出す。途中で止まっても部分的に使える状態を保つ |
| 外注先のネットワーク環境が悪い | タブレット入力不可 | PWAオフライン対応を早期検証。最悪オフライン入力→接続時同期 |
| ドメインモデルの手戻り | DB再設計 | Phase 0で徹底ヒアリング + Phase 2で早期にデータ検証 |
| RDSコストが想定超え | 予算オーバー | db.t4g.microで開始、必要時のみスケールアップ |
| 将来拡張で想定外の要件 | アーキテクチャ変更 | 現在のRDBとセンサー系を最初からレイヤー分離しておく |
| 退職時に引き継ぎ先がいない | システム放置 | Phase 5を省略しない。IaC + ドキュメントで外部委託も可能 |
