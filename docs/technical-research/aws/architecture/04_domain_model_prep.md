# 品質保証DB — ドメインモデル設計 情報整理シート

**日付:** 2026年3月5日  
**最終更新:** ヒアリング必須/現時点で決定可能を明確に分離、将来拡張シナリオ追加  
**目的:** DB設計に入る前に、把握済みの情報と未確認事項を整理し、抜け漏れを防ぐ

## 1. 現時点で決められること（ヒアリング不要）

以下は議論の中で方針が固まった、またはヒアリングなしで判断できる事項。

### 1-1. 業務の全体像（把握済み）

- 外注先でドローン本体・コントローラーを組み立てている
- 組み立て用パーツと保守用パーツの両方がある
- 受け入れ検査は抜取検査で実施
- 工程不具合の管理も必要

### 1-2. DB化の対象領域

- **受け入れ検査:** パーツ入荷時の品質確認記録
- **工程不具合:** 組み立て工程で発生した不具合の記録

### 1-3. 紐付け方針

- 抜取検査のため個体単位の紐付けは不可能 → **ロット単位で紐付ける**
- 受入ロットを自社で必ず採番し、サプライヤーロット番号は取得可能な場合に任意記録
- この方針はヒアリング結果でサプライヤーロット運用が判明しても破綻しない設計

### 1-4. アーキテクチャ方針

- **PWAファースト**（ネイティブアプリは不要と判明するまで作らない）
- **モジュラーモノリス**（マイクロサービスはこの規模・体制では過剰）
- **TypeScript統一**（フロント・バックエンド・IaC、一人開発の認知負荷低減）

### 1-5. データ膨張予測（現時点で判断可能な範囲）

- 受入検査: 5年で約3万件 → RDBとして極めて小規模
- 工程不具合: 5年で数千件 → 同上
- マスタ: 数百〜数千件で固定的
- **現時点のデータにはRDS PostgreSQL（最小インスタンス）で十分**
- Aurora Serverless v2は自動スケーリングのメリットが活きない規模でコスト過剰

### 1-6. 将来拡張の設計方針（現時点で織り込む）

- 出荷検査、フィールド故障、センサーデータ等の拡張可能性がある
- センサーデータが入ると年間数億レコード規模になりRDB単体では厳しい
- **現在のRDBはトランザクションデータ（検査・不具合）に専念させる**
- **大量データ（センサー・フライトログ）は別レイヤー（S3 + Athena等）で対応する前提**
- DBスキーマ設計時に将来テーブル追加しやすい構造にしておく（外部キーの設計等）

## 3. 想定されるコアエンティティ（仮）

ヒアリング結果によって変更の可能性あり。

- **サプライヤー** — 部品の購入先
- **パーツマスタ** — 部品の種類・型番・用途区分（組立用/保守用）
- **受入ロット** — 入荷単位の管理ID、サプライヤーロット番号（任意）
- **検査項目マスタ** — パーツごとの検査基準・合否基準
- **受入検査記録** — ロット単位の抜取検査結果
- **検査明細** — サンプルごとの個別測定値・判定
- **工程マスタ** — 組み立て工程の定義
- **工程不具合記録** — 不具合の内容・発生工程・原因分類・対処
- **機体/製品** — 完成品のシリアル管理（C-6の回答次第で追加）
