## 1. QuickSightとは

**AWSが提供するBI（Business Intelligence）ツール**。
データを可視化してダッシュボードを作れるサービス。

### 1.1 特徴

- **ノーコード/ローコード**でダッシュボード作成
- AWSサービス（S3, RDS, Athena等）と**直接連携**
- **AI機能**（異常検知、予測、自動サマリ）内蔵
- Webブラウザでアクセス可能

## 3. SPC・管理図はできるか？

### 3.1 結論

**できる**。ただし標準機能ではなく、**計算フィールド + 参照線**で実装する。

### 3.2 実装方法

QuickSightコミュニティで共有されている方法：

**UCL（上方管理限界）の計算式**:
```
avgOver({Days}, [], PRE_AGG) + (3 * stdevpOver({Days}, [], PRE_AGG))
```

**LCL（下方管理限界）の計算式**:
```
avgOver({Days}, [], PRE_AGG) - (3 * stdevpOver({Days}, [], PRE_AGG))
```

**ポイント**:
- `avgOver` / `stdevpOver` で全体の平均・標準偏差を計算
- `PRE_AGG` パラメータで集計前の値を使用
- Reference Line（参照線）として表示

**Source**: [Upper Control Limit and Lower Control Limit - QuickSight Community](https://community.amazonquicksight.com/t/upper-control-limit-and-lower-control-limit-for-statistical-control-analysis/42733)

### 3.3 管理図の実装イメージ

```
QuickSight上での管理図:

     UCL ─────────────────────  ← 参照線（計算フィールド）
                           ●
                      ●         ●
     CL  ────●───●───●───●────●────  ← 参照線（平均）
            ●       ●
     LCL ─────────────────────  ← 参照線（計算フィールド）
```

## 5. M3/M4でQuickSightを使う場合の構成

### 5.1 構成図

```
┌─────────────────────────────────────────────────────┐
│                  自作アプリ                          │
│  ┌──────────────┐                                   │
│  │  入力フォーム  │                                   │
│  │  （React等）  │                                   │
│  └──────┬───────┘                                   │
│         ↓                                           │
│  ┌──────────────┐                                   │
│  │   DB (RDS)   │                                   │
│  └──────┬───────┘                                   │
│         ↓                                           │
│  ┌──────────────┐                                   │
│  │  QuickSight  │ ← 分析・ダッシュボードはここで    │
│  │  ・管理図     │                                   │
│  │  ・パレート図 │                                   │
│  │  ・トレンド   │                                   │
│  └──────────────┘                                   │
└─────────────────────────────────────────────────────┘
```

### 5.2 データフロー

1. ユーザーが**入力フォーム**でデータ登録
2. データは**RDS（DB）**に保存
3. QuickSightがRDSに**直接接続**
4. ダッシュボードで**分析・可視化**

## 6. メリット・デメリット

### 6.1 QuickSightを使うメリット

| メリット | 説明 |
|---------|------|
| 開発コスト削減 | 分析UIを自作しなくていい |
| 保守コスト削減 | AWS任せ、自分で保守不要 |
| AI機能 | 異常検知・予測が使える |
| 高機能 | 多種多様なグラフ、フィルター、ドリルダウン |

### 6.2 QuickSightを使うデメリット

| デメリット | 説明 |
|-----------|------|
| 別サービス | 入力画面とダッシュボードが分離 |
| 学習コスト | QuickSightの使い方を覚える必要 |
| 月額コスト | +$25-30/月 |
| カスタマイズ限界 | 完全に自由ではない |

## 7. 比較: 分析も自作 vs QuickSight

| 観点 | 分析も自作 | QuickSight使う |
|------|-----------|---------------|
| 開発コスト | 高い | 低い |
| 保守コスト | 高い | 低い |
| カスタマイズ | ◎ 何でもできる | ○ ある程度 |
| UI統合 | ◎ 1アプリ内 | △ 別サービス |
| 月額コスト | 低い | +$25-30 |
| SPC・管理図 | 自作 | 計算フィールドで可 |

---

## 8. 結論

### 8.1 QuickSightは有効な選択肢

- **分析UIを自作しなくていい** → 開発コスト削減
- **管理図も実装可能**（計算フィールドで）
- **AI機能**も将来的に活用できる
- **月額$25-30は許容範囲**

### 8.2 ただし判断は保留

- 入力と分析が別サービスになる点は認識しておく
- **プロトタイプを作って使用感を確認**してから決める
- ヒアリングで分析要件が明確になってから最終判断
