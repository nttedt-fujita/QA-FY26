# 保守戦略の検討

**作成日**: 2026-03-04 (Session 5)
**前提**: 藤田さん一人体制での持続可能な運用

---

## 1. 問題の整理

Session 4で明確になった懸念:
- 「作った人が一生保守」を回避したい
- 藤田さんが異動・退職した場合にシステムが止まるリスク
- 品質Gメンバーはソフト開発スキルを持たない

---

## 2. 保守範囲の定義

### 保守の種類と担当

| 保守の種類 | 内容 | 頻度 | 担当候補 |
|-----------|------|------|---------|
| **日常運用** | データ入力、閲覧、レポート出力 | 毎日 | 品質G全員 |
| **軽微な変更** | フォーム項目追加、グラフ設定変更 | 月数回 | 品質Gメンバー（kintoneなら可能） |
| **障害対応** | アプリが動かない時の対処 | 年数回 | 藤田 or 外注 |
| **機能追加** | 新しい分析機能、新製品対応 | 四半期 | 藤田 or 外注 |
| **インフラ保守** | サーバ更新、セキュリティパッチ | 月1回 | kintone: 不要 / AWS: 藤田 or 外注 |
| **データバックアップ** | 定期バックアップ、リストア | 自動 | kintone: サイボウズ / AWS: 自動設定 |

## 3. 技術選定別の保守負荷

### A. kintone（+ 外部分析）の場合

```
kintone部分（保守ほぼゼロ）
├── インフラ: サイボウズが管理 → 保守不要
├── アプリ変更: ノーコード → 品質Gメンバーでも可能
├── バックアップ: サイボウズが管理 → 保守不要
└── セキュリティ: サイボウズが管理 → 保守不要

Python分析部分（保守あり）
├── 分析スクリプトの更新 → 藤田
├── ライブラリのアップデート → 藤田（年1-2回）
└── 実行環境の維持 → Lambda or ローカルPC
```

**年間保守工数の見積**: 約20-40時間/年（月2-3時間程度）

### B. 自前開発 (AWS) の場合

```
フロントエンド
├── フレームワーク更新（React/Vue等） → 年1-2回
├── UIの修正・改善 → 随時
└── ブラウザ対応確認 → 年1-2回

バックエンド
├── フレームワーク更新（FastAPI/Django等） → 年1-2回
├── セキュリティパッチ適用 → 月1回
├── API修正・機能追加 → 随時
└── ライブラリ依存関係の管理 → 月1回

インフラ
├── OS更新（Lightsailの場合） → 月1回
├── RDSメンテナンス → 自動（マイナーバージョン）
├── SSL証明書更新 → 年1回（ACMなら自動）
├── 監視・アラート確認 → 日常的
└── コスト最適化 → 四半期
```

**年間保守工数の見積**: 約100-200時間/年（月8-16時間程度）

## 4. 保守最小化のための設計原則

| # | 原則 | 具体策 |
|---|------|--------|
| 1 | **マネージドサービスを使う** | RDS > 自前PostgreSQL、Lambda > EC2 |
| 2 | **ノーコード部分を最大化** | 記録・閲覧はkintone、コード部分は分析のみ |
| 3 | **自動化する** | バックアップ自動化、CI/CD、監視アラート |
| 4 | **ドキュメントを残す** | 運用手順書、トラブルシューティングガイド |
| 5 | **属人性を排除する** | 設定をコード化(IaC)、手順を明文化 |

## 5. 外注する場合のスコープ

### 外注に適した範囲

| 範囲 | 理由 | 概算費用 |
|------|------|---------|
| **本番インフラ構築** | IaCで構築、引き渡し | 50-100万円 |
| **セキュリティ設計** | 専門知識が必要 | 設計費含む |
| **フロントエンド開発** | UI/UXの品質担保 | 100-200万円 |
| **年間保守契約** | 障害対応、セキュリティパッチ | 30-60万円/年 |

### 自分でやるべき範囲

| 範囲 | 理由 |
|------|------|
| **要求確認・要件定義** | 業務を理解している自分がやるべき |
| **ドメインモデリング** | 現場の言葉を知っている |
| **プロトタイプ開発** | 要件の検証が目的 |
| **分析ロジック** | 品質管理の知識が必要 |

設計レビュー時に以下を明確にしておく:

| # | 合意項目 | 選択肢 |
|---|---------|--------|

| 5 | **データのバックアップ** | 頻度、保持期間、リストア手順 |
| 6 | **技術選定の承認** | kintone / AWS / ハイブリッド |
| 7 | **外注の承認プロセス** | いくらまで裁量で使えるか |

## 7. 推奨: 段階的な保守体制

```
Phase 1（今〜3ヶ月）: 最小保守
├── kintoneでアプリ構築（保守ほぼゼロ）
├── 藤田が一人で運用
└── 保守工数: 月2-3時間

Phase 2（3〜6ヶ月）: 分析機能追加
├── Python分析スクリプトを追加
├── 運用手順書を作成
├── 小板橋さんにkintone操作を引き継ぎ
└── 保守工数: 月5-8時間

Phase 3（6ヶ月〜）: 安定運用
├── 保守手順が確立
├── 必要に応じて外注（年間保守契約）
├── 藤田はM1/M2に注力
└── 保守工数: 月2-3時間（安定化後）
```

**ポイント**: Phase 1で「動くもの」を出して信頼を得る → Phase 2以降の予算・体制の承認を得やすくなる
