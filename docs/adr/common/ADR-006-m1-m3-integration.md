# ADR-006: M1（GNSS評価）とM3（受入検査DB）の統合方針

**ステータス**: 承認済み
**日付**: 2026-03-11
**決定者**: 藤田
**影響範囲**: M1-B GNSS評価、M3 受入検査DB

---

## コンテキスト

M1のGNSS受入検査ツールとM3の受入検査DBは、以下の関係にある：

```
M3: 受入検査DB（全部品）
  │
  ├── バッテリー検査
  ├── プロポ検査
  ├── モーター検査
  │
  └── **GNSS検査** ← M1で今作っている
        │
        ├── 屋内検査（通信、FW、設定）
        └── 屋外検査（RTK FIX、感度）← 将来
```

**つまり**: M1のGNSS受入検査ツールは、M3の「GNSS部品」検査の自動化部分。

---

## 共通化すべき概念

M1で作る際に、M3と互換性を持たせる：

```
共通概念（M3と同じ）:
├── ロット（lot_id, lot_number, 期待値）
├── 検査記録（inspection_id, 検査日時, 操作者）
├── 検査結果（項目, 判定, 実測値, 期待値）
│
GNSS固有:
├── 装置シリアル番号（UBX-SEC-UNIQID）
├── FW比較機能（多数派・はずれ値）
└── UBXメッセージ（CFG-*, MON-VER等）
```

| 観点 | M3（全部品） | M1 GNSS | 整合性 |
|------|------------|---------|--------|
| ロット | 全部品にロット概念 | GNSSにもロット | 同じ概念 |
| 検査項目 | 部品ごとに異なる | GNSS固有項目 | 共通構造 |
| 判定 | Pass/Fail/特採 | Pass/Fail/Recorded | 拡張可能 |
| 装置シリアル | 一部の部品のみ | GNSSは必須 | オプショナル |
| FW比較 | 一部の部品のみ | GNSS固有 | 部品固有機能 |

---

## 検討した選択肢

### 選択肢A: 段階的統合（採用）

```
Phase 1: 屋内検査（今やってる）
  → 実機テスト → フィードバック
Phase 2: 屋外検査を追加
  → 実機テスト → フィードバック
Phase 3: M3と統合
```

**メリット**: 小さく検証できる、手戻りが小さい
**デメリット**: 時間がかかる

### 選択肢B: 一気に作る

```
Phase 1: 屋内+屋外を一気に作る
  → 実機テスト（屋内+屋外）
Phase 2: M3と統合
```

**メリット**: 早く完成形が見える
**デメリット**: 手戻りが大きくなる可能性

### 選択肢C: M3共通部分を先に設計

```
Phase 0: M3共通のドメインモデル設計
Phase 1: GNSS（屋内+屋外）を実装
Phase 2: 他部品（バッテリー等）を追加
```

**メリット**: M3との整合性が最初から取れる
**デメリット**: 設計に時間がかかる

---

## 決定

**選択肢A（段階的統合）を採用する**

### 理由

1. **両方プロトタイプ段階**: M1もM3もプロトタイプなので、今から統合設計に時間をかけるより、動くものを作ってフィードバックを得る方が有効。

2. **共通概念は設計済み**: ロット、検査記録、検査結果の構造はM1のDB設計ですでにM3互換を意識している。

3. **統合コストの見積もり**: M1完成後の統合は、DB移行とAPI統合程度。両方がSQLite/Rustなので技術的障壁は低い。

---

## 影響

- M1のDB設計はM3との将来統合を意識した構造にする
- ロット、検査記録、判定の概念はM3と互換性を持たせる
- 統合作業はM1屋内検査→屋外検査の後に実施
