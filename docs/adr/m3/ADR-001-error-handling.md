# ADR-001: エラーハンドリング方針

**日付**: 2026-03-09
**状態**: 承認済み
**決定者**: 藤田

## 背景

プロトタイプ段階でのエラーハンドリング方針を決める。
将来的にはDDD/CA的なエラー設計が必要だが、今は最小限で進める。

## 検討案

### 案A: 最小限（プロトタイプ用）— 採用

- 日本語メッセージを直接返却
- スタックトレースなし
- カスタムエラー型なし

**メリット**:
- 実装が速い
- 依存関係が増えない

**デメリット**:
- デバッグが困難（本番では問題）
- エラーの種類が判別しにくい

### 案B: cockroachdb/errors + カスタムエラー

- `cockroachdb/errors` でスタックトレース付与
- ドメインエラー型を定義
- コンテキスト経由でトレースID伝播

**メリット**:
- デバッグしやすい
- エラーの種類を型で判別

**デメリット**:
- 実装コストが高い
- プロトタイプには過剰

## 決定

**選択**: 案A（最小限）

**理由**: プロトタイプの目的は「動くものを見せてフィードバックを得る」こと。エラー設計は本番化時に対応する。

## 今後の計画

本番化フェーズで案Bに移行する際のタスク:

1. `cockroachdb/errors` を導入
2. ドメインエラー型を定義（`domain/errors.go`）
3. エラーコード体系を設計（`LOT_NOT_FOUND`, `INVALID_QUANTITY` 等）
4. ミドルウェアでエラーログ出力（スタックトレース付き）
5. コンテキストにトレースID付与
