# ADR-003: ロット一覧画面の設計判断

## ステータス

承認済み（2026-03-09）

## コンテキスト

Session 48の計画で「部品ごとのロット一覧」の必要性が検討された。

**背景**:
- 既存画面: ロット登録（/）、検査入力、検査記録一覧、ダッシュボード
- 不足: 登録済みロットの一覧表示、部品ごとの絞り込み

**Session 48 計画より**:
> - ロット管理画面の追加検討: 部品ごとのロット一覧があったほうが良いのでは？
> - 追加案: 部品選択 → その部品のロット一覧 → ロット詳細/検査履歴

## 決定事項

### 1. 表示形式: フラットテーブル + フィルター方式

**採用**: 全ロットをテーブル表示し、部品ドロップダウンでフィルター

**不採用**: 部品ごとのグループ表示（アコーディオンUI等）

**理由**:
- プロトタイプ段階では、既存の検査記録一覧（`/records`）と同じパターンで実装コスト最小
- 部品でフィルターすれば同等の効果が得られる
- 本番化時にグループ表示が必要なら、ヒアリング結果を踏まえて判断

### 2. サプライヤー情報: 非表示

**採用**: ロット一覧にサプライヤー列を含めない

**理由**:
- 現在のPart APIは`supplier_id`のみ返し、`supplier_name`を返さない
- サプライヤー名を表示するには追加API呼び出しが必要
- プロトタイプ段階では過剰な拡張を避ける

**本番化時の対応案**:
- Part APIを拡張して`supplier_name`をJOINで返す
- または、Supplier APIを追加して別途取得

### 3. ナビゲーション構成

**変更前**:
- ロット（登録画面） → 検査入力 → 一覧 → ダッシュボード

**変更後**:
- ロット登録 → ロット一覧 → 検査入力 → 検査記録 → ダッシュボード

**理由**:
- 「ロット」というラベルだけでは登録か一覧か不明
- 「一覧」だけでは何の一覧か不明（検査記録）
- 明確なラベル付けでユーザビリティ向上

## 影響範囲

- `/lots` ページ新規作成
- `Navigation.tsx` の navItems 更新
- 既存APIの変更なし
