**ステータス**: 暫定（検証待ち）
**日付**: 2026-03-11
**関連セッション**: Session 93

---

## コンテキスト

Session 93でActix-web APIエンドポイント（装置管理API）を実装した。
このとき、TDDスキルを使用せずに実装を先行させた。

### 実装したAPI

- `GET /api/devices` - 装置一覧取得
- `POST /api/devices/{path}/connect` - 接続（自動検出）
- `POST /api/devices/{path}/disconnect` - 切断

---

## 決定

### 1. APIハンドラーのテストは統合テストとして実装する

**理由**:
- APIハンドラーはDeviceManagerの薄いラッパー
- DeviceManager自体は既に138テストでカバー済み
- HTTPリクエスト/レスポンスの検証は統合テストが適切

### 2. ユニットテストの範囲

| コンポーネント | テスト種別 | 理由 |
|--------------|----------|------|
| DeviceManager | ユニットテスト | 複雑なロジック、モック可能 |
| API ハンドラー | 統合テスト | 薄いラッパー、HTTP層のテストが必要 |
| 型変換（DeviceResponse等） | ユニットテスト | 純粋な変換ロジック |

### 3. 統合テストで検証すべき項目

- [ ] 正常系: 装置一覧が取得できる
- [ ] 正常系: 接続成功時のレスポンス
- [ ] 異常系: 存在しないポートへの接続→404
- [ ] 異常系: 二重接続→409 Conflict
- [ ] 異常系: タイムアウト→504 Gateway Timeout

---

## 代替案

### 案B: TDD厳密適用

**手順**:
1. APIレスポンス型を先に定義
2. 各エンドポイントのテストを先に書く
3. テストを通すように実装

**却下理由**:
- プロトタイプ段階で優先度は低い
- DeviceManagerのロジックは既にテスト済み
