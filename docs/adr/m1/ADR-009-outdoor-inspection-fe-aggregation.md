# ADR-009: 屋外検査の集計処理をフロントエンドで実行

**ステータス**: 承認済み
**作成日**: 2026-03-12
**決定者**: 藤田
**関連セッション**: Session 123, 124

---

## コンテキスト

屋外検査機能では、検査中にサンプルを蓄積し、検査終了時に集計・判定を行う。
この集計処理をフロントエンド（FE）とバックエンド（BE）のどちらで行うかを決定する必要がある。

### 検査フロー

```
検査開始 → サンプル蓄積（30〜300回） → 検査終了 → 集計 → 判定 → 結果表示 → DB保存
```

---

## 決定

### 集計処理はフロントエンドで実行する

| 処理 | 実行場所 | 理由 |
|------|----------|------|
| サンプル蓄積 | **FE（React状態）** | リアルタイム表示と同時に蓄積 |
| 集計処理 | **FE** | 蓄積したデータをその場で処理 |
| 判定処理 | **FE** | 集計結果から即座に判定 |
| 結果表示 | **FE** | 判定後すぐに表示 |
| DB保存 | **BE** | 集計結果のみを送信 |

### 責務分担

| 層 | 責務 |
|----|------|
| FE | サンプル蓄積、集計、判定、表示 |
| BE | DB保存、履歴取得、検査結果一覧API |

---

## 根拠

### 1. リアルタイム蓄積との一貫性

検査中のサンプルはFE（React状態）で保持している。
BEに毎秒送信する方式は以下のデメリットがある:

- ネットワーク遅延の影響
- HTTPリクエストのオーバーヘッド
- サーバー側の状態管理が複雑化

### 2. 集計処理の軽量さ

30〜300サンプルの配列操作は、ブラウザで十分高速（<1ms）。
サーバーに送る必要がない。

### 3. DB保存は結果のみ

生サンプル（1秒ごとのNAV-STATUS/NAV-SIG）はDB保存対象外。
集計結果（FIX率、L2受信率、判定結果等）のみを保存する設計。

```
FE → BE: 集計結果（数十バイト）
FE → BE: 生サンプル（数KB）← 不要
```

### 4. オフライン動作の可能性

将来的にオフライン検査（ネットワーク未接続）をサポートする場合、
FEで完結する設計の方が拡張しやすい。

---

## 検討した代替案

### 案A: BEで集計（不採用）

```
FE → サンプル蓄積 → BE送信 → BE集計 → BE判定 → FE表示
```

**メリット**:
- 集計ロジックがBE一元化

**デメリット**:
- 検査終了時に大量データ送信
- ネットワーク遅延で結果表示が遅れる
- オフライン動作不可

### 案B: リアルタイムBE送信（不採用）

```
FE → 毎秒BE送信 → BE蓄積 → BE集計 → FE表示
```

**メリット**:
- サーバー側でサンプル管理
- 複数端末からの同時検査に対応しやすい

**デメリット**:
- 毎秒HTTPリクエスト → オーバーヘッド
- WebSocket導入 → 複雑化
- FEとBEで状態同期が必要

### 案C: FEで集計（採用）

**メリット**:
- シンプル（FEで完結）
- 高速（ネットワーク不要）
- オフライン拡張容易

**デメリット**:
- 集計ロジックがFE/BEで分散（現時点ではBEに集計ロジックなし）
