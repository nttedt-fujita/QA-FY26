# 品質管理設計レビュースキル

## 概要

M3（受入検査DB）/ M4（工程不良DB）関連の設計・図作成・レビュー時に、品質管理の視点を自動チェックするスキル。

**背景**: Session 25で品質管理フレームワーク（IQC/PQC/OQC、ロット、AQL、8D等）を調査したが、Session 27のAs-Is図作成時にその視点が反映されなかった。調査結果を設計作業に確実に反映するために作成。

---

## このスキルを使うタイミング

**使用すべき場面**:
- M3/M4関連のドメインモデリング
- As-Is / To-Be 図の作成・レビュー
- データ構造・DB設計
- ワークフロー・業務フロー図の作成
- プロトタイプの設計

**使用すべきでない場面**:
- 単純なコード実装（設計済みの場合）
- 分析スクリプトの作成
- データの集計・可視化のみ

---

## 必須チェックリスト

**IMPORTANT**: M3/M4関連の設計作業時、以下を必ず確認すること。

### 1. 品質管理プロセスの位置づけ

| チェック項目 | 確認内容 |
|-------------|---------|
| **IQC/PQC/OQCのどこか？** | 設計対象がIQC（受入）、PQC（工程内）、OQC（出荷）のどの位置か明確か |
| **M3とM4の関係** | M3=IQC、M4=IPQC/PQC。両者の紐づき（共通の「部品」で連携）を考慮したか |
| **工程全体の中での位置** | サプライヤ→IQC→製造→PQC→OQC→顧客 の流れの中で位置を図示したか |

```
サプライヤ → [IQC] → 製造工程 → [IPQC/PQC] → [FQC] → [OQC] → 顧客
              ↑          ↑              ↑          ↑
           受入検査    工程内検査      最終検査    出荷検査
             M3          M4
```

### 2. トレーサビリティの考慮

| チェック項目 | 確認内容 |
|-------------|---------|
| **ロット概念** | ロット番号の有無を確認したか。現行=なし、To-Be=導入必要 |
| **トレースバック** | 製品→原料の追跡（不良発生時の原因追跡）を考慮したか |
| **トレースフォワード** | 原料→製品の追跡（リコール時の影響範囲特定）を考慮したか |
| **共通IDの設計** | M3/M4を紐づける共通ID（部品マスタ、ロットID）を検討したか |

### 3. 検査基準の明確化

| チェック項目 | 確認内容 |
|-------------|---------|
| **全数 vs 抜取** | 検査が全数か抜取かを確認したか |
| **AQL基準** | 抜取検査の場合、AQL（合格品質限界）の設定を考慮したか |
| **合格/不合格基準** | 具体的な合否判定基準を設計に含めたか |

### 4. 問題解決フレームワーク（8D）

| チェック項目 | 確認内容 |
|-------------|---------|
| **問題記録** | 不良発生時の問題記録（何が、いつ、どこで）を設計に含めたか |
| **原因分析** | 原因究明のためのデータ項目（4M: 人・機械・方法・材料）を考慮したか |
| **対策記録** | 暫定対策・恒久対策の記録を設計に含めたか |
| **効果確認** | 対策後の効果確認（不良率推移等）を設計に含めたか |

### 5. 外部連携の制約

| チェック項目 | 確認内容 |
|-------------|---------|
| **品質協定書** | 委託先との品質協定書の有無を確認したか。現状=未締結 |
| **データ収集の実現性** | 協定書なしで委託先からデータ収集できるか検討したか |
| **段階的導入** | 協定書締結前/後で実現可能な範囲を分けて設計したか |

## As-Is / To-Be 図作成時の追加チェック

### As-Is図の場合

| チェック項目 | 確認内容 |
|-------------|---------|
| **現状の限界を明示** | ロット概念がない、トレーサビリティがない等の限界を図に含めたか |
| **品質管理観点のギャップ** | 理想的なIQC/PQCと比較した「できていないこと」を明示したか |

### To-Be図の場合

| チェック項目 | 確認内容 |
|-------------|---------|
| **導入すべき概念** | ロット、サプライヤID、AQL基準など、追加すべき概念を含めたか |
| **M3/M4の紐づき** | 共通の部品マスタ・ロットIDによる連携を設計に含めたか |
| **8Dフローの組み込み** | 不良→原因→対策→効果確認の流れを設計に含めたか |

## 現行Excelとのギャップ（常に念頭に置く）

| 概念 | 一般的な品質管理 | 現行Excel | To-Beで必要 |
|------|----------------|-----------|------------|
| ロット番号 | 必須 | **存在しない** | 導入必要 |
| サプライヤID | 追跡に必須 | **存在しない** | 導入必要 |
| 発注番号 | トレーサビリティ | **存在しない** | 導入必要 |
| AQL基準 | 抜取検査の基礎 | **不明** | 確認必要 |
| 8Dフロー | 問題解決の標準 | **存在しない** | 導入必要 |

---

## 禁止事項

### やってはいけないこと

- **Excelの構造をそのまま図解するだけ**（品質管理観点を必ず追加）
- **ロット/トレーサビリティを無視した設計**
- **M3とM4を独立したシステムとして設計**（紐づきを考慮）
- **品質協定書の制約を無視したM4設計**

## 使用例

### 設計作業開始時

```
M3/M4関連の設計作業を開始します。
qa-design-reviewスキルに従い、以下を確認します:
- [ ] IQC/PQC/OQCの位置づけ
- [ ] トレーサビリティの考慮
- [ ] 検査基準の明確化
- [ ] 8Dフレームワークの反映
- [ ] 外部連携の制約
```

### 図作成完了時

```
図を作成しました。qa-design-reviewスキルに従いレビューします:
- [x] 工程全体の中での位置を図示した
- [x] 現状の限界（ロットなし等）を明示した
- [ ] M3/M4の紐づきを図示した ← 追加が必要
```
