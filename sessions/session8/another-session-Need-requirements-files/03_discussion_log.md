# 品質保証DB構築 検討メモ（議論ログ）

**日付:** 2026年3月5日  
**最終更新:** データ膨張予測・アーキテクチャ議論・言語選定の考え方を追記  
**背景:** 品質保証部門への異動に伴い、受け入れ検査・工程不良のDB化ミッションを検討

---

## 要件の整理

### ビジネス要件

- 外注先でドローン/コントローラーを組み立てる工程の品質管理
- 作業者がタブレットで検査結果を入力
- クラウド経由で履歴確認・分析ができる仕組み

### DB化の対象（2領域）

- **受け入れ検査** — 組み立て用パーツ・保守用パーツの入荷時品質確認
- **工程不具合** — 組み立て工程で発生した不具合の記録・管理

### システムに求められること

1. **データ入力（外注先・現場）** — タブレットで作業者が簡単に入力できるUI
2. **データ蓄積（クラウド）** — 検査履歴・不良情報の一元管理
3. **閲覧・分析（社内）** — 履歴確認、傾向分析、レポート出力
4. **紐付け** — 受け入れ検査結果と工程不具合をDB上で関連付ける

---

## 受け入れ検査とロット管理の課題

### 抜取検査と紐付けの問題

- 受け入れ検査は抜取検査で実施 → 検査サンプルと工程使用個体が不一致
- 個体単位の紐付けは不可能 → **ロット単位での紐付けが現実的**
- ロット単位で「特定ロットから不具合が集中しているか」の分析が可能

### ロット定義の問題（未確定・要決定）

3つの選択肢を検討。

**① サプライヤーロット** — 購入先の製造単位番号。品質均一性の観点で最も合理的。サプライヤーの対応に依存。

**② 受入ロット（自社定義）** — 入荷タイミングで自社採番。運用は最もシンプル。1回の入荷にサプライヤー側複数ロットが混在する可能性。

**③ 混合方式** — 取れる場合はサプライヤーロット、取れない場合は受入ロット。データ粒度が混在。

**暫定方針:** 受入ロットを自社で必ず採番 + サプライヤーロット番号は任意項目として記録。

### 組み立て用 vs 保守用パーツ

- 用途によって検査基準や管理粒度が異なる可能性
- テーブル分離 or 用途区分フラグの判断はヒアリング後

### 工程不具合との紐付けの運用課題

- 不具合発生時に「この部品がどのロットか」を特定できる現場運用が前提
- ロット追跡方法（ラベル管理、入荷順推定など）の実態確認が必要

## アーキテクチャ上の議論

### ネイティブAndroidアプリ vs PWA

**結論: PWAファーストで検証。** 一人開発で2コードベース維持は非現実的。PWAならブラウザさえあれば端末を選ばず、ストア審査も不要。オフライン要件が重い場合のみネイティブを再検討。

### マイクロサービス vs モジュラーモノリス

**結論: モジュラーモノリス。** 一人開発でサービス間通信・個別デプロイ・分散トレーシングはオーバーエンジニアリング。コードレベルでモジュール境界を明確にし（受入検査/工程不具合/ロット管理/ユーザー管理）、デプロイは1つ。将来の分離が必要になったときに切り出せる準備だけしておく。

### メンテナンスの現実

どの構成を選んでもアプリの保守は将来誰かがやることになる。できるのは保守負荷の最小化のみ。IaC、ドキュメント、標準技術の選定で「引き継ぎ可能性」を担保する。

## 言語・技術選定の考え方

### 「どの言語を経験したか」より「なぜ選んだか」

AIがコードを書く時代において、特定言語の実装経験の価値は相対的に低下している。差がつくのは技術選定の意思決定プロセスを実体験として語れること。

- DynamoDBではなくRDBを選ぶ理由をリレーション特性から説明できる
- マイクロサービスではなくモジュラーモノリスを規模と体制から導ける
- PWAを選びネイティブを退ける根拠を運用コストと配布性から示せる

このプロジェクトの価値は「業務システムの技術選定・アーキテクチャ設計を一人で完遂した経験」として捉えるべき。

### TypeScript統一の判断根拠

「Goの実務経験を積むため」ではなく、一人開発・他ミッション並行・AI支援活用・保守引き継ぎ前提という**制約条件から導かれた合理的選択**として位置づける。フロント・バックエンド・IaCの言語統一による認知負荷低減が、この制約下では最も重要。

## データ膨張予測と将来拡張

※ 詳細なシナリオ別試算は `qa_db_data_growth_analysis_20260305.md` を参照

### 初回試算の反省

当初は「月50入荷 × 10項目 → 5年で3万件」という単一シナリオのベストケースのみ。以下の問題点を指摘し、シナリオ別の深掘り分析を実施。

- 前提値が全てヒアリング未実施の仮置き
- 事業成長率の影響（複利で5年後に大きく差が開く）を未考慮
- データ保持期間の法的要件（PL法で10年の可能性）を未考慮
- 画像撮影方針（不良のみ vs 全件）で桁が変わることを未検討

### シナリオ別試算の結果

3シナリオ（保守的/成長/急成長）で10年累計を試算。全シナリオでRDS PostgreSQLで対応可能。Aurora Serverless v2やDynamoDBが必要になるシナリオは存在しない。前提値はヒアリングで実数に置き換える必要あり。

### 将来の拡張シナリオ（※ヒアリングで確認すべき）

- 出荷検査データの追加 → 現在のRDBにテーブル追加で対応可能
- フィールド故障データの紐付け → 同上
- フライトログ・センサーデータ → RDBとは完全にレイヤー分離（S3+Athena）

### 設計への織り込み

- 現在のRDBはトランザクションデータ（検査・不具合・ロット）に専念
- 大量データ（センサー等）は別レイヤーで対応する前提
- RDSインスタンスのスケールアップ（micro→small→medium）で段階対応
- 現時点でセンサーデータ対応は実装しない（Phase分離）

## 設計上の前提・懸念事項

### 「自分がいなくても回る仕組み」の必要性

- 将来的な退職を見据え、属人化を避ける設計が必要
- 社内の人材不足は慢性的な課題
- ただし完全にメンテナンスフリーなシステムは存在しない

### ドメインモデル設計の重要性

- ツール選定の前にデータ構造の設計が最優先
- ロット定義を含む業務ルールの明確化がDB設計の土台
- これは品質保証部門としての業務でもある

## ツール選定の検討

### 案1: kintone

- **メリット:** ノーコード、公式サポート
- **デメリット:** ユーザー単位課金（約1,500円/月/人）、外注先分のコスト増、UIカスタマイズ限界

### 案2: Google Forms + Spreadsheet + Looker Studio

- **メリット:** ランニングコストほぼゼロ
- **デメリット:** 入力UIカスタマイズ性が低い

### 案3: AppSheet（Google）

- **メリット:** ノーコード、タブレットUI対応
- **コスト:** Core版 $5.83/月/ユーザー

### 案4: Power Apps + Power BI（Microsoft系）

- **メリット:** Microsoft 365環境との親和性

### 案5: AWS構成（一次選定）

- **構成:** Amplify + API Gateway + Lambda + RDS PostgreSQL + QuickSight
- **メリット:** サーバーレスAPI、従量課金、AWS実務経験、マネージドサービスで属人性排除
- **デメリット:** ドキュメント・IaC整備の初期工数

---

## 結論・方針（暫定）

- **ドメインモデル設計を最優先**で着手
- AWS構成を一次選定とし、ヒアリング結果で最終判断
- DB選定はデータ膨張予測を踏まえRDS PostgreSQL（最小インスタンス）を推奨
- 将来拡張パス（センサーデータ等）は設計段階で想定するが、現時点では実装しない
