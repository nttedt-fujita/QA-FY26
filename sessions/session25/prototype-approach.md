## 2. M3とM4の関係

### 品質管理の流れ

```
[サプライヤ] → [IQC: 受入検査] → [製造工程] → [IPQC: 工程不良]
                    ↓                              ↓
               M3: 不良発見              M4: 不良発見
                    ↓                              ↓
               「この部品に問題」 ←────────→ 「この部品起因？」
```

### 紐づきのポイント

| 紐づきの軸 | M3での役割 | M4での役割 |
|-----------|-----------|-----------|
| **部品（品名）** | 検査対象 | 不良発生箇所 or 原因部品 |
| **ロット番号** | 入荷ロット | 使用部品のロット（トレース用） |
| **時系列** | 入荷日・検査日 | 不良発生日 |

### 紐づけで可能になること

1. **M3で見逃した不良の検出**: M4で発見された不良の原因がM3（入荷部品）にあるか追跡
2. **サプライヤへのフィードバック**: M4の不良データをもとに、サプライヤ品質の評価
3. **検査基準の見直し**: M4で頻発する不良 → M3の検査項目追加を検討

## 3. プロトタイプの範囲

### 作れるもの / 作れないもの

| 機能 | 作れる？ | 根拠 |
|------|---------|------|
| **分析ダッシュボード** | ✅ | CSVデータ構造確定、分析ロジック完成（Session 19-23） |
| **CSVインポート** | ✅ | 変換・名寄せ・クレンジング済み |
| **月別/品名別集計** | ✅ | パレート分析相当、既存ツールで実現済み |
| データ入力フォーム | ❌ | As-Isモデル未確定、紙→デジタル移行の要件不明 |
| マスタ管理 | ❌ | To-Beモデルの検討が必要 |
| M4（工程不良）本格導入 | ❌ | **品質協定書未締結** |

### プロトタイプのスコープ

```
Phase 1: 分析・可視化（今できる）
├─ 現行Excelデータのインポート
├─ 月別サマリー表示
├─ 品名別パレート分析
├─ カテゴリ別（メカ/エレキ/Api）層別分析
└─ 「今どうなっているか」の把握

Phase 2: 入力のデジタル化（ヒアリング後）
├─ As-Isモデルに基づく入力フォーム
├─ 紙記録のデジタル化
└─ 工数削減の実現

Phase 3: M3/M4統合 + トレーサビリティ（協定書締結後）
├─ ロット概念の導入
├─ M3/M4の紐づけ
├─ 原因分析（8D相当）
└─ 効果確認の可視化
```

## 4. 共通データモデル（M3/M4紐づけ用）

### 共通マスタ（To-Beで必要）

| マスタ | M3での使用 | M4での使用 |
|--------|-----------|-----------|
| **部品マスタ** | 検査対象の特定 | 不良部品の特定 |
| **サプライヤマスタ** | 入荷元 | （間接的に参照） |
| **委託先マスタ** | - | 不良発生場所 |
| **作業者マスタ** | 検査担当者 | 発見者/作業者 |

### 紐づけのキー

```
[入荷ロット] ←── lot_id ──→ [製造記録]
      ↓                           ↓
[M3: 受入検査記録]         [M4: 工程不良記録]
      ↓                           ↓
   品名 ←───── 部品マスタ ─────→ 品名
```

### 現行Excelとのギャップ

| 概念 | 現行Excel | To-Be |
|------|----------|-------|
| lot_id | ❌ なし | ○ 必須 |
| supplier_id | ❌ なし | ○ 必須 |
| 委託先 | ❌ なし | ○ M4で必須 |
| 部品マスタ | ❌ 自由記述（品名） | ○ マスタ化 |

## 5. 技術的な方針

### Phase 1（分析・可視化）の技術選択

| 項目 | 方針 | 根拠 |
|------|------|------|
| データ形式 | CSV（既存ツールの出力） | Session 19-23で整備済み |
| 分析 | Python（既存スクリプト拡張） | tools/incoming_inspection/ |
| 可視化 | Streamlit or Jupyter Notebook | 手軽にダッシュボード作成可能 |

### Phase 2以降の技術選択（要検討）

| 項目 | 選択肢 | 判断材料 |
|------|--------|----------|
| プラットフォーム | kintone vs 自前開発 | Session 5で比較済み、P0-2でヒアリング |
| データベース | SQLite（ローカル）/ PostgreSQL（AWS） | データ量による |
| 入力UI | タブレット対応Webアプリ | 現場運用の要件次第 |

## 7. 判断のまとめ

| 判断 | 根拠 |
|------|------|
| **Phase 1（分析）から始める** | データ構造確定済み、ツール完成済み |
| **入力フォームは後回し** | As-Isモデル未確定、ヒアリングが必要 |
| **M4は協定書締結を待つ** | 協定書なしでは委託先にツール使用を強制できない |
| **M3/M4の紐づけはTo-Beで設計** | 現行Excelにはロット概念がない |
