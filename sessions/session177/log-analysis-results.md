## 傾向分析

### 統計

- **検査数**: 17件（計画では10件と思っていたが、実際は17件）
- **POST後BE処理回数**: 4〜12回（平均8.2回）
- **ずれ時間（検査7除く）**: 8秒〜214秒
- **典型的なずれ**: 60〜80秒
- **ロック待機警告数**: 100回

### 傾向

1. **ほぼ全ての検査でPOST後60-80秒BEが処理継続**
2. **検査17は8秒と短い** → 最後なので溜まっていたリクエストが少なかった
3. **検査1-2は特に長い（147秒、214秒）** → 初回は競合が激しかった

## 重要な発見

### 1. FE終了後もBEが平均70秒処理を続ける

- **16/17件**でPOST後にBEが数十秒処理を継続
- 唯一の例外: 検査17（最後の検査、8秒で終了）

### 2. Mutex競合が主因

ログに100件の警告:
```
⚠️ ロック待機が長い！(XXXX ms) - NTRIP転送と競合の可能性
```

- 最大ロック待機時間: **52999ms（約53秒）**
- gnss_state_api と ntrip_api が同じMutexを取り合っている

### 3. ロック待機時間の累積パターン

POST後、ロック待機時間が累積的に増加:
```
検査1終了後:  4850ms → 9851ms → 19850ms → 29995ms
検査2終了後: 13448ms → 32879ms → 40792ms
```

→ **キューに溜まったリクエストが順番に処理される**

### 4. 1回のAPI呼び出し所要時間

- **正常時**: 5-6秒（競合なし）
- **競合時**: 30-50秒（ロック待機が大部分）

## 問題の構造

```
FEポーリング（1秒間隔）
    ↓
API呼び出しキュー（非同期）
    ↓
Mutex取得待ち（直列化）
    ↓
NTRIP転送中はMutex保持（数秒〜数十秒）
    ↓
FEが30秒タイマーで終了しても、キューにはリクエストが残っている
    ↓
BEは残りのキューを処理し続ける
```

### 根本原因

1. **FEの1秒ポーリング vs BEの6秒処理**: 5回分のリクエストがキューに溜まる
2. **Mutex競合**: NTRIP転送中はgnss_state_apiがブロック
3. **FE終了時のキャンセルなし**: FEはPOSTして終了するが、pending中のAPIリクエストはキャンセルされない

---

## 根本原因

### 問題の数式化

```
FEポーリング間隔: 1秒
BE処理時間: 約6秒（競合なし）/ 最大50秒（競合あり）
検査時間: 30秒

30秒間のリクエスト数: 30回
30秒間でBEが処理できる数: 5回（競合なし）/ 1回未満（競合あり）
キューに溜まるリクエスト: 25〜29回
```

### 結論

**FEのポーリング間隔（1秒）とBEの処理能力（6秒/回）の不整合が根本原因**

Mutex競合は問題を悪化させているが、競合がなくても5倍のリクエストがキューに溜まる。

---

## 対策案

### 案A: FE側でポーリング間隔を調整（最優先）

**変更**: useOutdoorInspection.ts の useEffect 内のポーリング間隔を 1秒 → 6秒に変更

```typescript
// 現状
const POLLING_INTERVAL = 1000; // 1秒

// 変更後
const POLLING_INTERVAL = 6000; // 6秒
```

**メリット**:
- 1行の変更で済む
- キュー溜まりを根本解決
- BE負荷も軽減

**デメリット**:
- データ取得頻度が下がる（30秒で30回 → 5回）
- リアルタイム性が低下

### 案B: FE終了時にpendingリクエストをキャンセル

**変更**: AbortControllerを使用してFE終了時にキャンセル

**メリット**:
- 1秒ポーリングを維持できる
- 理論的に正しい解決

**デメリット**:
- FE変更が必要
- 案Aより実装が複雑

### 案C: BE側でMutex競合を解消（try_clone）

**変更**: Session 175で検討済み。serialport v4の`try_clone()`で読み書き分離

**メリット**:
- Mutex競合を根本解消
- 処理時間が6秒→数秒に短縮の可能性

**デメリット**:
- BE変更が大きい
- 案Aで十分な場合は過剰

### 案D: タイムアウト短縮

**変更**: gnss_state_api.rsのタイムアウトを2秒→500msに短縮

**メリット**:
- BE側の小さな変更

**デメリット**:
- データ欠損が増える
- 根本解決ではない

## 推奨

**段階的アプローチ（KISS原則）**:

1. **即座**: 案A（ポーリング間隔を6秒に）
   - 1行の変更でキュー溜まりを解消
   - 実装: 5分以内

2. **評価**: 案Aの効果を確認
   - 30秒検査でBE終了が10秒以内になればOK
   - スナップショット数が5回で十分か確認

3. **必要なら**: 案B or 案C
   - リアルタイム性が必要な場合のみ検討
