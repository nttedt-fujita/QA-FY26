# ドメインモデル統合検討

**作成日**: 2026-03-11 Session 86

---

## 1. 現状: 2つの別々のドメインモデル

### 屋外計測向け（Session 62）

```
デバイス（型番、S/N）
    │
    └── 計測セッション (1:N)
            ├── 評価条件（場所、環境、アンテナ）
            ├── 計測データ（時系列）
            │     ├── NAV-PVT（位置・精度）
            │     ├── NAV-STATUS（TTFF、FIX状態）
            │     ├── NAV-DOP（精度劣化）
            │     ├── NAV-SAT（衛星情報）
            │     ├── MON-SPAN（スペクトラム）
            │     └── MON-RF（ジャミング）
            └── 評価結果（判定、所見）
```

**目的**: RTK FIX時間、受信感度、精度を**継続的に記録**

### 屋内受入検査向け（Session 85/86）

```
ロット（ロット番号、期待値）
    │
    └── 検査 (1:N)
            ├── 装置シリアル
            ├── 検査日時
            └── 検査項目結果 (1:N)
                  ├── 通信疎通
                  ├── シリアル番号（SEC-UNIQID）
                  ├── FWバージョン（MON-VER）
                  ├── 出力レート（CFG-RATE）
                  └── ポート設定（CFG-PRT）
```

**目的**: 設定値を**一回確認**して合否判定

---

## 2. 違いの分析

| 観点 | 屋外計測 | 屋内受入検査 |
|------|---------|-------------|
| **時間軸** | 時系列データ（10Hz等で連続） | 1回のスナップショット |
| **判定** | 基準値との比較（範囲判定） | 期待値との一致/不一致 |
| **FW** | 計測しない | 多数派・はずれ値の比較 |
| **ロット** | 関係ない | 期待値の出所 |
| **データ量** | 大量（分〜時間の時系列） | 少量（5項目） |

---

## 3. 共通点

| 概念 | 屋外計測 | 屋内受入検査 | 共通化 |
|------|---------|-------------|--------|
| **デバイス/装置** | デバイス（型番、S/N） | 装置シリアル | ✅ 同じ |
| **セッション/検査** | 計測セッション | 検査 | ✅ 類似 |
| **UBXメッセージ** | NAV-*, MON-* | SEC-UNIQID, MON-VER, CFG-* | ✅ 同じライブラリ |

---

## 4. 統合案

### 案A: 2層構造（共通 + 専用）

```
M3: 受入検査DB（共通層）
├── 部品マスタ（バッテリー、GNSS、モーター...）
├── ロット（部品ごとの入荷単位）
└── 検査記録（部品ごとの検査結果）

    ↓ GNSSの場合のみ

M1 GNSS専用層:
├── 屋内検査（設定確認）
│     └── SEC-UNIQID, MON-VER, CFG-RATE, CFG-PRT
│
└── 屋外計測（性能評価）
      └── NAV-PVT, NAV-STATUS, NAV-DOP, NAV-SAT, MON-RF
```

**メリット**: M3と整合、部品ごとの拡張が容易
**デメリット**: 複雑になる可能性

### 案B: GNSSに特化（M3とは後で統合）

```
GNSS検査ツール（独立）
├── ロット（GNSS部品の入荷単位）
├── 装置（個別のGNSSモジュール）
│
├── 屋内検査
│     └── 設定確認（FW、Rate、Port）
│
└── 屋外計測
      └── 性能評価（RTK FIX、受信感度）
```

**メリット**: シンプル、早く完成
**デメリット**: M3との統合時に修正が必要

### 案C: 検査→計測の一貫フロー

```
装置
  │
  ├── 受入検査（屋内、1回のスナップショット）
  │     ├── FW確認
  │     ├── 設定確認
  │     └── 通信確認
  │
  └── 性能評価（屋外、時系列データ）← 受入検査に合格した装置のみ
        ├── RTK FIX時間
        ├── 受信感度
        └── 精度
```

**メリット**: 運用フローに沿っている
**デメリット**: 「受入検査」と「性能評価」の関係を明確にする必要
