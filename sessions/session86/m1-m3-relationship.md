# M1とM3の関係性整理

**作成日**: 2026-03-11 Session 86

---

## 1. 構造

```
M3: 受入検査DB（全部品）
  │
  ├── バッテリー検査
  ├── プロポ検査
  ├── モーター検査
  │
  └── **GNSS検査** ← M1で今作っている
        │
        ├── 屋内検査（通信、FW、設定）
        └── 屋外検査（RTK FIX、感度）← 将来
```

**つまり**: M1のGNSS受入検査ツールは、M3の「GNSS部品」検査の自動化部分。

---

## 2. M3との整合性を取るべき点

| 観点 | M3（全部品） | M1 GNSS | 整合必要？ |
|------|------------|---------|-----------|
| **ロット** | 全部品にロット概念 | GNSSにもロット | ✅ 同じ概念 |
| **検査項目** | 部品ごとに異なる | GNSS固有項目 | ✅ 共通構造 |
| **判定** | Pass/Fail/特採 | Pass/Fail/Recorded | ✅ 拡張可能に |
| **装置シリアル** | 一部の部品のみ | GNSSは必須 | ✅ オプショナルに |
| **FW比較** | 一部の部品のみ | GNSS固有 | 部品固有機能として設計 |

---

## 3. 共通化すべき概念

M1で作る際に、M3と互換性を持たせる:

```
共通概念（M3と同じ）:
├── ロット（lot_id, lot_number, 期待値）
├── 検査記録（inspection_id, 検査日時, 操作者）
├── 検査結果（項目, 判定, 実測値, 期待値）
│
GNSS固有:
├── 装置シリアル番号（UBX-SEC-UNIQID）
├── FW比較機能（多数派・はずれ値）
└── UBXメッセージ（CFG-*, MON-VER等）
```

---

## 4. 屋内/屋外の整理

### 屋内検査（今作っている）

| 項目 | UBXメッセージ | 判定方式 |
|------|-------------|----------|
| 通信疎通 | 応答確認 | Pass/Error |
| シリアル番号 | SEC-UNIQID | Pass/Error |
| FWバージョン | MON-VER | Recorded（後で比較） |
| 出力レート | CFG-RATE | Pass/Fail |
| ポート設定 | CFG-PRT | Pass/Fail |

### 屋外検査（将来）

| 項目 | 方法 | 判定方式 |
|------|------|----------|
| RTK FIX時間 | 測定 | 基準値と比較 |
| 受信感度 | 測定 | 基準値と比較 |
| 衛星捕捉数 | NAV-SAT | 基準値と比較 |

### 共通項目

| 項目 | 屋内 | 屋外 |
|------|------|------|
| FWバージョン | ✅ | ✅（同じ値） |
| シリアル番号 | ✅ | ✅（同じ値） |
| 通信疎通 | ✅ | ✅（再確認） |

## 5. 実装方針の選択肢

### A: 段階的（現在の計画）

```
Phase 1: 屋内検査（今やってる）
  → 実機テスト → フィードバック
Phase 2: 屋外検査を追加
  → 実機テスト → フィードバック
Phase 3: M3と統合
```

**メリット**: 小さく検証できる、手戻りが小さい
**デメリット**: 時間がかかる

### B: 一気に作る

```
Phase 1: 屋内+屋外を一気に作る
  → 実機テスト（屋内+屋外）
Phase 2: M3と統合
```

**メリット**: 早く完成形が見える
**デメリット**: 手戻りが大きくなる可能性

### C: M3共通部分を先に設計

```
Phase 0: M3共通のドメインモデル設計
Phase 1: GNSS（屋内+屋外）を実装
Phase 2: 他部品（バッテリー等）を追加
```

**メリット**: M3との整合性が最初から取れる
**デメリット**: 設計に時間がかかる
