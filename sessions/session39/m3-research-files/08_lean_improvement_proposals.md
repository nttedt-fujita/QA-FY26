# 受け入れ検査改善 — 少人数で最大効果を出すアプローチ案

**作成日**: 2026-03-07
**前提**: ベンチャー企業。人的リソースが限られており、大掛かりな仕組みづくりはできない。

---

## 共通の考え方

300h/月のうち「どこに工数が偏っているか」を把握し、最も効果が大きい 1〜2 箇所に集中して手を打つ。
全部品・全プロセスを一律に改善しようとしない。

---

## 提案 A：抜取検査計画の見直しだけで工数を減らす

### 概要

ツールも AI も作らない。検査のサンプル数と頻度を、データに基づいて適正化する。

### やること

1. 現場ヒアリングで「どの部品に何時間かかっているか」の大まかな内訳を把握（1〜2 日）
2. 工数上位の部品について、現在の抜取検査計画（AQL・サンプルサイズ）を確認
3. 過剰な検査をしている部品を特定し、JIS Z 9015 に基づいて適正化
4. 実績的に不良がほぼ出ていないサプライヤー/部品は「なみ検査→ゆるい検査」に切り替え

### 必要なリソース

- fujiojiさんの工数: 1〜2 週間（ヒアリング + 検査計画レビュー）
- ツール開発: なし
- コスト: ゼロ

### 期待効果

- サンプルサイズ適正化だけで 20〜30% の工数削減は現実的
- 300h → 210〜240h（60〜90h 削減 = 月 ¥21〜31 万相当）

### リスク

- 過去の不良実績データがないと、検査を軽減する根拠が弱い
- 「減らして大丈夫か」の判断に経験則頼みになる可能性

---

## 提案 B：Google スプレッドシート + Google フォームで記録基盤を最速構築

### 概要

専用ツールは作らない。Google フォーム（入力）+ スプレッドシート（DB 代わり）+ Google ドライブ（画像保存）で記録基盤を即日構築する。

### やること

1. Google フォームで検査記録の入力フォームを作成
   - 部品名（プルダウン）、サプライヤー（プルダウン）、ロットID、検査日
   - 判定（合格 / 不合格）、不良モード（プルダウン）、写真添付
2. 回答がスプレッドシートに自動蓄積
3. スプレッドシートのピボットテーブルやグラフで傾向を可視化
4. 1〜2 ヶ月データを貯めたらパレート分析

### 必要なリソース

- fujiojiさんの工数: 2〜3 日（フォーム設計 + 現場への導入）
- ツール開発: なし（Google Workspace で完結）
- コスト: Google Workspace の既存契約内ならゼロ

### 期待効果

- 記録の仕組みが即日稼働
- 1〜2 ヶ月後にはデータに基づく意思決定ができるようになる
- 将来の AI 連携に向けた学習データの蓄積が始まる

### リスク

- 検査員にフォーム入力の負荷がかかる（1 件 1〜2 分追加）
- 画像の整理・紐付けが手動になりがち
- スプレッドシートは件数が増えると重くなる（年間数千件程度なら問題なし）

---

## 提案 C：マルチモーダル LLM で「判定支援ツール」を最速で試す

### 概要

SageMaker 等の本格 ML パイプラインは組まず、Claude API や GPT-4o に検査画像を投げて判定補助させる。最もシンプルな AI 活用。

### やること

1. 提案 B の記録基盤を先に構築（前提）
2. 工数上位かつ目視判定が中心の部品を 1 種類選定
3. その部品の良品・不良品の写真を 20〜30 枚ずつ用意
4. Claude API に「この画像は良品か不良品か、根拠とともに判定せよ」とプロンプト
5. 判定精度を検査員の判定と比較評価
6. 実用に足るなら、検査フローに「AI 一次スクリーニング → 人間最終判定」を組み込む

### 必要なリソース

- fujiojiさんの工数: 3〜5 日（API 連携スクリプト + 評価）
- ツール開発: Python スクリプト数十行程度
- コスト: API 利用料 月 ¥5,000〜20,000

### 期待効果

- AI の実力と限界を最小コストで検証できる
- 「AI で検査できるのか」に対する社内への具体的な回答が出せる
- うまくいけば検査員の判定時間を 30〜50% 短縮

### リスク

- LLM の判定は専用モデルより精度が不安定
- 判定基準がプロンプト頼みで、検査規格としての正当性が弱い
- PoC としては有効だが、本番運用には追加の仕組みが必要

## 提案 D：提案 A + B の組み合わせ（推奨）

### 概要

即効性のある「検査計画見直し」と、中期的な「記録基盤構築」を並行で進める。
AI は記録データが貯まってから検討する。

### ロードマップ

```
Week 1-2:  現場ヒアリング（工数内訳 + 検査計画の確認）
    │
    ├─ 即効施策: 抜取検査計画の適正化（提案 A）
    │            → 工数 20〜30% 削減を狙う
    │
    ├─ 並行: Google フォーム + スプレッドシートで記録開始（提案 B）
    │
Week 3-4:  検査計画の変更を現場に展開
           記録運用の開始・定着支援
    │
Month 2-3: 記録データの蓄積・パレート分析
           → 次の改善ポイントを特定
    │
Month 3-4: 必要に応じて AI の PoC（提案 C）
           → データがあるので判断しやすい
```

### 必要なリソース

- fujiojiさんの工数: 最初の 1 ヶ月に集中的に 2〜3 週間
- ツール開発: なし
- コスト: ほぼゼロ

### 期待効果

- 1 ヶ月目で工数削減の成果が出る（抜取計画見直し）
- 2〜3 ヶ月目でデータに基づく改善サイクルが回り始める
- QA 組織としての最初の実績になる
