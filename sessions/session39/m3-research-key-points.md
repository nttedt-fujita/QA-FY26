## 全体サマリー

m3-research-filesは、DMAICアプローチで受け入れ検査プロセス改善を進めるための調査資料。
**核心メッセージ**: ツールやAIは手段であり目的ではない。問題の本質は「検査の有効性が定量化されていない」こと。

## ファイル別 重要ポイント

### 01_define_phase_report.md — Problem Statement

**最重要の一文**:
> 300hが「多すぎるのか適切なのか」を判断する根拠がない。検査の有効性（不良流出をどれだけ防いでいるか）が定量的に評価できない。

**見えていないこと（一次調査で明らかにすべき）**:
- 300hの内訳（どの作業に何時間か）
- 抜取検査計画の根拠（AQL、サンプルサイズの設定理由）
- 後工程での品質クレーム発生状況

**暫定結論**:
| 項目 | 正しい理解 |
|------|-----------|
| ツール制作 | ✗「作れば解決」→ ○「Measureのための記録基盤として必要」|
| AI活用 | 検査基準の明確化と記録の蓄積が先行すべき |
| サプライヤー | 「信用できない」を定量的に裏付けるデータがない |

### 02_investigation_checklist.md — 一次調査項目

**6カテゴリ・約30項目**:

| カテゴリ | 核心的な質問 |
|----------|-------------|
| A. 工数内訳 | 1ロットあたりの平均検査時間は？検査以外の付帯作業時間は？ |
| B. 検査プロセス | 抜取検査計画の根拠は？判定基準は明文化されているか？ |
| C. 記録の現状 | 何が記録されていて、何が記録されていないか？ |
| D. 品質実績 | 不合格率は？主な不良モードは？後工程でのクレームは？ |
| E. サプライヤー管理 | サプライヤーごとの品質実績の差は把握されているか？ |
| F. 既存の改善 | 過去に検査プロセス改善を試みたことはあるか？ |

**調査方法の推奨**:
- 現場観察 → VSM作成のためのプロセス把握
- ヒアリング → 工数内訳・ボトルネック・暗黙知の把握
- 書類確認 → 現行ルールと実態の乖離確認

### 03_framework_comparison.md — フレームワーク比較

**推奨適用順序**:

```
Phase 1: 現状把握（今すぐ着手可）
  - CoQ（品質コスト4分類）
  - VSM（プロセス可視化）
  - 特性要因図（4M分析）

Phase 2: 記録基盤の構築
  - Measureに必要なデータ項目の定義
  - 記録の仕組み（DB/ツール）の設計

Phase 3: データに基づく分析と最適化
  - パレート図（重点問題の特定）
  - FMEA（検査重点の再配分）
  - 抜取検査計画見直し（ISO 2859）

Phase 4: 検査プロセスの高度化
  - MSA/Gauge R&R（検査信頼性の定量化）
  - AI活用検討（判定基準が明確化された段階で）

Phase 5: 中長期的な検査負荷低減
  - SPC（サプライヤー品質の傾向管理）
  - SQM（サプライヤー工程能力の向上支援）
```

**サプライヤー層別管理**:
| Level | 状態 | 検査対応 |
|-------|------|----------|
| 1 | データなし・実績不明 | 全数検査 or 厳しい抜取 |
| 2 | データ蓄積中・傾向不安定 | 標準的な抜取検査 |
| 3 | 安定した品質実績あり | 軽減検査 |
| 4 | 工程能力証明済み（Cpk≥1.33） | 書類確認中心 |

**現状**: すべてのサプライヤーがLevel 1（定量根拠がないため）

### 06_ai_visual_inspection_comparison.md — AI外観検査サービス比較

**AI導入の前提条件**:
- 明確な判定基準
- 学習用画像データ（最低20-30枚）
- 撮像環境の統一

**現状の課題**: 判定基準の明文化と検査記録の蓄積が未整備 → AIモデルの学習データを準備できない

**サービス比較**:
| サービス | 難易度 | コスト目安 |
|----------|--------|-----------|
| SageMaker Canvas（ノーコード）| 低 | ~$100-200/月 |
| Google Visual Inspection AI | 低〜中 | 要問い合わせ |
| マルチモーダルLLM（Claude等）| 低 | 数千円〜数万円/月 |
| 専業ベンダー（Landing AI等）| 低 | 初期¥120万〜、月¥10万〜 |

**結論**: 今すぐAIを導入すべきではない。Phase 3以降に検討。

### 07_ai_integration_and_cost_analysis.md — AI連携設計・コスト分析

**記録ツール設計時のMust/Should/Could**:

| 区分 | 項目 |
|------|------|
| **Must** | 検査画像の保存機能、画像とメタデータの紐付け、不良モードの統一分類コード、良品/不良品のラベル付け、データエクスポート機能 |
| **Should** | 撮像条件の標準化、S3互換ストレージ、AI判定結果の記録フィールド、フィードバックUI |
| **Could** | AI推論API呼び出し、ダッシュボード、サプライヤースコアカード |

| 削減シナリオ | 削減率 | 月間削減額 |
|-------------|-------|-----------|

### 08_lean_improvement_proposals.md — 少人数で最大効果を出すアプローチ

**4つの提案**:

| 提案 | 概要 | コスト | 即効性 |
|------|------|--------|--------|
| A | 抜取検査計画の見直しだけで工数削減 | ゼロ | ◎ |
| B | Googleフォーム+スプレッドシートで記録基盤 | ゼロ | △ |
| C | マルチモーダルLLMでAI PoC | ¥5,000-20,000/月 | △ |


### 09_updates_and_supplements.md — 補足・修正差分

**追加で判明した事実**:
1. 記録はExcelと紙が混在（網羅的なデジタルデータなし）
2. 表記揺れの精査はある程度完了している
3. プロトタイプの記録ツールを作ろうとしていた
4. ベンチャーで人が少ない

**判断基準**:
- プロトタイプが1-2週間以内に現場投入できる → プロトタイプ続行
- それ以上かかる → Googleフォームで即日開始

**次回確認すべきこと**:
1. 既存Excelの中身（何が記録されていて、何ヶ月分あるか）
2. プロトタイプの現状（どこまで作ったか）
3. 現場の声（一番時間がかかっている部品・工程）
4. 社内AIの構成（AWS？GCP？オンプレ？）
5. セキュリティポリシー（画像をクラウドに保存してよいか）

## 著名人の視点での整理

### デミング（W. Edwards Deming）

**視点**: PDCAサイクル、統計的品質管理、システム思考

**本案件への適用**:
- 「記録がなければPDCAは回らない」→ 記録基盤は正当化される
- しかし「何を記録すべきか」を先に定義すべき
- 検査員個人を責めるのではなく、システム（プロセス）を改善せよ

### ジュラン（Joseph Juran）

**視点**: 品質三部作（品質計画・品質管理・品質改善）

**本案件への適用**:
- 検査は「品質管理」の活動
- 「品質改善」に進むには「品質計画」（何を達成したいか）が先
- 「重要な少数」に集中せよ（パレート分析）

### 石川馨

**視点**: 特性要因図、QCサークル、全員参加

**本案件への適用**:
- 「なぜ300hかかるのか」を4Mで構造化する
- 現場の声を聞く。検査員が「何に困っているか」を把握
- データに基づいて要因を特定し、優先順位をつける

### 田口玄一

**視点**: 品質工学、ロバスト設計、損失関数

**本案件への適用**:
- 検査で品質を確保するより、上流（設計・サプライヤー）で品質を作り込む方が本質
- サプライヤーの工程能力向上（SQM）が中長期的な解
- 検査コストも「品質コスト」の一部として捉える
