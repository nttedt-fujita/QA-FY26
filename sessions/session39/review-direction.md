# M3プロジェクト見直し方針

**作成日**: 2026-03-09
**目的**: Session 1-38で進めてきた内容を、m3-research-filesの視点で見直す方針を整理

---

## 1. 見直しの背景

### これまでの暗黙の前提

Session 1-38では、以下の前提で進めてきた:

> 「記録ツール（DB化）を作れば、改善サイクルが回るようになる」

### m3-research-filesで明確化された視点

> **Problem Statement**: 300hが「多すぎるのか適切なのか」を判断する根拠がない。検査の有効性が定量化されていない。

**核心的な指摘**:
- ツールは手段であり目的ではない
- 「ツールを作れば問題が解決する」は誤り
- 「問題を解決するプロセスの中で、記録基盤としてのツールが必要になる」が正しい

---

## 2. 見直すべき仮説（3つ）

### 仮説1: このリポジトリで定義した「要求」は本当に正しいか？

**検証ポイント**:
- EARS要求仮説（Session 7-8）は「ツールの機能」に偏っていないか
- 本当のNeed（変えちゃいけないもの）は何か

**参照**: [05-requirement-first.md](~/.claude/rules/05-requirement-first.md) の What vs How

### 仮説2: To-Beモデルの前提は妥当か？

**検証ポイント**:
- 現場ヒアリング（一次調査）なしで設計を進めていないか
- As-Isの理解が不十分なままTo-Beを設計していないか

**参照**: DMAIC の Define → Measure の順序

### 仮説3: プロトタイプの目的は何か？

**検証ポイント**:
- Session 38で「デモ・ヒアリング叩き台」と定義した
- しかし、Phase 1（現状把握）の道具として位置づけ直すべきか
- 「何を検証するためのプロトタイプか」が明確か

## 3. 見直しのアプローチ

### 3.1 特性要因図（石川馨）で原因を構造化

「なぜ検査に300hかかるのか」を4M（Man, Machine, Method, Material）で整理する。

```
検査工数300h
├── Man（人）
│   ├── 検査員のスキルばらつき
│   ├── 判定基準の暗黙知依存
│   └── 記録作業の負荷
├── Machine（設備）
│   ├── 検査機器の制約
│   └── 記録システムの不在
├── Method（方法）
│   ├── 抜取検査計画の根拠不明
│   ├── 不合格時フローの非効率
│   └── サプライヤー層別管理なし
└── Material（材料/対象）
    ├── 部品種類の多さ
    ├── サプライヤー品質のばらつき
    └── 不良モードの多様性
```

**次セッションでやること**: 一次調査チェックリスト（02）を使って、各要因の実態を確認

### 3.2 著名人の視点で考える

| 人物 | 視点 | 本案件への適用 |
|------|------|---------------|
| **デミング** | PDCAサイクル、データに基づく意思決定 | 記録がないとPDCAが回らない。記録基盤は必要。ただし「何を記録すべきか」を先に定義すべき |
| **ジュラン** | 品質三部作（計画・管理・改善）| 検査は「管理」。改善に進むには計画（何を達成したいか）が先 |
| **石川馨** | 特性要因図、全員参加のQC | 現場の声を聞く。検査員が「何に困っているか」を把握する |
| **田口玄一** | ロバスト設計、品質工学 | 検査で品質を確保するより、上流（設計・サプライヤー）で品質を作り込む方が本質 |

### 3.3 DMAICフェーズの現在地を明確化

```
Define ─→ Measure ─→ Analyze ─→ Improve ─→ Control
  │
  └── 今ここ（一次調査前）

Phase 1: 現状把握 ← 今ここ
Phase 2: 記録基盤の構築 ← プロトタイプはここに位置づけ
Phase 3: データ分析・検査最適化
Phase 4: 検査プロセス高度化（AI等）
Phase 5: 中長期的な検査負荷低減
```

## 4. プロトタイプの位置づけ（再定義）

### 変更前（Session 38まで）

> To-Beモデルを実装し、デモ・ヒアリング叩き台として使う

### 変更後（m3-research-files反映）

> Phase 2（記録基盤構築）のための最小限のツール
> 目的は「Measureに必要なデータを収集できるようにする」こと

### プロトタイプで検証すべきこと

1. **検査記録の入力フローは現場で使えるか**（操作性）
2. **記録すべきデータ項目は妥当か**（一次調査後に修正前提）
3. **画像保存の仕組みは機能するか**（将来のAI連携の布石）

### プロトタイプで検証しないこと（延期）

- To-Beモデルの完全な実装
- ダッシュボードの高度な分析機能
- AI連携

---

## 5. 次セッションでやること

1. **一次調査の準備**
   - チェックリスト（02）を現場用に整理
   - ヒアリング対象者・スケジュールの検討

2. **プロトタイプの設計見直し**
   - 07のMust/Should/Couldを反映
   - 最小限のスコープを再定義

3. **特性要因図の作成**
   - 仮説ベースで骨格を作成
   - 一次調査で検証する項目を明確化

---

## 6. 維持する方針

以下はm3-research-filesを踏まえても変更しない:

| 項目 | 理由 |
|------|------|
| プロトタイプを作る | 記録基盤として必要。ただし目的を再定義 |
| Goバックエンド | 技術選定の根拠は妥当 |
| TDDアプローチ | 品質確保の手法として有効 |
| 品質管理フレームワーク（IQC/ロット等）の知識 | To-Be設計時に必要 |
