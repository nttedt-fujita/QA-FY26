## 主な発見

### 1. サプライヤーロット番号の実態

**現状**:
- ロット管理自体がこれまでされていない（サプライヤーロット・社内ロット共に）
- 中国製部品はサプライヤー側でロット番号を発行していないことが多い
- 国産化を進めようとしているが、中国製部品も必ず残る

**判断**:
- サプライヤーロット番号フィールド追加は保留
- 現状の運用に合わない（NULLが大半を占める）
- ワークショップ後に再検討

**ドメインモデルの背景**:
- 入荷時点で社内ロットを定義する設計（`lots.lot_id`）
- サプライヤーロットではなく、社内管理のロット

### 2. 工数可視化の問題

**課題**:
- 田原さんに聞いても正確な数字が出せない
- この「分からない」状態自体が問題
- 工数記録の仕組みが存在しない
- 分単位で作業者が記録を取るのは現実的でない
- 勤怠管理・工数入力の仕組み自体から見直しが必要

**方針**:
- まず「何の作業をしているか」（プロセス）を正確に把握
- 工数の数字は後回し（仕組みを作ってから測定）

### 3. AI導入方針の転換

**当初の想定**:
- 員数確認のAI化でコスト削減

**実態**:
- 梱包変更作業でのコスト削減の方が効果が高い
- 作業時間の可視化がされていない（月に何人日かかっているかすら不明）
- どの作業にどれくらい時間がかかっているか分からない

### 4. プロセスの粒度が粗すぎる

**現在のSIPOC**:
1. 部品入荷
2. 検査準備（梱包確認）
3. 検査実施（外観・員数）
4. 記録作成（Excel入力）
5. 入庫/不合格処理

**問題**:
- 抽象的すぎて、クローズドクエスチョンができない
- 「これでいいか？」と聞けるレベルまで分解できていない

**必要な粒度**（例）:
1. 部品入荷（納品受け取り）
2. 作業場所への移動
3. 検査段取り（場所・人員確保）
4. 開梱作業
5. **梱包変更作業**（隠れコスト）
6. 員数確認
7. 外観検査
8. 測定（ノギス等）
9. 入庫のための梱包作業
10. 入庫

### 5. 全体フローの把握が必要

**受入検査だけでなく**:
- 発注 → 入庫予定決定 → 納品 → **受入検査** → 入庫 → 出庫 → 組立 → 出荷検査 → 倉庫 → 出荷
- 全体フローを可視化することで、隠れコストや問題箇所を特定

**次セッションで作成**:
- 全体フロー確認用SIPOC（発注 → 出荷まで）
- 受入検査部分の詳細化

## 学び・気づき

### 1. 現場の実態を確認せずに設計判断しない

**Session 264での判断**:
- サプライヤーロット番号を「優先度A（必須）」と判断
- トレーサビリティの観点から必要と考えた

**実態**:
- ロット管理自体がされていない
- 中国製部品はサプライヤーロットなし
- 追加しても大半がNULL

**教訓**: 現場の運用実態を確認してから設計判断する

### 2. 工数可視化の仕組みがないことが根本問題

**AI導入の前提**:
- どの作業にどれくらい時間がかかっているか把握できている

**実態**:
- 工数の可視化ができていない
- 田原さんに聞いても正確な数字が出せない

**教訓**: 仕組みを作ってから測定する。測定できない状態でAI導入を検討しても効果測定ができない

### 3. プロセスの粒度は現場が答えられるレベルまで分解

**抽象的な記述**:
- 「検査準備（梱包確認）」

**クローズドクエスチョンができる記述**:
- 「開梱作業」「梱包変更作業」「員数確認」「外観検査」

**教訓**: 「これでいいか？」と聞けるレベルまで分解する
