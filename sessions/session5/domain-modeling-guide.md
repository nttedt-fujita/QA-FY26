# ドメインモデリング 学習・準備ガイド

**作成日**: 2026-03-04 (Session 5)
**目的**: M3/M4のDB設計前に、業務ドメインを正しく理解・モデル化する

---

## 1. なぜドメインモデリングか

| やり方 | リスク |
|--------|--------|
| いきなりDB設計 | 現場の業務に合わないテーブルができる |
| **ドメインモデリング → DB設計** | 業務の言葉でモデルを作り、それをDBに落とす |

Session 4で確認した方針:
> DB設計の前にドメインモデリング（ユビキタス言語→モデル→DB設計）が必要

## 2. DDD（ドメイン駆動設計）の基本概念

### 品質管理ドメインに必要な概念だけ抜粋

| 概念 | 説明 | M3/M4での例 |
|------|------|-------------|
| **ユビキタス言語** | チーム全員が同じ意味で使う言葉 | 「ロット」「検収」「NG品」 |
| **エンティティ** | IDで一意に識別されるもの | ロット、部品、サプライヤ |
| **値オブジェクト** | IDを持たず、値で比較されるもの | 測定値、判定結果(PASS/FAIL) |
| **集約** | 一貫性を保つエンティティの塊 | ロット + そのロットの検査結果 |
| **境界づけられたコンテキスト** | 同じ言葉が違う意味を持つ領域の境界 | 受入検査 vs 工程検査 |

### 使わなくてよい概念（今の段階では）

- リポジトリパターン → DB設計フェーズで考える
- ドメインイベント → 将来の拡張で必要になったら
- サガ/CQRS → 小規模システムには過剰

**ポイント**: DDDのすべてを適用する必要はない。「ユビキタス言語」と「モデル図」が最も価値が高い。

## 3. 品質管理ドメインのモデリング手順

### Step 1: ユビキタス言語の収集（ヒアリングと並行）

ヒアリング時に現場で使われている言葉を集める。

#### 収集のコツ

| やること | やらないこと |
|---------|-------------|
| 現場の人が使う言葉をそのまま記録 | 自分でIT用語に置き換えない |
| 「それって〇〇のことですか？」と確認 | 意味を推測して進めない |
| 同じ言葉が違う意味で使われていたら両方記録 | 曖昧なまま放置しない |

#### ユビキタス言語集テンプレート

| 用語 | 定義 | 同義語・別名 | 使用コンテキスト | 確認状態 |
|------|------|------------|----------------|---------|
| ロット | （ヒアリングで確認） | ？ | 受入検査/工程 | 未確認 |
| 検収 | （ヒアリングで確認） | 受入検査？ | ？ | 未確認 |
| NG品 | （ヒアリングで確認） | 不良品？不適合品？ | ？ | 未確認 |
| 返品 | （ヒアリングで確認） | ？ | ？ | 未確認 |
| 再検査 | （ヒアリングで確認） | ？ | ？ | 未確認 |
| サイレントチェンジ | FW/HWバージョンが事前通知なく変更されること | 仕様変更？ | 受入検査 | 仮定義 |

**目標**: ヒアリング後に20-30語を収集

### Step 2: ドメインモデル図の作成

概念とその関係を図にする（UMLクラス図に近い形式）。

#### 初期の概念モデル（仮説）

```
[サプライヤ] 1──*[発注]
                    │
                    1
                    │
[部品] 1──*[入荷ロット] 1──*[検査結果]
                │                 │
                │                 └── 測定値、判定
                │
                1
                │
           *[不良レコード]
                │
                ├── 不良コード（3階層）
                ├── 原因コード（4M1E）
                └── 処置（修理/廃棄/返品）
```

**注意**: これは仮説。ヒアリング後に修正する。

### Step 3: 業務フローの整理

現在の業務の流れを時系列で可視化する。

#### 受入検査フロー（仮説）

```
入荷通知 → 入荷確認 → 数量確認 → サンプル抜取 → 検査実施
    → 合否判定 → [合格] → 倉庫入庫 → 記録
                → [不合格] → 不適合処理 → 記録
```

#### 工程不良フロー（仮説）

```
不良発見 → 不良報告 → 分類（不良コード付与）
    → 原因分析 → 処置（修理/廃棄/返品）
    → 記録 → [該当ロットの追跡]
```

### Step 4: レビュー

モデル図を石川さん・小板橋さんに見せて確認:
- 「この理解で合っていますか？」
- 「この言葉は現場で使いますか？」
- 「抜けている概念はありますか？」

## 4. モデルからDB設計への変換

ドメインモデルが確定した後、以下の順で変換する:

```
概念モデル（ドメインモデル）
    ↓  概念をテーブルに対応づけ
論理モデル（ER図）
    ↓  型、制約、インデックスを決定
物理モデル（CREATE TABLE文）
```

| 変換ルール | 例 |
|-----------|-----|
| エンティティ → テーブル | ロット → incoming_lots テーブル |
| エンティティの属性 → カラム | ロット数量 → lot_qty カラム |
| 1対多の関連 → 外部キー | ロット→検査結果 = lot_id FK |
| 値オブジェクト → カラムまたは別テーブル | 判定(PASS/FAIL) → ENUM型 |
