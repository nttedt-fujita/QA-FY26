# ログ分析方針

**目的**: FE/BEの時間ずれ問題を理解し、傾向を把握する

---

## 問題の認識

- FEで「30秒検査終了」と表示された後も、BEは数十秒動き続けている
- FEのポーリング間隔（1秒）とBEの処理時間（約6秒）の不整合が原因の可能性

## 分析手順

### 1. 各検査のログを抽出・分割

```bash
# 例: 2回目の検査（06:21:49〜06:22:19頃）
sed -n '1200,1400p' prototype/m1-gnss/backend/logs/gnss-backend.log.2026-03-13 > session176/logs/inspection-02.log
```

### 2. 各検査で確認すること

| 項目 | 確認内容 |
|------|----------|
| 検査開始推定時刻 | 最初のGNSS-STATE API呼び出し |
| FE終了時刻 | POST /api/outdoor-inspection-results |
| BE終了時刻 | 最後のGNSS-STATE完了ログ |
| FE/BEのずれ | BE終了 - FE終了 |
| API呼び出し回数 | GNSS-STATE API呼び出し開始の数 |
| 1回あたり処理時間 | 全メッセージ取得完了の総時間 |

### 3. 傾向を整理

全検査のデータを表にまとめ、傾向を確認する:
- ずれは一定か、増加傾向か
- 処理時間のばらつき
- 競合の発生頻度
