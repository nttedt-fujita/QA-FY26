# TDD ワークフローにおける「承知の上で踏み倒す」判断についてのメモ

日付: 2026-03-13

## きっかけ

mizchi氏のポスト（2026-03-12）より:

> C#のヘルスバーグ先生もMoonBitのbobzhang1988先生も、計算機理論としては完全でないことを承知で実用性のために踏み倒している部分があり、その取捨選択の上手さがホビー言語との差である

## 核心の考え方

**「知らずに妥協する」と「知った上で妥協する」は全く別物**

- 理論的完全性を追い求めると、実用的なアウトプットから遠ざかる
- 重要なのは「どこを妥協するか」の取捨選択の質
- これは言語設計だけでなく、日常の開発判断・QA判断にそのまま当てはまる

## 今のTDDワークフローで感じているジレンマ

### 現在の3フェーズ構成

1. **文脈・背景理解** → 承認
2. **振る舞いの記述** → 承認
3. **テストリスト作成** → 承認 → 実装

### 問題

- 各フェーズで「抜け漏れはないか？」と聞くと、毎回何か見つけてくる
- しかしその中には **本当に重要な見落とし** と **聞かれたから絞り出しただけのもの** が混在している
- 区別するコストが負担になっている
- 全フェーズで漏れゼロを目指すのは「ホビー言語的アプローチ」（理論的完全性の追求）に陥るリスクがある

### LLMの構造的特性として理解すべきこと

- 「抜け漏れはないか？」と聞かれると、**なくても何か見つけようとするバイアス**がある
- 「ありません」より「こういう観点もあります」の方が"良い回答"として学習されているため
- 聞くたびに何か出てくるのは構造的に当然

## 改善の方向性: フェーズごとに「判断基準」を明示する

各フェーズで守るべきスコープを限定し、「抜け漏れありますか？」ではなく「**この判断基準に照らして、優先度の高い見落としはありますか？**」に変える。

### フェーズ別の判断基準（案）

| フェーズ | スコープの限定 |
|---|---|
| 文脈理解 | 「このフィーチャーが解決するユーザーの問題は何か」だけに絞る |
| 振る舞い記述 | 「公開インターフェースから観測できる振る舞いだけ」に限定する |
| テストリスト | 「正常系・異常系・境界値の3カテゴリで各2〜3件」のように上限を設ける |

### 期待される効果

- Claudeが絞り出してくるノイズが減る
- 出てきたものの重要度を判断しやすくなる
- 「ここまでは承知の上で踏み倒す」ラインが明確になる

## 次のアクション

- [ ] 上記の判断基準をTDDルールファイルに反映する
- [ ] 実際のタスクで試して、判断基準の粒度が適切か検証する
- [ ] 「聞かれたから出した」ノイズが減ったか体感を確認する
