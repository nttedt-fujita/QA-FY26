## 1. 今回の作業で起きた問題

### 問題1: テーブル設計から始めてしまった

**状況**:
- Session 61で「DB設計案の検討」があったため、テーブル設計（measurement_sessions, nav_pvt_records等）をいきなり提案
- ユーザーから「ドメインモデリングとテーブル設計は別だろ」と指摘

**学び**:
- ドメインモデリング（概念整理）→ テーブル設計（技術実装）の順序が重要
- 技術非依存で概念を整理してから、技術に落とし込む

### 問題2: 要求資料だけを見てモデリングした

**状況**:
- 10-tool-design-notes.mdの自動化マッピング表だけを見て概念を列挙
- ユーザーから「資料に書いてないことも考慮」と指摘

**学び**:
- 要求資料は「何を記録するか」であって「ドメインの全体像」ではない
- ユースケース、評価対象、評価条件、評価者、評価結果など、暗黙の概念がある

### 問題3: 叩くたびにホコリが出た

**状況**:
- 指摘されるたびに抜けが見つかる（NAV-STATUS, DOP, デバイス, 評価条件...）
- 体系的なチェックがなく、場当たり的に修正

**学び**:
- 実際の作業フローを追って検証する方法が有効だった
- チェックリスト的なアプローチが必要

## 2. 有効だったアプローチ

### 作業フロー追跡法

実際の評価作業フローを追って、各ステップで必要な概念を洗い出す。

```
1. 評価対象のGNSSモジュールを用意 → デバイス
2. テスト環境セットアップ → 評価条件
3. ツール起動 → 計測開始 → セッション
4. データ収集 → 計測データ
5. 計測終了 → セッション終了
6. データ分析 → 集計・統計
7. 判定 → 評価基準、評価結果
8. レポート → 出力
```

## 3. スキル化で考慮すべき観点

### 3-1. ドメインモデリングの手順案

1. **目的の明確化**: このシステムは何のためにあるか（What）
2. **ユースケース列挙**: 誰が、いつ、何をするか
3. **作業フロー追跡**: 実際の作業を追って概念を洗い出す
4. **要求資料との突合**: 資料に明示されている概念を確認
5. **暗黙の概念の発掘**: 主語（誰が）、対象（何を）、条件（どこで、いつ）、結果（どうなったか）
6. **関係性の整理**: エンティティ間の1:N, N:M関係
7. **スコープ判断**: プロトタイプで何を含めるか

### 3-2. チェックリスト案

- [ ] 評価/操作の**主体**（誰が）は明確か？
- [ ] 評価/操作の**対象**（何を）は明確か？
- [ ] **条件・文脈**（どこで、いつ、どんな環境で）は考慮したか？
- [ ] **結果・判定**（どうなったか）は考慮したか？
- [ ] **時間軸**（開始、終了、履歴）は考慮したか？
- [ ] 実際の**作業フロー**を追って検証したか？
