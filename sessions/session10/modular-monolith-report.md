# モジュラーモノリス調査レポート

**作成日**: 2026-03-05
**目的**: M3/M4開発におけるアーキテクチャ選択の判断材料

---

## 1. モジュラーモノリスとは何か

### 定義

モジュラーモノリスは、**単一のコードベース・単一のデプロイ単位**でありながら、内部を**疎結合なモジュール**に分割するアーキテクチャパターン。

- 各モジュールは特定のビジネス機能をカプセル化
- モジュール間は明確なインターフェース（API）で通信
- 従来のモノリスとマイクロサービスの中間に位置するハイブリッドアーキテクチャ

### 特徴

| 特徴 | 説明 |
|------|------|
| 単一デプロイ | 全モジュールを一緒にデプロイ |
| モジュール分離 | 各モジュールは独自のビジネスロジックを持つ |
| 疎結合 | モジュール間はパブリックAPIで通信 |
| 高凝集 | 関連する機能が同じモジュールに集約 |

---

## 2. マイクロサービスとの違い

| 観点 | モジュラーモノリス | マイクロサービス |
|------|------------------|----------------|
| **デプロイ** | 単一ユニット | 独立した複数サービス |
| **データベース** | 共有 or モジュールごとのスキーマ | サービスごとに独立 |
| **通信** | インプロセス（関数呼び出し） | ネットワーク経由（HTTP/gRPC等） |
| **インフラ** | 最小限（1サーバーで可） | 複雑（API Gateway、ロードバランサー、サービスディスカバリ等） |
| **チーム** | 小〜中規模チーム向け | 大規模チーム向け |
| **コスト** | 低い | 高い |
| **移行パス** | マイクロサービスへの移行が容易 | - |

---

## 3. いつ使うべきか

### モジュラーモノリスが適するケース

- **スタートアップ / MVP**: 初期段階ではシンプルさが重要
- **限定的なスケーラビリティ要件**: 大規模トラフィックを想定しない
- **小〜中規模チーム**: DevOps専門家がいない場合
- **将来のマイクロサービス移行を見据える**: モジュール境界を明確にしておけば後から分離可能

### マイクロサービスが適するケース

- **大規模・複雑なシステム**
- **チームが大きく、独立したデプロイが必要**
- **NetflixやShopifyレベルのスケール**

## 4. M3/M4への適用評価

### M3/M4の状況

| 項目 | 状況 |
|------|------|
| チーム規模 | 開発者: 藤田さん1人（+ 外注可能性） |
| データ量 | 年間500〜1,500件（Session 8の予測） |
| ユーザー | 品質Gメンバー（数名） |
| スケーラビリティ要件 | 低い |
| 将来の拡張性 | 分析機能の追加可能性 |

### 評価結果

| 評価 | 判断 |
|------|------|
| **モジュラーモノリス適用** | ✅ **適切** |
| **マイクロサービス適用** | ❌ **オーバーキル** |

### 理由

1. **規模が小さい**: Netflix規模のスケールは不要
2. **一人運用**: インフラ複雑性は最小限にすべき
3. **コスト**: マイクロサービスはインフラコストが高い
4. **保守性**: 単一デプロイの方が保守が容易

## 5. 推奨アーキテクチャ

### M3/M4に適した構成

```
[モジュラーモノリス]

  ┌──────────────────────────────────────────┐
  │              単一アプリケーション          │
  │ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
  │ │ 受入検査  │ │ 工程不良  │ │   分析   │  │
  │ │ モジュール│ │ モジュール│ │ モジュール│  │
  │ └──────────┘ └──────────┘ └──────────┘  │
  │      ↓            ↓            ↓        │
  │ ┌──────────────────────────────────────┐ │
  │ │          共有データベース             │ │
  │ └──────────────────────────────────────┘ │
  └──────────────────────────────────────────┘
```

### メリット

1. **シンプルなデプロイ**: 1つのアプリをデプロイするだけ
2. **保守容易**: 一人でも運用可能
3. **低コスト**: 最小構成でOK
4. **将来の拡張**: 必要に応じてモジュールを分離可能

---

## 6. 結論

| 項目 | 結論 |
|------|------|
| **アーキテクチャ選択** | モジュラーモノリス |
| **マイクロサービス** | M3/M4には不要 |
| **理由** | 規模・チーム・コストの観点から最適 |

**補足**: 別セッションで検討された「PWA + API」構成はモジュラーモノリスと矛盾しない。フロントエンド（PWA）とバックエンド（API）を分離しつつ、バックエンドをモジュラーモノリスとして構成することが可能。
