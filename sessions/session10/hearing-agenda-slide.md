# 1. 課題認識の確認

## 確認したいこと

- 今の受入検査・工程不良の記録で、**何が困っていますか？**
- 理想的には**どうなっていてほしい**ですか？

## なぜ確認するか

- 課題認識が揃っていないと、解決策の方向性がズレる
- 「DB化すればOK」ではなく、**何を解決したいか**を先に揃える

# 2. ヒアリング計画

## 誰に聞くか

| 対象 | 聞くこと |
|------|---------|
| **石川さん** | 大枠の認識、分析要件、将来構想 |
| **小板橋さん** | 受入検査の運用詳細、現行Excelの実態 |
| **製造現場担当者** | 工程不良の記録方法、入力負荷 |

## 確認したいこと

- この順番・対象で良いか？
- 他に聞くべき人はいるか？

---

# 3. プロダクトのクオリティ

## 認識を揃えたい

| 観点 | 選択肢 |
|------|--------|
| **完成度** | MVP（最小限）で早く出す？ / しっかり作り込む？ |
| **分析機能** | 基本グラフで十分？ / SPC・管理図まで必要？ |
| **UI** | 使えればOK？ / 使いやすさも重視？ |
| **保守** | 最低限？ / ドキュメント・引き継ぎ準備も？ |

## なぜ確認するか

- 作り込みすぎると時間がかかる
- 足りないと使われない
- **ちょうど良いライン**を合意したい

---

# 4. 次のステップ

## 想定している流れ

```
[今ここ] 認識合わせ
    ↓
ヒアリング（小板橋さん、製造現場）
    ↓
要件整理
    ↓
技術選定の最終判断（kintone or AWS）
    ↓
プロトタイプ開発
```

## 確認したいこと

- この流れで良いか？
- スケジュール感（いつまでに何を？）
