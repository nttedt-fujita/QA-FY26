## 1. プロジェクト概要

### 1.1 背景と目的

現在、受入検査・工程内検査に関する複数のツールを社内で開発中である。これらのツールは現時点ではプロトタイプ段階だが、今後チーム内で日常的に利用するにあたり、以下の3つの課題を解決する必要がある。

- **配布の課題:** M1ツール（GNSS/LiDAR評価）は各検査担当者のWindows PCで動作させる必要がある。非エンジニアでもインストール・利用できるパッケージ化が必要
- **アクセスの課題:** M3/M4ツール（受入検査・工程内検査）は複数ユーザーが利用するため、クラウド上での共有アクセス基盤が必要
- **データの課題:** M1ツールでは複数PCに検査データが分散する。シリアル番号ベースでデータを統合する仕組みが必要

### 1.2 対象ツール一覧

| ID | ツール名・用途 | 配布形態 | ステータス |
|----|---------------|---------|-----------|
| **M1A** | GNSS評価・受入検査ツール | デスクトップ（Windows） | プロトタイプ完了 |
| **M1B** | LiDARセンサー評価・受入検査ツール | デスクトップ（Windows） | 未着手 |
| **M3** | 受入検査ツール（DB化） | クラウド（AWS） | プロトタイプ |
| **M4** | 工程内不良検査ツール（DB化） | クラウド（AWS） | プロトタイプ |

### 1.3 現行技術スタック

| レイヤー | 技術 |
|---------|------|
| バックエンド | Rust（Actix-web） |
| フロントエンド | Next.js |
| データベース | SQLite |
| ハードウェア連携 | USB-to-UART経由 GNSSレシーバー（シリアル通信） |
| 開発環境 | Ubuntu Linux |
| ソースコード管理 | GitHub |

### 1.4 制約条件

- エンドユーザーはプログラム経験なし（インストーラ配布が前提）
- 配布先はWindows PCのみ
- 屋外でのGNSS検査時にネットワーク接続がない可能性がある
- 開発環境はUbuntu Linux（配布先Windowsとのクロスビルドが必要）

## 2. M1ツール パッケージ化設計

### 2.1 解決すべき問題

M1Aは現在、開発者のLinux環境でのみ動作する。これを非エンジニアの検査担当者が自身のWindows PCにインストールし、USBコンバーター経由でGNSSレシーバーに接続して使えるようにする必要がある。

### 2.2 パッケージ化の選択肢

#### 案A：バンドル方式（最小工数）

Actix-webをexeにコンパイルし、Next.jsの静的出力と合わせて Inno Setup でインストーラにまとめる方式。ユーザーはブラウザ上でツールを操作する。

- **ユーザー体験:** インストーラ実行 → デスクトップアイコンクリック → ブラウザが自動で開く
- **利点:** 既存コードの変更がほぼ不要。構造がシンプル
- **欠点:** ブラウザ依存（localhost URLが見える）。サーバープロセスの起動・終了・ポート衝突の管理を自前で実装する必要がある

#### 案B：Tauriサイドカー方式

Actix-webバイナリをTauriのサイドカープロセスとして同梱。Tauriが専用ウィンドウ（WebView）を提供し、プロセスのライフサイクル管理も担当する。

- **ユーザー体験:** インストーラ実行 → デスクトップアイコンクリック → 専用ウィンドウでツールが開く
- **利点:** ネイティブアプリに近いUX。プロセス管理をTauriに委任可能
- **欠点:** Tauriのプロジェクトセットアップとサイドカー設定の学習コストがある

### 2.3 比較と判断

| 観点 | 案A: バンドル | 案B: Tauriサイドカー |
|------|-------------|-------------------|
| 既存コード変更 | ほぼ不要 | ほぼ不要 |
| 追加開発量 | ランチャー実装が必要 | Tauri設定のみ |
| UX品質 | ブラウザベース | 専用ウィンドウ |
| プロセス管理 | 自前実装 | Tauriが管理 |
| インストーラ | Inno Setup | NSIS（Tauri自動生成） |
| 横展開性 | 都度構築 | パターン再利用可能 |

**判断: 案B（Tauriサイドカー方式）を採用**

根拠は3点。

1. **既存コードを維持できる。** サイドカー方式はActix-webバイナリをそのまま横に置く形であり、API層やシリアル通信コードの書き換えが発生しない。「包む」だけでネイティブアプリ化できる
2. **プロセス管理を自前で作らなくて済む。** 案Aではランチャー（プロセス起動、タスクトレイ常駐、ポート衝突回避、終了処理）を自前実装する必要があり、この部分の工数は案Bの学習コストと同等かそれ以上。案BではTauriがこれを引き受ける
3. **M1B・マージツールへの横展開が効く。** Tauriサイドカーの構成パターンを一度確立すれば、M1B（LiDAR）やマージツールにも同じパターンを適用できる。案Aでは都度ランチャーを作ることになる

### 2.4 Tauriサイドカー構成の詳細

#### ディレクトリ構成（インストール後）

```
M1A.exe                ← Tauriメインアプリ（WebView）
resources/
  m1a-server.exe       ← Actix-web（サイドカー）
  m1a.db               ← SQLite
drivers/
  cp2102_driver/       ← USBドライバ
```

#### 既存コードへの影響

- **変更不要:** Actix-web APIロジック、シリアル通信コード（serialportクレート）、SQLite操作、Next.jsフロントエンド全般
- **追加作業:** Tauriプロジェクト初期化（`tauri init`）、`tauri.conf.json` のサイドカー設定（バイナリパス・起動引数）、WebViewの接続先ポート設定

#### シリアルポート通信について

USB-to-UART経由のシリアル通信はサイドカー（Actix-web）側で処理されるため、Tauriのセキュリティモデル（allowlist）とは独立して動作する。既存のserialportクレートのコードに変更は不要。

### 2.5 ビルド・配布パイプライン

#### 課題

開発環境がUbuntu Linux、配布先がWindowsであるため、Windows向けバイナリを生成する仕組みが必要。Linux上でのクロスコンパイルはシリアルポート関連クレートのWindows SDK依存でトラブルが発生しやすい。

#### 方針: GitHub Actions によるWindows CIビルド

クロスコンパイルではなく、GitHub ActionsのWindowsランナー上でビルドを行う。

1. 開発者がGitHubにpush（またはタグ作成）
2. GitHub ActionsのWindowsランナーが起動
3. Tauriの公式アクション（`tauri-action`）がビルド実行
4. `.msi` インストーラがGitHub Releasesに自動アップロード
5. エンドユーザーがReleasesページからダウンロード・インストール

**この方式を選ぶ理由:** Windowsネイティブ環境でビルドするため、シリアルポート周りの依存解決が確実。Tauriが公式CIテンプレートを提供しており、設定コストが低い。また、このCIパイプラインに後述の脆弱性チェックも統合できるため、一石二鳥。

#### USBドライバの同梱

TauriのNSISインストーラにカスタムステップを追加し、USB-to-UARTコンバーター用ドライバ（FTDI / CP2102等）を自動インストールする。非エンジニアユーザーがドライバを別途入手・インストールする手順を排除するため。

> **要確認:** 使用中のUSB-to-UARTコンバーターのチップ種別（FTDI / CP2102 / CH340等）。ドライバの同梱方法に影響する。

## 3. M3/M4 クラウド化設計

### 3.1 クラウド化の動機

M3（受入検査）とM4（工程内不良検査）はドメインモデルが異なるがDBは接続可能な設計。複数ユーザーが同一データにアクセスする運用を想定しているため、ローカルSQLiteではなくクラウド上での稼働が必要になる。加えて、写真ベースのAI不良検査をAWSサービスで実現したいという要件がある。

### 3.2 インフラ構成案

| コンポーネント | 選択肢A（検証段階） | 選択肢B（本番運用） |
|-------------|------------------|-----------------|
| コンピュート | AWS Lightsail | ECS Fargate（コンテナ） |
| データベース | SQLite + EFSマウント | RDS PostgreSQL |
| 画像ストレージ | S3 | S3 |
| AI検査 | Amazon Bedrock（Claude Vision） | Rekognition Custom Labels / SageMaker |

**段階的移行を推奨する理由:** 初期段階でFargate + RDS構成にすると、セットアップ工数とランニングコストの両方が増える。Lightsail + SQLiteでまず動くものをクラウドに載せ、同時アクセスの実需やパフォーマンス要件が見えた時点でRDSに移行する方が、判断材料を持った上で投資できる。

> **注意:** SQLiteは同時書き込みに弱いため、複数ユーザーが同時にデータ登録する頻度が高い場合は、早期にRDS PostgreSQLへの移行を検討すること。

### 3.3 AI不良検査のアプローチ

写真をアップロードしてAIに不良判定させる機能について、以下の段階的アプローチを推奨する。

| Phase | アプローチ | 判断基準 |
|-------|----------|---------|
| **Phase 1（PoC）** | Amazon Bedrock（Claude Vision API）でプロンプトベースの判定 | まずは実現可能性と判定精度のベースラインを確認 |
| **Phase 2（精度向上）** | Rekognition Custom Labels で専用モデルを学習 | Phase 1で精度が要件に満たない場合に移行 |
| **Phase 3（高度化）** | SageMaker でカスタムモデル | 特殊な不良パターンへの対応が必要な場合 |

**Phase 1から始める理由:** Bedrockはモデル学習が不要で、プロンプト（テキスト指示）の調整だけで判定ロジックを構築できる。検査写真のサンプル数が少ない初期段階に適しており、PoCとしての立ち上がりが最も速い。判定精度のベースラインが得られれば、Phase 2以降の投資判断の根拠にもなる。

### 3.4 コスト概算（月額）

アーキテクチャ確定前の概算値。画像処理量・保存量によって大幅に変動する。

| 構成 | 月額目安 | 想定ユースケース |
|-----|---------|---------------|
| Lightsail + SQLite | $20〜50 | 検証・少人数での試用 |
| Fargate + RDS + Bedrock | $50〜150 | 本番小規模運用 |
| 上記 + Rekognition | $150〜300+ | 画像検査量が多い場合 |

> **次のアクション:** アーキテクチャ確定後に AWS Pricing Calculator での詳細試算を実施する。

## 4. 脆弱性チェック体制

### 4.1 なぜ脆弱性チェックが必要か

デスクトップアプリ（M1）もクラウドアプリ（M3/M4）も、依存ライブラリに既知の脆弱性が混入するリスク（サプライチェーン攻撃）がある。RustクレートやnpmパッケージはOSSであり、過去にも脆弱性が報告されている。CI/CDパイプラインに組み込むことで、pushのたびに自動でチェックが走る体制を作る。

### 4.2 レイヤー別チェックツール

| チェック対象 | ツール | コスト | 対象ツール | 導入優先度 |
|------------|-------|-------|-----------|----------|
| Rust依存ライブラリ | `cargo audit` | 無料 | M1A, M1B, M3, M4 | 高（CI構築と同時） |
| npm依存ライブラリ | `npm audit` | 無料 | M1A, M1B, M3, M4 | 高（CI構築と同時） |
| コンテナイメージ | Trivy（OSS） | 無料 | M3, M4（クラウド） | 中（コンテナ化後） |
| アプリケーション層 | OWASP ZAP | 無料 | M3, M4（Web公開時） | 中（クラウド公開後） |
| クラウド設定 | AWS Security Hub / Checkov | 一部有料 | M3, M4 | 低（本番運用時） |

**導入戦略:** `cargo audit` と `npm audit` はGitHub ActionsのCIパイプライン構築（ロードマップStep 2）と同時に導入できるため、最も早期に実現可能。コンテナ・アプリ層・クラウド設定のチェックは、それぞれの基盤が整った段階で順次追加する。

### 4.3 M1ツール（デスクトップ）への適用

デスクトップアプリであっても以下は適用可能であり、推奨する。

- **`cargo audit`:** Rustクレートの既知脆弱性をスキャン
- **`npm audit`:** Node.jsパッケージの既知脆弱性をスキャン
- ネットワーク通信がある場合はOWASP ZAPによる動的スキャンも適用可能

これらはM1A用のGitHub Actions CIに組み込み、追加の運用コストなく自動実行させる。

## 5. データ統合設計

### 5.1 問題の定義

M1Aツールを複数の検査担当者が各自のPCで使用する場合、検査データが各ローカルのSQLiteファイルに分散する。現行のデータ構造では、機器シリアル番号をキーに屋内検査・屋外検査のデータが紐づいているが、この分散したデータを一元的に参照・分析する手段がない。

### 5.2 方式の検討

| 方式 | 概要 | メリット | デメリット |
|-----|------|---------|----------|
| 共有フォルダ | NAS等にSQLiteファイルを配置し全員が参照 | 構成が最もシンプル | SQLiteの同時書き込みでロックエラーのリスク |
| **エクスポート＆マージ** | 各ユーザーがローカルで作業し、マージツールで統合 | オフライン対応可。スケーラブル | マージツールの開発が必要 |
| 中央DBサーバー | PostgreSQL等を社内サーバーに構築 | 同時アクセス問題なし | 屋外検査時のオフライン対応が複雑 |

**判断: エクスポート＆マージ方式を採用**

根拠は2点。

1. **屋外検査のオフライン要件。** GNSS検査は屋外で実施する。その際にネットワーク接続がない可能性があり、中央DBサーバーや共有フォルダへのリアルタイム接続を前提にできない。ローカルDBで完結する方式でなければ、屋外での検査業務そのものが止まるリスクがある
2. **段階的に発展できる。** まずはエクスポート＆マージで運用を開始し、将来M3/M4がクラウド化された時点で、マージ済みデータをクラウドDBにアップロードする導線を追加すれば、全検査データのクラウド集約も実現できる。初期投資を抑えつつ拡張パスを残せる

## 6. マージツール設計

### 6.1 ツールの位置づけ

データ統合のための専用GUIツールを、M1Aと同一技術スタック（Rust + Next.js + SQLite）で開発し、Tauriサイドカー方式で配布する。

**M1Aとは別アプリにする理由:**

- **役割の分離:** M1Aは検査担当者用、マージツールは管理者用。不要な機能が見えることで操作ミスや混乱を招くリスクを避ける
- **拡張性:** 将来M1B、M3、M4のデータ統合にも対応する可能性がある。独立アプリであれば、M1Aのリリースサイクルに影響を与えずにマージツールを拡張できる

### 6.2 画面構成

#### メイン画面

- DBファイルのドラッグ＆ドロップ（または「ファイル追加」ボタン）で複数の `.db` を読み込み
- 読み込んだDBの一覧表示（ファイル名、レコード数、最終更新日時）
- 「マージ実行」ボタン

#### プレビュー画面（マージ前確認）

- シリアル番号ベースで「新規追加されるデータ」「衝突するデータ」を一覧表示
- 衝突データは色分け・アイコンで視覚的に識別
- 衝突の解決方法を選択するUI（マージルールに基づく）

#### 結果画面（マージ後）

- 統合結果のサマリー（追加件数、更新件数、スキップ件数）
- `master.db` のダウンロード / 保存先指定
- 衝突レポート（CSV）のエクスポート

### 6.3 マージ処理フロー

```
ユーザーA.db ─┐
ユーザーB.db ─┼→ マージツール → master.db（統合済み）
ユーザーC.db ─┘      ↓
                  conflict_report.csv（衝突レポート）
```

### 6.4 マージルール（衝突解決）

シリアル番号をキーとした衝突パターンと暫定的な処理方針を以下に整理する。ケース2の詳細ルールは別途要件定義を実施する。

| ケース | 状況 | 処理方針 |
|-------|------|---------|
| ケース1 | 片方のDBにしか存在しないデータ | そのまま取り込み |
| ケース2 | 同じシリアル + 同じ検査種別で複数のDBに存在 | **要件定義にて決定**（候補: タイムスタンプ優先 / 全保持 / 人間判断） |
| ケース3 | 屋内と屋外で別ユーザーが別々に検査 | 正常ケース。シリアル番号で紐づけてマージ |

### 6.5 M1A側の追加機能（エクスポート）

マージツールとの連携のため、M1A側に以下のエクスポート機能を追加する。

- **エクスポートボタン:** 期間指定 or 全件で `.db` / JSON / CSV を出力
- **ソースID付与:** エクスポート時に、どのPCから出たデータかを識別するメタデータを付与。マージ時のトレーサビリティを確保
- **APIエンドポイント:** `/api/export` として Actix-web 側に実装。指定期間 or 指定シリアルのデータをJSON形式で返却

JSON/CSVエクスポートはマージ以外の用途（Excel報告書への取り込み等）にも活用可能。

### 6.6 リポジトリ構成

```
github-org/
├── m1a/                  ← 既存のM1Aツール
├── m1a-merge-tool/       ← マージツール（新規）
└── shared/               ← 共有スキーマ定義、型定義
    └── db-schema/
```

共有のDBスキーマ定義を切り出すことで、M1Aとマージツール間のDB操作の整合性を保つ。将来M1B等を追加する際も同じスキーマ定義を参照する。

### 6.7 将来的なデータ集約

M3/M4をクラウド化した際に、M1のマージ済みデータもクラウドDBにアップロードする導線を構築する。最終的にはM1〜M4の全検査データがクラウド上に集約される形を目指す。

## 7. 実装ロードマップ

各Stepの工数は1名作業の目安。並行作業や習熟度で変動する。

### Phase 1: M1Aデスクトップ配布基盤（目安 4〜6週間）

| Step | 内容 | 工数目安 | 前提 |
|------|------|---------|------|
| **Step 1** | **M1A Tauriセットアップ** | 2〜3週間 | — |
| | - 既存リポジトリにTauriプロジェクトを追加（`tauri init`） | | |
| | - `tauri.conf.json` でサイドカー設定（Actix-webバイナリ指定） | | |
| | - WebViewが `localhost:{port}` を開く設定 | | |
| | - Linux上で動作確認 | | |
| **Step 2** | **GitHub Actions CIパイプライン** | 1週間 | Step 1完了 |

## 8. 備考・オープンな課題

| 項目 | 内容 | 影響範囲 | 対応時期 |
|-----|------|---------|---------|
| Next.js出力方式 | SSR / API Routesの使用状況により static export 可否が変わる。Tauriサイドカー方式なら影響なし | M1パッケージ化 | Step 1着手時に確認 |
| USBドライバチップ種別 | FTDI / CP2102 / CH340 等の確認が必要。ドライバ同梱方式に影響 | M1パッケージ化 | Step 3着手前に確認 |
| M3/M4 DB設計 | ドメインモデルは別だがDB接続可能な設計。クラウド移行時のスキーマ移行計画が必要 | M3/M4クラウド化 | Step 4着手時 |
| M1Bへの横展開 | M1AでTauriサイドカーのパターンを確立後、LiDARツールへ同一パターンを適用 | M1B開発 | Phase 1完了後 |
| コスト試算の精緻化 | アーキテクチャ確定後に AWS Pricing Calculator での詳細試算を実施 | M3/M4クラウド化 | Step 4完了時 |
| マージルール詳細 | 同一シリアル + 同一検査種別の衝突時の解決方針。別途要件定義を実施 | データ統合 | Step 7着手前 |
| クラウド集約移行計画 | M1ローカルデータ → マージ → クラウドアップロードの段階的移行パスを具体化 | 全体 | Phase 3完了後 |
