## 1. QA（品質保証）とQC（品質管理）の違い

### 定義

| 項目 | QA（Quality Assurance / 品質保証） | QC（Quality Control / 品質管理） |
|------|-----------------------------------|--------------------------------|
| 定義 | 製品・サービスの性能や機能を**保証するための活動** | 製品・サービスの品質に問題がないかを**製造プロセスの段階から確認する活動** |
| 視点 | **ユーザー視点** — 顧客の期待に応えているか | **現場の作り手視点** — 工程内で品質問題を出さない |
| 範囲 | 企画 → 設計 → 製造 → 出荷 → アフターフォロー（全ライフサイクル） | 製造 → 出荷（製造工程が中心） |
| 時間軸 | 長期的・継続的 | 工程ごと・短期的 |

> **ソース抜粋**: 「品質管理は品質保証の業務の一つ」 — 現代の品質管理の実務では、QCはQAのサブセットとして位置づけられている。
> 出典: [ISOプロ — QAとQCの違い](https://activation-service.jp/iso/column/7762)

### ISO 9000での体系

ISO 9000の世界では以下の関係が成り立つ：

```
品質マネジメント（QM: Quality Management）
  ├── 品質管理（QC: Quality Control）
  ├── 品質保証（QA: Quality Assurance）
  └── 品質改善（QI: Quality Improvement）
```

> **ソース抜粋**: 「ISO 9000の世界では、QM（品質マネジメント）＝QC（品質管理）＋QA（品質保証）＋QI（品質改善）」
> 出典: [Tqm9000](https://www.tqm9000.com/news/2019/11/25/iso-qms-misund2-30/)

### ソフトQAとして理解すべきポイント

藤田さんはソフト開発からソフトQAに異動したため、以下の視点転換が重要：

| ソフト開発者の視点 | ソフトQAの視点 |
|-------------------|---------------|
| 「コードが正しく動くか」 | 「製品全体として品質要求を満たしているか」 |
| 単体テスト・結合テスト | システムテスト・受入テスト・フィールド検証 |
| バグを直す | バグが出ない**仕組み**を作る |
| 機能実装 | 検証手法の策定・データに基づく判断 |

## 2. ISO 9001:2015 品質マネジメントシステム

### 品質マネジメント7原則

ISO 9001:2015で規定されている7つの品質マネジメント原則：

| # | 原則 | 英語 | 内容 |
|---|------|------|------|
| 1 | **顧客重視** | Customer Focus | 顧客の現在・将来のニーズを理解し、要求事項を満たし、期待を超える |
| 2 | **リーダーシップ** | Leadership | トップマネジメントがQMSの構築・実施・継続的改善に責任を持つ |
| 3 | **人々の積極的参加** | Engagement of People | 組織の全員が参加し、顧客満足の目標達成における自分の役割を理解する |
| 4 | **プロセスアプローチ** | Process Approach | 業務をプロセス単位で管理し、非効率を特定して継続的に改善する |
| 5 | **改善** | Improvement | 現在うまくいっていても、常に改善の方法を探求する |
| 6 | **客観的事実に基づく意思決定** | Evidence-Based Decision Making | 経験や勘だけでなく、分析されたデータと情報に基づいて判断する |
| 7 | **関係性管理** | Relationship Management | 全てのステークホルダーが互恵的で対等な関係を維持する |

> 出典: [ISOプロ — 品質マネジメント7原則](https://activation-service.jp/iso/column/1729)

**藤田さんへのポイント**: 特に **原則6「客観的事実に基づく意思決定」** が重要。センサー評価もDB化も、すべて「データに基づいて品質を判断する」ための活動。

### PDCAサイクル

ISO 9001の根幹をなす継続的改善のフレームワーク：

```
Plan（計画）→ Do（実行）→ Check（評価）→ Act（改善）
     ↑                                        │
     └────────────────────────────────────────┘
```

藤田さんのミッションに当てはめると：

| フェーズ | センサー評価の例 |
|---------|----------------|
| **Plan** | 評価手法・基準値・試験計画の策定 |
| **Do** | 実際にセンサーを試験する |
| **Check** | 試験結果をデータベースに記録・分析 |
| **Act** | 基準未達なら改善要求、手法改良 |

## 3. 受入検査（IQC: Incoming Quality Control）

### 概要

受入検査とは、サプライヤから納入された部品・材料が自社の品質基準を満たしているかを確認する工程。

### AQL（Acceptable Quality Level / 合格品質限界）抜取検査

全数検査が現実的でない場合に、統計的に合理的なサンプル数で合否判定を行う手法。

**関連規格**: JIS Z 9015-1 (= ISO 2859-1)「計数値検査に対する抜取検査手順」

#### 抜取検査の3ステップ

> **ソース抜粋**:
> 1. **ロットサイズ・AQL・検査水準を確認** — ロットの大きさ、クライアントが定めたAQL値、検査水準を把握
> 2. **サンプル文字を特定** — ロットサイズと検査水準の交点からサンプル文字を決定
> 3. **サンプルサイズと合格判定個数を特定** — サンプル文字とAQL値の交点から、検査すべき数と合格判定個数(Ac)/不合格判定個数(Re)を決定
>
> 出典: [QTEC — AQLの知識](https://www.qtec.or.jp/knowledge/inspection/aql/)

#### 具体例

| 条件 | 値 |
|------|-----|
| ロットサイズ | 500個 |
| 検査水準 | II（通常） |
| AQL | 2.5% |
| → サンプル文字 | H |
| → サンプルサイズ | **50個** |
| → 合格判定個数(Ac) | 3個以下 → 合格 |
| → 不合格判定個数(Re) | 4個以上 → 不合格 |

### 検査の3段階切り替え

```
    通常検査 ←→ きつい検査（不良多発時）
        ↕
    ゆるい検査（品質安定時）
```

## 4. 工程不良管理

### パレート分析

不良の発生を「重要な少数と取るに足りない多数」に分類する手法。

```
不良件数
│ ████████████  (40%)  はんだ不良
│ ████████       (25%)  部品ズレ
│ ██████         (15%)  外観キズ
│ ████           (10%)  動作不良
│ ██              (5%)  その他
│ █               (5%)  ...
└─────────────────────────── 不良モード
```

**80-20の法則**: 不良の80%は20%の原因に起因する → 重点管理すべき項目を特定

### QC7つ道具

品質管理の基本ツール群。データに基づく問題分析・改善に使用：

| # | 道具 | 用途 |
|---|------|------|
| 1 | **チェックシート** | データ収集・記録の標準化 |
| 2 | **パレート図** | 重点管理項目の特定 |
| 3 | **特性要因図（魚骨図）** | 問題の原因分析（Man/Machine/Material/Method） |
| 4 | **ヒストグラム** | データの分布・ばらつきの把握 |
| 5 | **散布図** | 2変数間の相関分析 |
| 6 | **管理図** | 工程の安定性監視（SPC） |
| 7 | **グラフ** | データの可視化・トレンド把握 |

### 8D問題解決手法

重大な品質問題が発生した際の体系的な解決プロセス：

| Step | 内容 | 説明 |
|------|------|------|
| D0 | 準備 | 問題の認識、チーム編成の判断 |
| D1 | チーム編成 | 関連知識を持つメンバーでチームを組む |
| D2 | 問題の定義 | 5W2Hで問題を明確に記述 |
| D3 | 暫定対策 | 顧客への影響を最小化する応急処置 |
| D4 | 根本原因分析 | なぜなぜ分析、特性要因図で真因特定 |
| D5 | 恒久対策の選定 | 根本原因を除去する対策の立案・検証 |
| D6 | 恒久対策の実施 | 対策の実施と有効性確認 |
| D7 | 再発防止 | 水平展開、標準化、教育 |
| D8 | チーム評価 | 活動の振り返りとチームへの感謝 |

### SPC（Statistical Process Control / 統計的工程管理）

> **ソース抜粋**: 「SPC（Statistical Process Control）は、製品の品質を安定的に維持し、不良品を減らすために、製造工程を統計的手法で管理するアプローチ」
> 出典: [ITトレンド — SPC](https://it-trend.jp/process_management/article/464-0007)

管理図を用いて工程が「統計的管理状態」にあるかを監視する：

```
     UCL (上方管理限界) ─────────────
        ・    ・
  CL  (中心線) ─── ・ ─── ・ ─── ・ ────
              ・         ・
     LCL (下方管理限界) ─────────────
```

管理限界外にデータが出た場合 → 異常原因の調査が必要

## 5. FMEA（Failure Mode and Effects Analysis / 故障モード影響解析）

### 概要

> **ソース抜粋**: 「FMEAは "Failure Mode and Effects Analysis" の頭文字を取った略称で、日本語では "故障モード影響解析" と呼ばれる」
> 出典: [skillnote — FMEA](https://skillnote.jp/knowledge/fmea/)

### 設計FMEAと工程FMEAの違い

| 項目 | 設計FMEA（DFMEA） | 工程FMEA（PFMEA） |
|------|-------------------|-------------------|
| 対象 | 製品の**設計段階** | **製造工程** |
| 分析対象 | 製品を構成する**部品** | **作業内容・プロセス** |
| 目的 | 潜在的故障を予測し**設計改善** | 工程の不具合を低減し**品質維持** |
| 例 | 「バッテリーセルが膨張する」 | 「はんだ温度が規定外になる」 |

### RPN（Risk Priority Number / リスク優先度指数）

```
RPN = 影響度(S) × 発生頻度(O) × 検出難易度(D)
```

| 評価項目 | スコア範囲 | 意味 |
|---------|-----------|------|
| **影響度（Severity）** | 1〜10 | 故障が発生した場合の影響の大きさ（10=重大事故） |
| **発生頻度（Occurrence）** | 1〜10 | 故障モードが発生する確率（10=非常に高い） |
| **検出難易度（Detection）** | 1〜10 | 故障を出荷前に検出できる難しさ（10=検出不可能） |

**RPNの最大値**: 10 × 10 × 10 = **1000**

> **ソース抜粋**: 「算出されたRPNの数値が大きいほど、該当の故障モードの影響が大きく厳しいことを示しています」
> また、「影響度が著しく大きい場合は、RPNが小さくても対策が必要」
> 出典: [skillnote — FMEA](https://skillnote.jp/knowledge/fmea/)

### FMEA実施の7ステップ

1. 評価フォーマット作成（社内共通フォーマット統一）
2. 工程・作業・部品の記入（漏れなく抽出）
3. 故障モードの抽出
4. 3観点スコアリング（S × O × D）
5. RPN算出
6. 対策検討（基準値超えの故障モードに対して）
7. 関係者レビュー・修正

## 6. ソフトウェア開発者からQAへの知識マッピング

藤田さんが既に持っているソフト開発の知識がQA業務にどう活きるか：

| ソフト開発で培った知識 | QA業務での活用 |
|---------------------|---------------|
| テスト設計・テストケース作成 | 検証手法の策定（同じ「テスト設計」の考え方） |
| バグトラッキング（Jira等） | 工程不良データベース設計に直接応用可能 |
| データ分析・可視化 | SPCの管理図作成、パレート分析 |
| バージョン管理 | サイレントチェンジ監視の仕組み構築 |
| CI/CDパイプライン | 自動検査・データ収集の仕組み構築 |
| SQLデータベース設計 | 受入検査DB・工程不良DBの設計 |
