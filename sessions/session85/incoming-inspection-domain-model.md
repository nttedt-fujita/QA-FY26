# 受入検査 ドメインモデル

**作成日**: 2026-03-11 Session 85
**対象**: GNSS評価ツール（受入検査機能）

---

## 1. 目的

GNSS装置（F9P）の受入検査を自動化し、検査結果を記録する。

---

## 2. ドメイン概念

### 2-1. エンティティ一覧

| エンティティ | 説明 | 主な属性 |
|-------------|------|---------|
| **ロット** | 同一仕様の装置群 | ロット番号、期待値（FW、設定） |
| **装置** | 検査対象のGNSSモジュール | シリアル番号、所属ロット |
| **検査** | 1回の検査作業 | 検査日時、操作者、対象装置 |
| **検査結果** | 各項目の判定 | 項目、判定、実測値、期待値 |

### 2-2. 関係図

```
ロット
  │
  ├── 期待値
  │     - FWバージョン（例: "HPG 1.32"）
  │     - 出力レート（例: 100ms）
  │     - ポート設定（例: UBX+NMEA）
  │
  └── 装置 (1:N)
        │
        └── 検査 (1:N) ← 同じ装置を複数回検査できる
              │
              ├── 検査日時
              ├── 操作者
              │
              └── 検査結果 (1:N) ← 5項目分
                    - 項目（通信疎通/FW/シリアル/出力レート/ポート設定）
                    - 判定（Pass/Fail/Error）
                    - 実測値
                    - 期待値（ロットから取得）
```

---

## 3. 検査項目と期待値の関係

| 検査項目 | 期待値の出所 | 判定ロジック |
|----------|-------------|-------------|
| 通信疎通 | なし（固定） | 応答があればPass |
| FWバージョン | **ロット** | ロットの期待値と一致でPass |
| シリアル番号 | なし | 取得成功でPass |
| 出力レート | **ロット** | ロットの期待値と一致でPass |
| ポート設定 | **ロット** | ロットの期待値と一致でPass |

---

## 4. 5軸チェック

| 軸 | 質問 | 回答 |
|---|---|---|
| **主体** | 誰が？ | 操作者（品質G） |
| **対象** | 何を？ | 装置（GNSSモジュール） |
| **条件** | どんな条件で？ | ロットの期待値 |
| **結果** | どうなった？ | 検査結果（Pass/Fail/Error） |
| **時間軸** | いつ？履歴？ | 検査日時、同一装置の複数回検査履歴 |

---

## 5. 運用フロー

```
1. ロット登録（期待値を設定）
     ↓
2. 装置をUSB接続
     ↓
3. 装置にロットを紐づけ（初回のみ）
     ↓
4. 検査開始
     ↓
5. 各項目を実行、ロットの期待値と比較
     ↓
6. 検査結果を保存
```

## 8. Session 85での気づき（途中）

### 運用イメージの修正

当初「ロットを先に登録して期待値を決める」と考えていたが、実際の運用は違う:

1. **最初の1台を検査** → その値が「仮の期待値」
2. **複数台検査** → 多数派がわかってくる
3. **はずれ値が見える** → 違うFWが混じっている等

### ドメインモデルへの影響（未反映）

| 概念 | 当初の理解 | 修正が必要 |
|------|-----------|-----------|
| 期待値 | 事前に固定 | 最初の1台 or 多数派 |
| ロット | 期待値を持つ | 検査結果のグループ |
| 判定 | Pass/Fail | 一致/不一致（どちらも記録） |
| はずれ値 | Fail扱い | 比較対象として保存 |

### 未整理の項目

- **屋外で確認する項目**の要求整理
- **全結果を保存して比較可能に**する設計
- **多数派・はずれ値の可視化**
