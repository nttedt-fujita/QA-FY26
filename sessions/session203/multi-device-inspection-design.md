# 複数台GNSS検査フローの設計

作成: Session 203 (2026-03-16)

---

## 1. 要求（What）

### 最終目標

**見た目が全部同じGNSSレシーバーを、検査後に物理的に識別できるようにしたい**

### 背景・課題

- GNSSレシーバーは外見上すべて同一（区別がつかない）
- シリアル番号は**接続しないとわからない**
- 複数台接続して検査した後、NGが出ても「どの個体がNGか」が物理的にわからない

### 理想的なフロー

```
1. レシーバー複数台をUSBに接続
2. 画面に接続された装置一覧が表示される（シリアル番号付き）
3. 設定確認（メッセージスキャン等）
4. まとめて検査実行
5. 結果表示（シリアル番号ごとに合格/NG）
6. NGのレシーバーを物理的に特定 ← ★ここが課題
```

## 2. 識別方式の検討

### 検討した方式

| 方式 | 概要 | メリット | デメリット |
|------|------|---------|-----------|
| A) 抜き差し検知 | 切断を検知して「○○が切断されました」と表示 | 確実、追加機材不要 | 毎回抜き差しが必要 |
| B) 接続時ラベリング | 1台ずつ接続→シリアル表示→ラベル貼り | 一度貼れば以降も識別可能 | 初回の手間、毎回新品なら毎回貼る |
| C) USBポート固定 | ハブのポート番号と対応づけ | 運用でカバー | Linuxではポート対応が不安定 |
| **D) LED点滅** | 基板のTX/RX LEDを点滅させて識別 | 直感的、追加機材不要 | LEDがない基板では使えない |

### 採用方式

**D) LED点滅方式** を優先検証する

- USB-UART変換基板には通常TX/RX LEDが載っている
- 意図的にデータを送信すれば、TX LEDが点滅する
- UIから「このデバイスを特定」ボタンを押す → LED点滅 → 物理識別

**フォールバック**: LED点滅が使えない場合は A) 抜き差し検知 を併用

## 3. LED点滅テスト用バイナリ設計

### 目的

LED点滅方式が実際に動作するか検証するためのテストツール

### 仕様

```
cargo run --bin blink_test -- /dev/ttyACM0 3
```

| 引数 | 必須 | 説明 |
|------|------|------|
| path | ○ | デバイスパス（例: /dev/ttyACM0） |
| duration | - | 点滅時間（秒）。デフォルト: 3 |

### 動作

1. 指定デバイスのシリアルポートを開く（115200bps）
2. MON-VERポーリングコマンドを100ms間隔で送信
3. 指定時間経過後に停止
4. 送信中は進捗を表示

### ファイル構成

```
prototype/m1-gnss/backend/
├── Cargo.toml              ← [[bin]] 追加
├── src/
│   ├── main.rs             ← 既存（Webサーバー）
│   └── bin/
│       └── blink_test.rs   ← 新規
```

### Cargo.toml 追記内容

```toml
[[bin]]
name = "blink_test"
path = "src/bin/blink_test.rs"
```

### 実装概要

```rust
use m1_gnss::ubx::common::build_ubx_poll;
use serialport::SerialPort;
use std::time::{Duration, Instant};

fn main() {
    // 引数パース（std::env::args使用）
    let path = std::env::args().nth(1)
        .expect("Usage: blink_test <path> [duration_sec]");
    let duration: u64 = std::env::args().nth(2)
        .map(|s| s.parse().unwrap_or(3))
        .unwrap_or(3);

    println!("デバイス: {}", path);
    println!("点滅時間: {}秒", duration);

    // シリアルポート開く
    let mut port = serialport::new(&path, 115200)
        .timeout(Duration::from_millis(100))
        .open()
        .expect("ポートを開けませんでした");

    // MON-VER poll (0x0A, 0x04)
    let poll_cmd = build_ubx_poll(0x0A, 0x04);

    println!("点滅開始...");
    let start = Instant::now();
    let mut count = 0;

    while start.elapsed().as_secs() < duration {
        if let Err(e) = port.write_all(&poll_cmd) {
            eprintln!("送信エラー: {}", e);
            break;
        }
        count += 1;
        print!(".");
        std::io::Write::flush(&mut std::io::stdout()).ok();
        std::thread::sleep(Duration::from_millis(100));
    }

    println!();
    println!("完了: {}回送信", count);
}
```

### 依存関係

- `serialport` - 既存（Cargo.tomlに定義済み）
- `m1_gnss::ubx::common::build_ubx_poll` - 既存

新規依存の追加は不要。

## 4. 検証手順

1. blink_testバイナリをビルド
2. レシーバーを1台接続
3. `cargo run --bin blink_test -- /dev/ttyACM0 3` を実行
4. **基板のLEDが点滅するか目視確認**
5. 結果を記録

### 期待される結果

- TX LED（または通信LED）が約100ms間隔で点滅する
- 点滅が確認できれば、この方式でデバイス識別が可能

### LEDが点滅しない場合

- 基板にLEDがない → 抜き差し検知方式にフォールバック
- LEDが電源インジケータのみ → 抜き差し検知方式にフォールバック

### API設計


| パラメータ | 型 | 説明 |

## 6. 対応台数

### 現時点の想定

- **最大3台**を想定（手元の検証環境）
- 将来的には増やす可能性あり

### UI設計への影響

- 3台程度なら縦並びリストで十分
- 5台以上になるとグリッド表示など検討が必要
