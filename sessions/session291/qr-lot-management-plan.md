## 解決する3つの課題

| # | 課題 | 詳細 |
|---|------|------|
| 1 | ロット概念がない | 入荷部品にロット番号が付与されておらず、どのロットがどこにあるか把握不能 |
| 2 | 記録がデータになっていない | 受入検査のExcel記録のみ。梱包変更・小分けは記録なし（隠れコスト） |
| 3 | 問題発生時に追跡できない | 不良品が市場流出しても、どのロット・サプライヤーか特定不能 |

## Phase 1：ロット番号の設計

### 採番ルール

```
TA - 20260325 - 001
│      │        │
│      │        └─ 連番（同日複数入荷時、3桁）
│      └────────── 入荷日（YYYYMMDD）
└─────────────── 部品コード（例: TA=テールアーム）
```

**ポイント**: 最初はシンプルに。複雑な採番ルールは現場に定着しない。

### QRコードに含める情報

| 区分 | 項目 | 備考 |
|------|------|------|
| 必須 | ロット番号 | 上記採番ルールに従う |
| 必須 | 部品名・品番 | 部品の正式名称と品番 |
| 必須 | サプライヤー名 | 仕入先の識別 |
| 必須 | 入荷日 | 入荷した日付 |
| 必須 | 員数 | 梱包変更後も更新して貼り直し |
| 推奨 | 検査結果 | 合格 / 不合格 / 条件付合格 |

※ 梱包変更・小分け時は「員数」フィールドを更新し、ラベルを貼り直す

---

## 運用フロー（スキャンのタイミング）

```
1. 入荷時      → QRコード発行・貼付（ロット番号を採番してラベル印刷）
2. 検査完了時  → スキャン＋検査結果を記録（合格/不合格をスプレッドシートに登録）
3. 梱包変更時  → スキャン＋員数を更新【隠れコスト可視化の鍵】作業時間も記録
4. 入庫時      → スキャン＋棚番を記録（どの棚に格納したか紐付け）
5. 出庫時      → スキャン＋出庫先を記録（組立業者など出庫先を登録）
```

**梱包変更時のスキャンで「隠れコスト」を自動的に定量化できる**

## 必要なものと初期コスト

| 区分 | アイテム | 費用 |
|------|---------|------|
| 初期 | ラベルプリンター（DYMO / ブラザーQL等） | 1〜3万円 |
| 月額 | ラベルシール（市販のシール用紙） | 数百円/月 |
| 初期 | スキャナー（スマートフォンで代替可能） | 0円 |
| 月額 | 管理システム（Googleスプレッドシート） | 0円 |
| **合計** | **初期費用** | **1〜3万円** |
| | **月額ランニング** | **ほぼ0円** |

AI化（160〜430万円）の約1/100のコスト

## 導入スケジュール

| 時期 | フェーズ | 内容 |
|------|---------|------|
| Week 1-2 | 設計 | ロット番号ルール決定、スプレッドシート設計、QRコード項目確定 |
| Week 3 | 構築 | QRコード生成の仕組み構築（Google Apps Script）、テスト印刷 |
| Week 4 | 試験運用 | ラベルプリンター導入、1品番のみで試験運用、現場フィードバック収集 |
| Month 2 | 全展開 | 全品番に展開、スキャナー配備、運用マニュアル整備 |
| Month 3 | 分析開始 | 隠れコスト定量化、データ蓄積、Phase 2へ移行準備 |

**まず1品番のみで試験運用 → 現場確認 → 全展開**

---

## 期待される効果

| 効果 | 内容 |
|------|------|
| トレーサビリティ確保 | 不良発生時にロット・サプライヤーを即特定。市場流出リスクの管理が可能に |
| 隠れコストの定量化 | 梱包変更・小分け作業の時間を記録。不可視だったコストが数値化 |
| Phase 2への基盤構築 | VSM・ムダ分析に必要なデータが自然に蓄積 |
| 低リスクで即導入可能 | 初期費用1〜3万円。スマホスキャンで現場負担最小限 |

---

## まとめ

1. AI化（160〜430万円・回収15〜58年）は現規模では効果なし
2. QRコード管理で「ロット追跡」「隠れコスト可視化」を同時解決
3. 初期費用1〜3万円・月額ほぼ0円で即導入可能
4. 1品番の試験運用からスタートし、Month 3にはPhase 2移行
