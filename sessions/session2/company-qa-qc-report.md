## 1. 自社の立ち位置

### 部品購入型ドローンメーカーとは

自社（AirGrow等を開発）は、**部品を自社で製造しない**。中国サプライヤ等から部品を購入し、組み立てて最終製品にするメーカー。

```
サプライヤ（部品製造）→ 自社（組立・検査・出荷）→ 顧客
  EVPEAK（バッテリー）
  GREPOW（バッテリー）
  SKYDROID（通信機器）
  Lidarメーカー
  GNSSメーカー
```

**品質上の特徴**:
- 部品そのものの品質は**サプライヤに依存**する
- 自社でコントロールできるのは「選定」「受入検査」「組立工程」「最終検証」
- **サプライヤ品質管理（SQM）が生命線**

> 参考: [SG Systems — Supplier Quality Management](https://sgsystemsglobal.com/glossary/supplier-quality-management-sqm/)
> "SQM is the end-to-end framework for selecting, qualifying, monitoring and continuously improving external partners that supply materials, components, services and contract manufacturing."

## 2. 品質管理フロー（IQC → IPQC → OQC）

部品購入型メーカーの品質管理は、**3段階**で構成される。

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ SQM     │ →  │ IQC     │ →  │ IPQC    │ →  │ OQC     │
│サプライヤ│    │受入検査  │    │工程検査  │    │出荷検査  │
│品質管理  │    │         │    │         │    │         │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
  ↑ QA領域                                      ↓
  選定・監査                                   顧客へ
  品質契約
```

### 各段階の内容（自社の文脈）

| 段階 | 英語 | 自社での活動 | 担当イメージ |
|------|------|-------------|-------------|
| **SQM** | Supplier Quality Management | サプライヤ選定・評価・監査・品質契約 | 品質G全体 |
| **IQC** | Incoming Quality Control | 部品の受入検査（AQL抜取 or 全数） | ふじた（DB化）、品質G |
| **IPQC** | In-Process Quality Control | 組立工程中の検査（はんだ、組付け等） | こいたばし（H12G評価等） |
| **OQC** | Outgoing Quality Control | 完成機の最終検査（飛行テスト含む） | 品質G全体 |

> 参考: [ワートレ — 製造の様々な段階でのQC](https://worldtrade.jp/manufacturing/qc-quality-control-phase/)

### QAとQCの役割分担（自社の場合）

| 活動 | QA（品質保証） | QC（品質管理） |
|------|---------------|---------------|
| **視点** | 顧客・ユーザーに品質を**保証**する | 工程内で品質問題を**検出・防止**する |
| **範囲** | 全ライフサイクル（設計〜アフターサービス） | 製造工程（IQC〜OQC） |
| **自社での例** | センサー評価手法の策定、検証基準の設定、DB化による品質可視化 | 受入検査の実施、工程内検査、出荷検査 |
| **藤田さんの位置** | **ここ** — 仕組みを作る側 | 仕組みに従って検査を実行する側 |

> 参考: [ISOプロ — QAとQCの違い](https://activation-service.jp/iso/column/7762)

**ポイント**: 藤田さんの仕事は「検査する人」ではなく、**「検査の仕組みを作る人」（QA）**。

## 3. 中国サプライヤ管理の特有課題

### 3-1. サイレントチェンジ

> **定義**: 部品や材料のサプライヤーが、発注企業の許可なく勝手に製品の仕様を変更し納品すること。
> 出典: [ナノクシード — サイレントチェンジとは](https://nanoxeed.co.jp/technologyinfo/silentchange/)

**なぜ中国サプライヤで多いか**:
- 国内商習慣・マインドの違い（コスト最適化を優先する傾向）
- トレーサビリティ体制が未整備
- 「仕様変更＝改善」と認識されるケースがある

**自社で起こりうるシナリオ**:

| リスク | 影響 | 検出方法 |
|--------|------|---------|
| バッテリーセルの材料変更 | 発火・膨張リスク、容量低下 | 定期的な分解検査、重量測定 |
| FWの無断アップデート | 通信異常、互換性問題 | FWバージョン記録・比較（M3で対応） |
| 基板部品の代替品使用 | 性能劣化、信頼性低下 | X線検査、電気特性測定 |
| Lidarの光学系変更 | 測距精度低下 | 定量的精度評価（M1で対応） |

> 参考: [NITE — 身近に潜むサイレントチェンジ問題](https://www.nite.go.jp/data/000087880.pdf)

**既存の対策（スライドより）**:
- PO（発注書）発行時にバージョン連絡を必須化（いしかわ担当）

**強化すべき対策**:
1. **品質契約（QPA）** — 仕様変更時の事前通知義務を契約に明記
2. **受入検査DB（M3）** — FW/HWバージョンを記録し、変更を自動検出
3. **定期サンプル検査** — ロットごとに重量・外観・電気特性を記録して傾向管理

### 3-2. ロット間ばらつき

中国サプライヤの製品は、ロット間で品質がばらつきやすい。

**原因**:
- 製造設備のメンテナンス頻度のムラ
- 原材料調達先の変動
- 作業者のスキルばらつき

**対策（自社で取れる手段）**:
- IQCでの抜取検査データを蓄積し、ロット間のCp/Cpk推移を監視（M4で対応）
- 管理限界を超えたロットはサプライヤにフィードバック

## 4. 品質Gの活動マッピング

スライドの取り組み一覧を、QA/QC活動の体系にマッピングする。

### QA活動（品質保証 = 仕組みを作る）

| 取り組み | 担当 | QA活動の種別 |
|---------|------|-------------|
| センサー定量的評価手法の策定（M1） | ふじた | **検証手法の策定** |
| 点群データ検証方法の策定（M2） | ふじた | **検証手法の策定** |
| FMEA（設計FMEA・工程FMEA）の実施 | いしかわ | **リスク分析・予防** |
| 経年劣化部品の検証方法策定 | いしかわ・こいたばし | **信頼性試験の策定** |
| 東南アジア安全規格習得 | みんな | **規制適合** |

### QC活動（品質管理 = 検査を実行する）

| 取り組み | 担当 | QC活動の種別 |
|---------|------|-------------|
| EVPEAK/GREPOW/SKYDROID継続評価 | いしかわ | **SQM（サプライヤ監視）** |
| 受入検査データベース化（M3） | ふじた | **IQC（受入検査の仕組み化）** |
| 工程不良データベース化（M4） | ふじた | **IPQC/OQCデータの体系化** |
| 帯電散布の検証方法策定 | こいたばし | **特殊工程の品質検証** |
| H12G EVT/DVT量産評価 | こいたばし | **製品評価（開発〜量産移行）** |
| 次世代バッテリ・充電器の導入前評価 | いしかわ | **新規部品の事前評価** |

## 5. 藤田さんの「ソフトQA」としての役割

### 役割の定義

藤田さんは「ソフトウェア開発のスキルを持つQA担当者」。QA（仕組みを作る側）として、ソフト開発スキルを武器に品質の仕組みを構築する。

### ソフト開発スキルの活かし方

| ソフト開発スキル | QA業務での活用 | 対応ミッション |
|----------------|---------------|---------------|
| **DB設計（SQL）** | 受入検査DB・工程不良DBのスキーマ設計 | M3, M4 |
| **データ分析（Python）** | SPC管理図、パレート分析、サイレントチェンジ検出 | M3, M4 |
| **テスト設計** | センサー評価手法の策定（テスト計画の考え方は同じ） | M1, M2 |
| **自動化（スクリプト）** | 検証パイプライン構築、レポート自動生成 | M2 |
| **バージョン管理** | FW/HWバージョン管理、変更追跡の仕組み | M3 |
| **CI/CD** | 自動検査・データ収集パイプラインの構築 | M2, M3, M4 |

### 他メンバーとの連携ポイント

```
いしかわ（SQM・FMEA）
  │ サプライヤの品質データ → M3（受入検査DB）に投入
  │ FMEAの故障モード → M4（工程不良DB）の不良コードに反映
  │
ふじた（DB・データ分析・評価手法）
  │ DB化したデータ → ダッシュボードで品質G全員が閲覧可能に
  │ センサー評価手法 → いしかわのFMEAのセンサー系リスク評価に活用
  │
こいたばし（製品評価・特殊工程）
  │ H12G評価データ → M4（工程不良DB）に投入
  │ 帯電散布の検証結果 → M2（点群検証）の品質指標に反映の可能性
```

## 6. まとめ

### 自社QA/QCの全体像

1. **部品購入型メーカー**として、サプライヤ品質管理（SQM）が最重要
2. 品質管理フローは **SQM → IQC → IPQC → OQC** の4段階
3. 中国サプライヤ特有の課題（サイレントチェンジ、ロット間ばらつき）に対応が必要
4. 品質Gの活動はQA（仕組み）とQC（実行）の両面にまたがる

### 藤田さんの位置付け

- **QA（仕組みを作る側）** が主な役割
- ソフト開発スキルは「DB設計」「データ分析」「テスト設計」「自動化」でQA業務に直結
- 4つのミッション（M1〜M4）は全て **「データに基づく品質判断の仕組み」** を作る仕事
