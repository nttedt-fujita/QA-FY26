# ヒアリング準備（確度ベース）

**作成**: Session 317 (2026-03-26)
**目的**: 最小工数で効果の出せるターゲットを特定し、「ギリギリ求められるハードルを超える」ための合意形成
**方針**: 1回で終わらせない。繰り返し聞くことで初回と違う内容が出てくる

---

## 基本方針

### 分析結果の使い方

**IMPORTANT**: 分析結果（CSV分析・マニュアル分析・確度マトリクス）は、こちらの理解を押し付けるためのものではない。

**使い方**: 実務者の意見・回答の**根拠確認**に使う
- 「あなたの話はこのデータと合っていますね」→ 確度が上がる
- 「ここはデータと食い違いますが、なぜですか？」→ 新しい発見につながる
- 実務者の話が先、データは裏取りに使う

### スコープの方針

**現実**:
- 藤田さんはM1〜M4の4ミッション担当。M3/M4に全工数は注げない
- M1-B(GNSS)は4月から再開予定。ツール完成4/30のマイルストーンがある
- 全部を理想形にする工数はない

**方針**:
- **最小工数で効果の出せるところ**から手をつける
- **将来的に別のシステムに移管**することも見据えた設計にする（今回が完成形ではない）
- **求められているギリギリのハードルを超える**レベルに業務量を想定する
- ヒアリングの目的 = **理想を追求するため**ではなく、**ターゲットの絞り込みと合意形成のため**

**ヒアリングの2段階**:

```
【第1段階】課題抽出 = 広く網羅的に
  → せっかく関わるので、課題の全体像はきちんと把握する
  → ここを雑にやると、的外れな所に工数をかけるリスクがある

【第2段階】優先順位付け + 合意形成 = 絞り込む
  → 抽出した課題から「最小工数で効果が出る」ターゲットを選ぶ
  → 今回やらないことを合意する（将来のシステム移管に先送り）
```

**ヒアリングで明らかにしたい核心**:
1. **課題の全体像**（品質管理プロセスのどこに、どんな課題があるか）
2. **何が「求められているハードル」か**（誰が何をいつまでに期待しているか）
3. **最小工数で超えられるポイントはどこか**（課題全体の中で一番効果が出るところ）
4. **今回やらないことの合意**（課題として認識した上で、将来に先送り）

### 3周構造

| 周回 | やること | 狙い |
|------|---------|------|
| **1周目** | 「こう理解しているけど合ってる？」 | 現状の認識合わせ。事実の修正 |
| **2周目** | 「理想はどうなりたい？一番困ってることは？」 | 要求の引き出し。表面的な回答が多い |
| **3周目以降** | 1周目・2周目の回答を突き合わせる | 矛盾や新情報が出る。「実はこれが本当に困ってて...」が引き出される |

**Session 260の教訓**: 田原さんヒアリングで「見えていた作業」と「実際の作業フロー」が大きく異なることが判明した。1回のヒアリングでは表面しか見えない。
