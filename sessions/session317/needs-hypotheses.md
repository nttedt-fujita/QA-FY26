## 2. 新規要求仮説（S316-317の分析から導出）

### N-01: kintone製品マスタと受入検査を紐づけられる（Event-driven）

> **When** 受入検査を記録するとき、**システムは** kintone機体マスタ(71)/オプションマスタ(72)のS/Nと **紐づけて記録できる**

- **ステータス**: 仮説
- **確度**: D（kintone受入製品アプリとの関係が不明なため）
- **根拠**: ドメインモデルR2「受入検査Excel↔kintone断絶」[A]
- **ヒアリングで確認**: kintone受入製品アプリと受入検査Excelは同じ検査の話か
- **将来移管への影響**: S/Nベースの紐づけがあれば将来のシステムにも引き継げる

### N-02: 工程内不良を最低限記録できる（Ubiquitous）

> **システムは**、工程内で発見された不良の **発生日、製品/部品、不良内容、対処を記録する**

- **ステータス**: 仮説
- **確度**: D（記録方法自体が不明）
- **根拠**: ドメインモデルG2「工程内不良記録がデジタル管理されていない」[A]
- **ヒアリングで確認**: I-3（記録方法の実態）を先に確認。記録の必要性の合意が先

### N-03: 市場不具合の原因分類と工程記録を紐づけられる（Event-driven）

> **When** kintone農業問合せで「製造ミス」「検査漏れ」を原因と分類したとき、**システムは** 該当する工程記録を **参照できる**

- **ステータス**: 仮説
- **確度**: D（工程記録が存在しないため現時点では実現不可）
- **根拠**: ドメインモデルR4「工程不良↔CS問合せ断絶」[A] + kintone原因分類に「製造ミス」がある[A]
- **将来移管への影響**: この紐づけがないと8Dフロー（問題→原因→対策→効果確認）が回らない

### N-04: PSI集計の自動化（Event-driven）

> **When** 週次の集計タイミングが来たとき、**システムは** kintone/GEN/群馬通商のデータを **自動で統合しPSI形式で出力する**

- **ステータス**: 複数情報源から推定
- **確度**: B（PSI分析[A] + ヒアリング[B]で要求は明確。データソースの全リストが確度B）
- **根拠**: 掛川さん週2h [B]、PSI突合で粒剤12ヶ月完全一致 [A]
- **制約**: 製販会議が月1化の可能性（社長フィードバック S312）

### N-05: 将来のシステム移管を妨げない設計（Unwanted）

> **If** 将来kintoneや他システムへの移管が決定されたとき、**システムは** 記録されたデータを **標準的な形式で出力・移行できる**

- **ステータス**: 仮説（藤田さんの方針 S317）
- **確度**: —（設計方針であり確度の対象外）
- **根拠**: 「将来的に別のシステムに移管も見据えた今回の対処」という方針

## 3. ヒアリングでのEARS活用

ヒアリングで実務者から引き出した要求を、EARS形式で構造化する。

### 聞き方のテンプレート

**現状確認時**: 要求の背景を聞く
```
「〇〇の場合、今はどうしていますか？」
→ 回答をEARSのWhen/While/Ifで構造化
```

**要求引き出し時**: あるべき姿を聞く
```
「〇〇のとき、理想的にはどうなっていてほしいですか？」
→ 回答をEARS形式に変換して確認
  「つまり、〇〇したとき、システムは△△する、という理解で合っていますか？」
```

### EARSチェックポイント（rules/05参照）

聞いた要求をEARS形式に変換する際の確認:
- [ ] **主語は明確か？** — 「システム」「品質担当者」等
- [ ] **トリガーは明確か？** — When/While/Ifで条件を示したか
- [ ] **動作は明確か？** — 「〇〇する」で具体的に
- [ ] **曖昧な言葉を避けたか？** — 「適切に」「速やかに」「など」は禁止
