### 1.2 m3-research-filesの推奨

**提案D（推奨）**: 抜取検査計画見直し + Googleフォームで即日開始

```
Week 1-2:  現場ヒアリング（工数内訳 + 検査計画の確認）
    │
    ├─ 即効施策: 抜取検査計画の適正化 → 20-30%削減
    │
    ├─ 並行: Googleフォーム + スプレッドシートで記録開始
    │
Week 3-4:  記録運用の開始・定着
    │
Month 2-3: データ蓄積 → パレート分析 → 次の改善ポイント特定
```

## 2. 根本的な問題

**特性要因図（fishbone-diagram.md）で判明した未確認事項**:

| 優先度 | 未確認事項 | 影響 |
|--------|-----------|------|
| **P0** | 300hの内訳（どの作業に何時間か） | ツールで何を改善すべきか不明 |
| **P0** | 検査員は何人か、1人あたり工数は | システム規模の判断材料がない |
| **P1** | 抜取検査計画の根拠 | 計画見直しで即効削減できるか判断できない |
| **P1** | 検査以外の付帯作業の内容 | ツールで削減できる工数か判断できない |

**結論**: 一次調査なしでツールを作っても、**何を解決するか不明**。

## 3. 見直しの選択肢

### Option A: プロトタイプ続行（現状維持）

**条件**: プロトタイプが1-2週間以内に現場投入できる場合

| 作業 | 所要時間 |
|------|---------|
| Phase A（バックエンド） | 1-2セッション |
| Phase B（フロントエンド） | 1-2セッション |
| Phase C（統合） | 0.5セッション |
| **合計** | **2.5-4.5セッション** |

**リスク**:
- 一次調査の結果、設計変更が必要になる可能性大
- 現場の本当の課題に合っていないかもしれない

### Option B: 一次調査優先 → Googleフォームで即日開始

**条件**: 現場投入までに2週間以上かかる場合（推奨）

| 週 | 作業 |
|----|------|
| Week 1-2 | 現場ヒアリング + 特性要因図の検証 |
| Week 1-2 | Googleフォーム + スプレッドシートで記録開始 |
| Week 3-4 | 抜取検査計画の見直し（即効施策） |
| Month 2-3 | データ蓄積 → 本当に必要なツールを設計 |

**メリット**:
- コストゼロで記録基盤が稼働
- 即効性のある成果（20-30%削減）が出せる
- データに基づいてツール設計ができる

### Option C: ハイブリッド（一次調査 + プロトタイプ簡素化）

**条件**: プロトタイプの資産を活かしつつ、一次調査も実施

| 作業 | 内容 |
|------|------|
| 一次調査 | Week 1-2で並行実施 |
| プロトタイプ簡素化 | 入力画面のみ作成（ダッシュボードは既存Streamlit利用） |
| 判断 | 調査結果でスコープ再定義 |

**簡素化後のスコープ**:
- ロット登録（✅完成）
- 検査記録入力（最小限）
- ダッシュボードは既存Streamlitを流用

## 5. 推奨アプローチ

### 推奨: Option B（一次調査優先）

**理由**:
1. 一次調査なしでツールを作っても、的外れになる可能性が高い
2. Googleフォームなら即日開始可能、コストゼロ
3. データが貯まってからツール設計した方が精度が高い
4. 抜取検査計画見直しで即効性のある成果が出せる

**プロトタイプの扱い**:
- 一旦凍結（削除はしない）
- 一次調査の結果、本当に必要な機能が見えたら再開
- または、Googleフォームで限界が見えたタイミングで再開

### アクションプラン

```
【今すぐ】
1. 特性要因図を現場ヒアリングの叩き台として使用
2. 一次調査チェックリストを整理（次タスク）

【Week 1-2】
3. 現場ヒアリング実施（工数内訳、検査計画、付帯作業）
4. Googleフォームで記録開始（並行）

【Week 3-4】
5. 抜取検査計画の見直し（JIS Z 9015準拠）
6. 記録運用の定着支援

【Month 2-3】
7. 蓄積データでパレート分析
8. 本当に必要なツールを設計（プロトタイプ再開 or 新設計）
```
