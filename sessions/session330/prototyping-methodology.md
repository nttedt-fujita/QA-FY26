# プロトタイピング方法論: ヒアリング先行型 vs 早期プロトタイプ型

作成: Session 330 (2026-03-30)
目的: PSIプロトタイプ開発アプローチの判断材料として収集

---

## 1. 要件定義でヒアリングが限界になる理由

**出典**: [スパイスファクトリー: プロトタイプ開発](https://spice-factory.co.jp/management/prototype/)、[発注ラウンジ: プロトタイプとは](https://hnavi.co.jp/knowledge/blog/prototype/)

### 限界のメカニズム

```
発注者（業務担当者）がシステム開発に不慣れ
    ↓
「最終完成品をイメージしにくい」
    ↓
最初の要件定義段階で要求が出そろわない
    ↓
「機能の細部や操作イメージまで記述することはかなり難易度が高い」
    ↓
完成後に「こんなはずじゃなかった」が発生
```

### 業務システムで特に顕著な理由

- 業務ドメインの暗黙知（例: 「住所欄にkintoneのレコード番号を書く」という運用）はヒアリングで引き出しにくい
- 担当者自身も「何が不便か」を言語化できていないケースがある
- 実際に画面を触って初めて「ここが違う」と気づくことが多い

## 2. プロトタイプファースト型の特徴

**出典**: [スパイスファクトリー](https://spice-factory.co.jp/management/prototype/)、[rekaizen: プロトタイプ開発](https://rekaizen.com/article/detail/business-system/16909/)

### 定義

```
プロトタイプ型: 要件定義・設計では細かな部分まで厳密に詰めない
                 最初のプロトタイプでは「メイン機能・検討が必要な機能」を実装
                 ↓ 実際に操作してもらう
                 ↓ フィードバックで仕様を具体化
                 ↓ 次イテレーション
```

### メリット

| メリット | 内容 |
|---------|------|
| 早期の要件明確化 | 「初期からすべての要求が出そろわなくても開発を進められる」 |
| 認識齟齬の回避 | 実装に近い形で確認できるため「こんなはずじゃなかった」を避けられる |
| 高品質な要件収集 | 実際に操作して「想定していなかった使用方法や課題を発見できる」 |
| 信頼関係構築 | クライアントと開発者の信頼関係が構築されやすい |

### デメリット・リスク

| デメリット | 内容 |
|-----------|------|
| 大規模システムに不向き | 画面数が多いと作成に時間がかかる |
| 仕様拡大リスク | 「あれもこれも」で開発コスト・期間が増加する可能性 |
| 開発負担増 | 「格段に開発工数がかかる」 |

## 3. ヒアリング先行型との比較

| 観点 | ヒアリング先行型 | プロトタイプファースト型 |
|------|---------------|----------------------|
| 要件確定のタイミング | 開発前（仕様書・ワイヤーフレームで合意） | 開発しながら確定 |
| ユーザーの理解度要件 | 高（完成イメージを言語化できる必要） | 低（操作しながら発見できる） |
| 認識ずれ発覚タイミング | 完成後（コスト大） | 早期（コスト小） |
| 向いているケース | 要件が明確・変更が少ない | 要件が曖昧・業務フロー未整理 |
| リスク | 手戻りコスト大 | 仕様拡大による期間延長 |

## 4. 自社の状況への当てはめ

### 現状の「ヒアリング限界」の具体例（実証済み）

| 発見した暗黙知 | 発見方法 |
|--------------|---------|
| 住所欄にkintoneレコード番号を記載（掛川さん運用） | S320ヒアリング後のExcel直接調査 |
| フラグ管理の実態（EC/仕損テキスト記載） | S327 Excelファイル直接分析 |
| kintone→GEN出庫フローが週次手動CSV | S324b ヒアリング |
| 群馬通商Excelが中継ハブになっている | S326 作業フロー可視化 |

→ **これらは全て「ヒアリングだけでは出てこなかった」情報**。
→ Excelを直接見る/現場観察/作業フロー図作成によって発覚。

### 推奨アプローチ: ヒアリング先行 + 早期プロトタイプ組み合わせ型

```
フェーズ1: 現状フロー把握ヒアリング（現在実施中）
    ↓
フェーズ2: 現状フローを可視化（作業フロー図・SIPOC）← S266〜326実施済み
    ↓
フェーズ3: 最小限プロトタイプで操作感を確認（次ステップ）
    - 画面触ってもらって「ここが違う」を引き出す
    - 完成品ではなくモックでよい
    ↓
フェーズ4: フィードバックで要件精緻化 → 本実装
```

### なぜ「早期プロトタイプ先行」が自社に向いているか

1. **業務担当者がシステム開発に不慣れ**: 現場はExcel/kintone運用が中心
2. **暗黙知が多い**: 「住所欄にレコード番号」等、言語化されていない運用が多数存在
3. **要件が流動的**: AI化断念・方針転換（S256/S260）が示す通り、要件は変わりやすい
4. **小規模チーム**: 藤田さん1名の開発 → 大規模ヒアリングより素早い試作→修正サイクルが合う

## 5. Discovery and Delivery（双発型開発）

業界での呼称: **Dual Track Development / Discovery and Delivery**

```
Discoveryトラック: ユーザーリサーチ → プロトタイプ → 検証 → 要件確定
        ↕ 並行
Deliveryトラック: 確定した要件 → 設計 → 実装 → リリース
```

**ポイント**: Discoveryが常に先を走り、Deliveryに「検証済み要件」を供給する。
両者を並行させることで、開発速度と要件精度を両立。

**自社での適用**: 現在まさにDiscoveryトラック（現状把握・ヒアリング・可視化）を走らせている段階。次はDeliveryトラック（PSIプロトタイプ実装）へ。
